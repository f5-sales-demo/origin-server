## نظرة عامة على التطبيقات

يعمل كل تطبيق كـ 4 حاويات Docker موزعة الحمل خلف مجموعات nginx upstream، ويمكن الوصول إليها من خلال وكيل nginx العكسي على الجهاز الافتراضي. تضمن الجلسات الثابتة أن التطبيقات ذات الحالة (Juice Shop، DVWA، VAmPI، CSD Demo) تُوجَّه بشكل متسق. جميع التطبيقات معرضة للثغرات عمداً ومصممة لاختبارات الأمان.

استبدل `<ORIGIN>` بـ `http://<PUBLIC_IP>` في جميع الأمثلة أدناه. بعد النشر عبر Terraform، احصل على عنوان IP بواسطة `terraform output -raw public_ip`.

## OWASP Juice Shop

| | |
|---|---|
| **المسار** | `/juice-shop/` |
| **الصورة** | `bkimminich/juice-shop:latest` |
| **النسخ** | 4 (المنافذ 3001-3004)، ثابتة عبر `hash $cookie_token`، ذاكرة تخزين وكيل مؤقتة (60 ثانية TTL) |
| **الموارد** | 2 وحدة معالجة / 1 جيبي بايت ذاكرة لكل نسخة |
| **إطار العمل** | Node.js / Angular |
| **المشروع** | [owasp.org/www-project-juice-shop](https://owasp.org/www-project-juice-shop/) |

OWASP Juice Shop هو أحدث تطبيق ويب معرض للثغرات وأكثرها صيانة نشطة. يغطي قائمة OWASP Top 10 بالكامل مع أكثر من 100 تحدٍّ في تطبيق تجارة إلكترونية واقعي.

### سيناريوهات العرض التوضيحي

| السيناريو | الفئة | متجه الهجوم |
|---|---|---|
| تجاوز تسجيل الدخول بحقن SQL | WAF | `' OR 1=1--` في حقل البريد الإلكتروني لتسجيل الدخول |
| XSS منعكس | WAF، CSD | حقن سكريبت عبر معامل البحث |
| XSS مبني على DOM | CSD | حمولة في جزء URL |
| مصادقة مكسورة | WAF | هجوم القوة الغاشمة على تسجيل الدخول، التلاعب بـ JWT |
| إساءة استخدام API | أمان API | وصول غير مصرح به إلى نقاط نهاية `/api/` |
| كشف البيانات الحساسة | أمان API | الوصول إلى بيانات المستخدم دون تصريح |
| CSRF | WAF | تزوير الطلب عبر المواقع على تغييرات الملف الشخصي |

### الوصول

```bash
curl -s "http://<PUBLIC_IP>/juice-shop/" -o /dev/null -w "%{http_code}"
```

## DVWA (تطبيق الويب المعرض للثغرات)

| | |
|---|---|
| **المسار** | `/dvwa/` |
| **الصورة** | مخصصة `dvwa-fpm:latest` (php-fpm + nginx، مبنية من `ghcr.io/digininja/dvwa:latest`) |
| **النسخ** | 4 (المنافذ 8101-8104)، ثابتة عبر `hash $cookie_PHPSESSID` |
| **الموارد** | 0.5 وحدة معالجة / 256 ميبي بايت ذاكرة لكل نسخة |
| **قاعدة البيانات** | MariaDB 10.11 مشتركة (حاوية `dvwa-db`، 1 وحدة معالجة / 768 ميبي بايت) |
| **إطار العمل** | PHP 8 / php-fpm / MariaDB |
| **بيانات الاعتماد** | `admin` / `password` |

DVWA هو المعيار الصناعي لاختبار WAF. يحتوي على مستويات أمان قابلة للتعديل (منخفض، متوسط، عالٍ، مستحيل) تضيف تدريجياً التحقق من المدخلات وترميز المخرجات.

### سيناريوهات العرض التوضيحي

| السيناريو | الفئة | مستوى الأمان |
|---|---|---|
| حقن SQL | WAF | منخفض: `' OR 1=1#` بسيط، عالٍ: حقن SQL أعمى |
| حقن الأوامر | WAF | منخفض: `; ls`، عالٍ: أحرف مفلترة |
| تضمين الملفات | WAF | منخفض: اجتياز المسار المباشر، عالٍ: معقم |
| XSS (منعكس) | WAF، CSD | منخفض: `<script>` أساسي، عالٍ: تجاوز مشفر |
| XSS (مخزّن) | WAF، CSD | منخفض: سكريبت دائم في سجل الزوار |
| رفع الملفات | WAF | منخفض: رفع shell PHP، عالٍ: تصفية الامتدادات |
| القوة الغاشمة | الدفاع ضد البوتات | محاولات تسجيل دخول آلية |

### مستويات الأمان

عيّن مستوى الأمان في `/dvwa/security.php` بعد تسجيل الدخول:

- **منخفض** -- لا يوجد تحقق من المدخلات. كل هجوم ينجح. جيد لعروض توقيعات WAF.
- **متوسط** -- تصفية أساسية. بعض الهجمات تتطلب ترميزاً أو تقنيات تجاوز.
- **عالٍ** -- تصفية قوية. فقط التقنيات المتقدمة تنجح. جيد لإظهار حدود WAF.
- **مستحيل** -- تنفيذ آمن بالكامل. يوضح البرمجة الدفاعية السليمة.

### الإعداد الأولي

يتطلب DVWA إعداد قاعدة بيانات لمرة واحدة بعد النشر الأول:

1. انتقل إلى `http://<PUBLIC_IP>/dvwa/setup.php`
2. انقر على **Create / Reset Database**
3. سجّل الدخول باستخدام `admin` / `password`

## VAmPI (واجهة برمجة تطبيقات معرضة للثغرات)

| | |
|---|---|
| **المسار** | `/vampi/` |
| **الصورة** | `erev0s/vampi:latest` مع نقطة دخول gunicorn (4 عمّال) |
| **النسخ** | 4 (المنافذ 5101-5104)، ثابتة عبر `ip_hash` (SQLite لكل نسخة) |
| **الموارد** | 0.5 وحدة معالجة / 512 ميبي بايت ذاكرة لكل نسخة |
| **إطار العمل** | Python / Flask / gunicorn |
| **المشروع** | [github.com/erev0s/VAmPI](https://github.com/erev0s/VAmPI) |

VAmPI مصمم خصيصاً لاختبار قائمة OWASP API Security Top 10. يوفر واجهة REST API واقعية مع ثغرات متعمدة. تعمل كل نسخة بـ gunicorn مع 4 عمّال وقاعدة بيانات SQLite خاصة بها. تضمن الجلسة الثابتة `ip_hash` أن التسجيل وتسجيل الدخول من نفس عنوان IP للعميل يصلان دائماً إلى نفس النسخة.

### سيناريوهات العرض التوضيحي

| السيناريو | OWASP API Top 10 | الطريقة |
|---|---|---|
| تصريح مكسور على مستوى الكائن (BOLA) | API1 | الوصول إلى بيانات مستخدمين آخرين عبر التلاعب بالمعرّفات |
| مصادقة مكسورة | API2 | معالجة ضعيفة للرموز، عدم وجود تحديد معدل |
| كشف مفرط للبيانات | API3 | API تُرجع بيانات أكثر مما يحتاجه العميل |
| التعيين الجماعي | API6 | تعديل حقول المسؤول عبر معاملات غير متوقعة |
| حقن SQL | API8 | الحقن من خلال معاملات API |
| إدارة أصول غير سليمة | API9 | نقاط نهاية API غير موثقة |

### نقاط نهاية API

```bash
# Register a new user
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123","email":"test@test.com"}'

# Login
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123"}'

# List users (excessive data exposure)
curl "http://<PUBLIC_IP>/vampi/users/v1"
```

## httpbin

| | |
|---|---|
| **المسار** | `/httpbin/` |
| **الصورة** | `kennethreitz/httpbin:latest` مع تجاوز CMD لـ gunicorn (`-w 4 -k gevent --timeout 30`) |
| **النسخ** | 4 (المنافذ 8201-8204)، توزيع دوري (بدون حالة) |
| **الموارد** | 0.5 وحدة معالجة / 256 ميبي بايت ذاكرة لكل نسخة |
| **إطار العمل** | Python / Flask / gunicorn + gevent |
| **المشروع** | [httpbin.org](https://httpbin.org) |

httpbin هو خدمة طلب/استجابة HTTP بسيطة مفيدة لعروض API الأساسية واختبار ترويسات الطلب والتحقق من سلوك الوكيل.

### نقاط النهاية المفيدة

| نقطة النهاية | الغرض |
|---|---|
| `/httpbin/get` | تُرجع بيانات طلب GET (الترويسات، المعاملات، المصدر) |
| `/httpbin/post` | تُرجع بيانات طلب POST (الجسم، النموذج، JSON) |
| `/httpbin/headers` | تُرجع ترويسات الطلب |
| `/httpbin/ip` | تُرجع عنوان IP المصدر |
| `/httpbin/user-agent` | تُرجع ترويسة User-Agent |
| `/httpbin/status/:code` | تُرجع رمز حالة HTTP المحدد |
| `/httpbin/delay/:seconds` | تؤخر الاستجابة بـ N ثانية |
| `/httpbin/anything` | تُرجع أي شيء تم تمريره في الطلب |

### مثال

```bash
# Check what headers the origin sees (useful for verifying F5 XC header injection)
curl -s "http://<PUBLIC_IP>/httpbin/headers" | jq .

# Test a specific HTTP status code
curl -s -o /dev/null -w "%{http_code}" "http://<PUBLIC_IP>/httpbin/status/403"
```

## whoami (تشخيصات الطلب)

| | |
|---|---|
| **المسار** | `/whoami/` |
| **الصورة** | `traefik/whoami:latest` |
| **النسخ** | 4 (المنافذ 8082-8085)، توزيع دوري (بدون حالة) |
| **الموارد** | 0.25 وحدة معالجة / 64 ميبي بايت ذاكرة لكل نسخة |
| **إطار العمل** | Go |
| **المشروع** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami هو خادم صدى طلبات خفيف الوزن من Traefik. يعرض كل تفاصيل طلب HTTP الوارد كما يراه الخادم الأصلي -- اسم المضيف، عناوين IP، جميع الترويسات، الطريقة، وعنوان URL. هذه هي أهم أداة تشخيصية للتحقق من أن F5 XC يحقن الترويسات الصحيحة.

### حالات الاستخدام الرئيسية

| حالة الاستخدام | ما يجب البحث عنه |
|---|---|
| التحقق من حقن ترويسات F5 XC | ترويسات `X-Forwarded-For`، `True-Client-IP`، `X-Volterra-*` |
| تأكيد رؤية عنوان IP للعميل | `X-Real-IP` مقابل `RemoteAddr` |
| تصحيح الإيجابيات الخاطئة لـ WAF | مقارنة ترويسات الطلب قبل وبعد F5 XC |
| التحقق من وسم الدفاع ضد البوتات | ترويسات `X-Volterra-Bot-Type`، `X-Volterra-Bot-Verified` |
| فحص إنهاء TLS | `X-Forwarded-Proto` يُظهر `https` عندما ينتهي TLS عند F5 XC |

### مثال

```bash
# Basic request -- see what the origin receives
curl "http://<PUBLIC_IP>/whoami/"

# Simulate request through F5 XC (with injected headers)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

مثال على المخرجات:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA (تطبيق GraphQL المعرض للثغرات)

| | |
|---|---|
| **المسار** | `/dvga/` |
| **الصورة** | `dolevf/dvga:latest` |
| **النسخ** | 4 (المنافذ 5201-5204)، ثابتة عبر `ip_hash` (SQLite لكل نسخة) |
| **الموارد** | 0.5 وحدة معالجة / 256 ميبي بايت ذاكرة لكل نسخة |
| **إطار العمل** | Python / Flask / GraphQL |
| **المشروع** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA مصمم خصيصاً لاختبار الثغرات الخاصة بـ GraphQL. يوفر واجهة GraphiQL في `/dvga/` لاستكشاف الاستعلامات التفاعلي ونقطة نهاية GraphQL API في `/dvga/graphql`. تستخدم كل نسخة قاعدة بيانات SQLite خاصة بها، لذا تضمن الجلسات الثابتة `ip_hash` حالة متسقة.

### سيناريوهات العرض التوضيحي

| السيناريو | الفئة | متجه الهجوم |
|---|---|---|
| حقن GraphQL | أمان API | استعلامات خبيثة عبر تضمين السلاسل النصية |
| رفض الخدمة | أمان API | استعلامات متداخلة بعمق، استعلامات مجمعة، استنزاف الموارد |
| تجاوز التصريح | أمان API | الوصول إلى بيانات غير مصرح بها عبر GraphQL |
| كشف المعلومات | أمان API | استعلامات الاستبطان التي تكشف تفاصيل المخطط |
| هجوم التجميع | أمان API | عمليات متعددة في طلب واحد |

### نقاط النهاية الرئيسية

```bash
# GraphiQL UI (interactive IDE)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# Introspection query -- enumerate the full schema
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# List pastes (example query)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# Create a paste (mutation)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

RESTaurant هو واجهة REST API معرضة للثغرات بأسلوب اللعب تغطي OWASP API Security Top 10 2023. يستخدم FastAPI مع واجهة Swagger UI التوثيقية التلقائية في `/restaurant/docs`. تتشارك جميع النسخ الأربع في قاعدة بيانات PostgreSQL واحدة، لذا يعمل توزيع الحمل الدوري بدون جلسات ثابتة.

### سيناريوهات العرض التوضيحي

| السيناريو | OWASP API Top 10 2023 | الطريقة |
|---|---|---|
| تصريح مكسور على مستوى الكائن (BOLA) | API1 | الوصول إلى طلبات مستخدمين آخرين عبر التلاعب بالمعرّفات |
| مصادقة مكسورة | API2 | معالجة ضعيفة للرموز، حشو بيانات الاعتماد |
| تصريح مكسور على مستوى خاصية الكائن | API3 | التعيين الجماعي على حقول الملف الشخصي للمستخدم |
| استهلاك غير مقيد للموارد | API4 | عدم وجود تحديد معدل على نقاط النهاية |
| تصريح مكسور على مستوى الوظيفة (BFLA) | API5 | الوصول إلى نقاط نهاية المسؤول كمستخدم عادي |
| تزوير الطلب من جانب الخادم (SSRF) | API7 | التلاعب بطلبات URL من جانب الخادم |
| تكوين أمني خاطئ | API8 | رسائل خطأ مفصلة، بيانات اعتماد افتراضية |

### نقاط النهاية الرئيسية

```bash
# Swagger UI
curl -sf "http://<PUBLIC_IP>/restaurant/docs" -o /dev/null -w "%{http_code}"

curl -s "http://<PUBLIC_IP>/restaurant/openapi.json" | jq .info

curl -s "http://<PUBLIC_IP>/restaurant/orders/1" \
```

## crAPI (واجهة برمجة التطبيقات السخيفة تماماً من OWASP)

| | |
|---|---|
| **المنفذ** | `8888` (مخصص -- ليس ببادئة مسار) |
| **الصور** | `crapi/crapi-web`، `crapi/crapi-identity`، `crapi/crapi-community`، `crapi/crapi-workshop`، PostgreSQL، MongoDB، MailHog |
| **النسخ** | 7 خدمات مصغرة (نسخة واحدة لكل منها) |
| **الموارد** | ~3.0 وحدة معالجة / ~2.0 جيبي بايت ذاكرة إجمالاً |
| **إطار العمل** | React SPA + خدمات مصغرة Java/Go/Python |
| **المشروع** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI هو المشروع الرائد من OWASP لاختبار أمان API. يعمل كـ 7 خدمات مصغرة على منفذ مخصص (8888) لأن تطبيق React SPA يُضمّن مسارات API الخاصة به بشكل ثابت ولا يمكن تقديمه خلف بادئة مسار. يسمح NSG بالحركة الواردة على المنفذ 8888.

يلتقط MailHog جميع رسائل البريد الإلكتروني المرسلة بواسطة crAPI (التحقق من الحساب، إعادة تعيين كلمة المرور). يمكن الوصول إلى MailHog عبر نفق SSH على المنفذ 18025.

### فئات التحديات

| الفئة | الثغرات |
|---|---|
| BOLA (تصريح مكسور على مستوى الكائن) | الوصول إلى مركبات ومطالبات وتقارير مستخدمين آخرين |
| BFLA (تصريح مكسور على مستوى الوظيفة) | التصعيد إلى مسؤول، الوصول إلى نقاط نهاية مقيدة |
| التعيين الجماعي | تعديل حقول محمية (الدور، الرصيد) عبر API |
| SSRF (تزوير الطلب من جانب الخادم) | التلاعب بجلب URL من جانب الخادم |
| التلاعب بـ JWT | تزوير أو تعديل رموز JWT لتصعيد الصلاحيات |
| حقن NoSQL | حقن استعلامات في نقاط نهاية مدعومة بـ MongoDB |
| كشف مفرط للبيانات | API تُرجع بيانات مستخدم حساسة |

### البدء

```bash
# Verify crAPI is running
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# Sign up a new user
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# Log in
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# Access MailHog (via SSH tunnel for email verification)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# Then open http://localhost:18025 in your browser
```

### بنية crAPI

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, user auth, JWT)
             -> crapi-community (Go, forums, posts)
             -> crapi-workshop (Python, vehicle service)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (email capture, port 18025)
```
