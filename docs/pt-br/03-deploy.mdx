Todos os arquivos Terraform estão no diretório [`terraform/`](https://github.com/f5xc-salesdemos/origin-server/tree/main/terraform). Clone o repositório e faça o deploy diretamente:

```bash
git clone https://github.com/f5xc-salesdemos/origin-server.git
cd origin-server/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and SSH key path
```

## Configuração do Terraform

### Estrutura de Arquivos Terraform

O diretório terraform contém 9 arquivos seguindo o Demo Resource Standard:

- `versions.tf` — Restrições de versão do Terraform e dos providers (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Configuração dos providers Azure RM e Azure AD
- `data.tf` — Fontes de dados do Azure AD para resolução automática do deployer
- `locals.tf` — Resolução do deployer, nomenclatura de recursos do Azure Cloud Adoption Framework, tags padrão
- `main.tf` — Grupo de recursos (nomeado `rg-origin-server-{environment}-{deployer}`)
- `variables.tf` — Todas as variáveis de entrada (1 obrigatória, 8 opcionais)
- `network.tf` — VNet (10.200.0.0/16), subnet, IP público, NSG (portas 22/80/443/8888), NIC
- `vm.tf` — VM Ubuntu 24.04 com cloud-init via templatefile()
- `outputs.tf` — 25 outputs (15 padrão + 10 URLs de aplicações específicas do componente)

`main.tf` contém apenas o grupo de recursos:

```hcl file=../_data/main.tf
```

`variables.tf` define todos os parâmetros configuráveis. O identificador `deployer` é resolvido automaticamente a partir da sua conta Azure AD — você só precisa definir `subscription_id`. O padrão de `vm_size` é `Standard_D16s_v3` (16 vCPU, 64 GiB RAM), dimensionado para 41 contêineres Docker:

```hcl file=../_data/variables.tf
```

### Infraestrutura de Rede

`network.tf` cria a VNet, subnet, IP público, NSG (portas 22/80/443/8888) e NIC. A porta 8888 é necessária para o crAPI, que é executado em uma porta dedicada:

```hcl file=../_data/network.tf
```

### Máquina Virtual com Cloud-Init

`vm.tf` cria a VM Ubuntu 24.04 e passa o script de provisionamento cloud-init. O SSD Premium de 60 GiB fornece espaço suficiente para imagens Docker e volumes de contêineres:

```hcl file=../_data/vm.tf
```

### Provisionamento Cloud-Init

`cloud-init.yaml` provisiona a VM com uma configuração robusta. Uma biblioteca auxiliar compartilhada (`/usr/local/lib/cloud-init-helpers.sh`) fornece lógica de repetição em todas as operações de rede e registro de progresso em `/var/log/cloud-init-progress.log`. O polling de verificação de integridade ativo substitui o sleep de duração fixa para a prontidão dos contêineres. Ele provisiona:

- **Ajuste de kernel**: TCP keepalive, reutilização de TIME_WAIT, 2M tw_buckets, 64K somaxconn
- **nginx**: 8 blocos upstream com sessões persistentes (ip_hash, cookie hash), pools keepalive (64--128), cache de proxy para o Juice Shop, nível error_log crit
- **Docker Compose**: 41 contêineres em 9 aplicações com limites de CPU/memória por contêiner
- **Builds personalizados**: DVWA-FPM (php-fpm com pm.max_requests=200), CSD Demo (Flask + gunicorn), RESTaurant (clonado do GitHub)
- **Reciclagem de workers**: uvicorn --limit-max-requests 200 (RESTaurant), pm.max_requests 200 (DVWA-FPM) para evitar vazamentos de memória sob carga sustentada

```yaml file=../_data/cloud-init.yaml
```

## Deploy

```bash
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and SSH key path

terraform init

terraform plan

terraform apply
```

O Terraform exibe o IP público, o comando SSH e as URLs das aplicações após um deploy bem-sucedido.

## Pós-Deploy

Após a conclusão do `terraform apply`, aguarde de 5 a 10 minutos para que o cloud-init termine de instalar o Docker, baixar as imagens de contêiner e configurar o nginx. Os pulls de imagens Docker são o gargalo — somente o Juice Shop tem cerca de 400 MiB.

Verifique o endpoint de integridade:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

Resposta esperada:

```json
{
  "status": "healthy",
  "component": "origin-server",
  "applications": ["juice-shop", "dvwa", "vampi", "httpbin", "whoami", "csd-demo", "dvga", "restaurant", "crapi"]
}
```

### Configuração do Banco de Dados DVWA

O DVWA requer uma inicialização única do banco de dados. Abra `http://<PUBLIC_IP>/dvwa/setup.php` em um navegador e clique em **Create / Reset Database**. As credenciais padrão são `admin` / `password`.

:::note
O banco de dados SQLite do VAmPI é inicializado automaticamente pelo seu script de entrada gunicorn. Cada contêiner executa `db.create_all()` na inicialização, portanto nenhuma intervenção manual é necessária.
:::

## Conectando aos Componentes Downstream

Após o deploy, outros componentes utilizam os outputs do servidor de origem como suas entradas:

```bash
# CDN Simulator — pass origin-server's public IP as its upstream
cd ../cdn-simulator/terraform
origin_ip=$(cd ../../origin-server/terraform && terraform output -raw public_ip)
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF

# Traffic Generator — pass the F5 XC load balancer FQDN (not the origin IP directly)
cd ../traffic-generator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
target_fqdn     = "your-xc-load-balancer.example.com"
EOF
```

| Output | Componente Downstream | Variável de Entrada |
|---|---|---|
| `public_ip` | CDN Simulator | `origin_server` (com prefixo `http://`) |
| `public_ip` | CDN Simulator | `origin_host` (com sufixo `:80`, sem scheme) |
| `public_ip` | F5 XC Origin Pool | Endereço do servidor de origem |
