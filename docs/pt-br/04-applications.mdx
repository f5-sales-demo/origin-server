## Visão Geral das Aplicações

Cada aplicação é executada como 4 contêineres Docker com balanceamento de carga por trás de pools upstream do nginx, acessíveis através do proxy reverso nginx na VM. Sessões fixas (sticky sessions) garantem que aplicações com estado (Juice Shop, DVWA, VAmPI, CSD Demo) sejam roteadas de forma consistente. Todas as aplicações são intencionalmente vulneráveis e projetadas para testes de segurança.

Substitua `<ORIGIN>` por `http://<PUBLIC_IP>` em todos os exemplos abaixo. Após implantar via Terraform, obtenha o IP com `terraform output -raw public_ip`.

| **Imagem** | `bkimminich/juice-shop:latest` |

OWASP Juice Shop é a aplicação web vulnerável mais moderna e ativamente mantida. Ela cobre todo o OWASP Top 10 com mais de 100 desafios em uma aplicação de e-commerce realista.

### Cenários de Demonstração

| Cenário | Categoria | Vetor de Ataque |
|---|---|---|
| Bypass de login por SQL Injection | WAF | `' OR 1=1--` no campo de email do login |
| XSS Refletido | WAF, CSD | Injeção de script via parâmetro de busca |
| XSS baseado em DOM | CSD | Payload no fragmento da URL |
| Autenticação quebrada | WAF | Força bruta de login, manipulação de JWT |
| Abuso de API | Segurança de API | Acesso não autorizado a endpoints `/api/` |
| Exposição de dados sensíveis | Segurança de API | Acesso a dados de usuários sem autorização |
| CSRF | WAF | Cross-site request forgery em alterações de perfil |

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **Caminho** | `/dvwa/` |
| **Imagem** | Custom `dvwa-fpm:latest` (php-fpm + nginx, construída a partir de `ghcr.io/digininja/dvwa:latest`) |
| **Instâncias** | 4 (portas 8101-8104), fixas via `hash $cookie_PHPSESSID` |
| **Recursos** | 0.5 CPU / 256 MiB RAM por instância |
| **Banco de Dados** | MariaDB 10.11 compartilhado (contêiner `dvwa-db`, 1 CPU / 768 MiB) |
| **Framework** | PHP 8 / php-fpm / MariaDB |
| **Credenciais** | `admin` / `password` |

DVWA é o padrão da indústria para testes de WAF. Possui níveis de segurança ajustáveis (Low, Medium, High, Impossible) que progressivamente adicionam validação de entrada e codificação de saída.

### Cenários de Demonstração

| Cenário | Categoria | Nível de Segurança |
|---|---|---|
| SQL Injection | WAF | Low: trivial `' OR 1=1#`, High: SQLi cego |
| Injeção de Comando | WAF | Low: `; ls`, High: caracteres filtrados |
| Inclusão de Arquivo | WAF | Low: traversal de caminho direto, High: sanitizado |
| XSS (Refletido) | WAF, CSD | Low: básico `<script>`, High: bypass codificado |
| XSS (Armazenado) | WAF, CSD | Low: script persistente no livro de visitas |
| Upload de Arquivo | WAF | Low: upload de shell PHP, High: filtragem de extensão |
| Força Bruta | Defesa contra Bots | Tentativas automatizadas de login |

### Níveis de Segurança

Defina o nível de segurança em `/dvwa/security.php` após fazer login:

- **Low** -- Sem validação de entrada. Todos os ataques funcionam. Bom para demonstrações de assinaturas de WAF.
- **Medium** -- Filtragem básica. Alguns ataques requerem codificação ou técnicas de bypass.
- **High** -- Filtragem forte. Apenas técnicas avançadas têm sucesso. Bom para mostrar limitações do WAF.
- **Impossible** -- Implementação totalmente segura. Demonstra codificação defensiva adequada.

### Configuração Inicial

O DVWA requer uma configuração única do banco de dados após a primeira implantação:

1. Navegue até `http://<PUBLIC_IP>/dvwa/setup.php`
2. Clique em **Create / Reset Database**
3. Faça login com `admin` / `password`

| Gerenciamento Inadequado de Ativos | API9 | Endpoints de API não documentados |

## whoami (Diagnóstico de Requisições)

| | |
|---|---|
| **Caminho** | `/whoami/` |
| **Imagem** | `traefik/whoami:latest` |
| **Instâncias** | 4 (portas 8082-8085), round-robin (sem estado) |
| **Recursos** | 0.25 CPU / 64 MiB RAM por instância |
| **Framework** | Go |
| **Projeto** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami é o servidor leve de eco de requisições do Traefik. Ele exibe cada detalhe da requisição HTTP recebida conforme a origem a vê -- hostname, endereços IP, todos os cabeçalhos, método e URL. Esta é a ferramenta de diagnóstico mais importante ao verificar se o F5 XC está injetando os cabeçalhos corretos.

### Casos de Uso Principais

| Caso de Uso | O Que Observar |
|---|---|
| Verificar injeção de cabeçalhos do F5 XC | Cabeçalhos `X-Forwarded-For`, `True-Client-IP`, `X-Volterra-*` |
| Confirmar visibilidade do IP do cliente | `X-Real-IP` vs `RemoteAddr` |
| Depurar falsos positivos do WAF | Comparar cabeçalhos da requisição antes/depois do F5 XC |
| Validar marcação de defesa contra bots | Cabeçalhos `X-Volterra-Bot-Type`, `X-Volterra-Bot-Verified` |
| Verificar terminação TLS | `X-Forwarded-Proto` mostra `https` quando TLS termina no F5 XC |

### Exemplo

```bash
# Requisição básica -- veja o que a origem recebe
curl "http://<PUBLIC_IP>/whoami/"

# Simular requisição através do F5 XC (com cabeçalhos injetados)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

Exemplo de saída:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

| | |
|---|---|

DVGA foi construído especificamente para testar vulnerabilidades específicas do GraphQL. Fornece uma IDE GraphiQL em `/dvga/` para exploração interativa de consultas e um endpoint de API GraphQL em `/dvga/graphql`. Cada instância usa seu próprio banco de dados SQLite, então sessões fixas `ip_hash` garantem estado consistente.

### Cenários de Demonstração

| Cenário | Categoria | Vetor de Ataque |
|---|---|---|
| Injeção GraphQL | Segurança de API | Consultas maliciosas via interpolação de string |
| Negação de Serviço | Segurança de API | Consultas profundamente aninhadas, consultas em lote, exaustão de recursos |
| Bypass de Autorização | Segurança de API | Acesso a dados não autorizados através do GraphQL |
| Divulgação de Informações | Segurança de API | Consultas de introspecção revelando detalhes do schema |
| Ataque de Lote | Segurança de API | Múltiplas operações em uma única requisição |

### Endpoints Principais

```bash
# GraphiQL UI (IDE interativa)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# Consulta de introspecção -- enumerar o schema completo
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# Listar pastes (consulta de exemplo)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# Criar um paste (mutação)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **Porta** | `8888` (dedicada -- sem prefixo de caminho) |
| **Imagens** | `crapi/crapi-web`, `crapi/crapi-identity`, `crapi/crapi-community`, `crapi/crapi-workshop`, PostgreSQL, MongoDB, MailHog |
| **Instâncias** | 7 microsserviços (instância única cada) |
| **Recursos** | ~3.0 CPU / ~2.0 GiB RAM total |
| **Framework** | React SPA + microsserviços Java/Go/Python |
| **Projeto** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI é o projeto emblemático da OWASP para testes de segurança de API. Executa como 7 microsserviços em uma porta dedicada (8888) porque o React SPA tem seus caminhos de API codificados de forma fixa e não pode ser servido por trás de um prefixo de caminho. O NSG permite tráfego de entrada na porta 8888.

MailHog captura todos os emails enviados pelo crAPI (verificação de conta, redefinição de senha). Acesse o MailHog via túnel SSH na porta 18025.

### Categorias de Desafios

| Categoria | Vulnerabilidades |
|---|---|
| BOLA (Broken Object Level Authorization) | Acessar veículos, pedidos e relatórios de outros usuários |
| BFLA (Broken Function Level Authorization) | Escalar para admin, acessar endpoints restritos |
| Mass Assignment | Modificar campos protegidos (role, saldo) via API |
| SSRF (Server Side Request Forgery) | Manipular busca de URL do lado do servidor |
| Manipulação de JWT | Forjar ou modificar tokens JWT para escalação de privilégios |
| Injeção NoSQL | Injetar consultas em endpoints baseados em MongoDB |
| Exposição Excessiva de Dados | API retorna dados sensíveis de usuários |

### Primeiros Passos

```bash
# Verificar se o crAPI está em execução
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# Cadastrar um novo usuário
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# Fazer login
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# Acessar MailHog (via túnel SSH para verificação de email)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# Então abra http://localhost:18025 no seu navegador
```

### Arquitetura do crAPI

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, autenticação de usuário, JWT)
             -> crapi-community (Go, fóruns, posts)
             -> crapi-workshop (Python, serviço de veículos)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (captura de email, porta 18025)
```
