## Integração com Origin Pool do F5 XC

Este servidor de origem foi projetado para ser utilizado como um **membro do origin pool** atrás de um HTTP load balancer do F5 Distributed Cloud. A plataforma F5 XC fornece WAF, Bot Defense, API Security e Client-Side Defense na frente desta origem.

### Arquitetura

```
End User -> F5 XC HTTP LB (WAF/Bot/API/CSD) -> Origin Server VM (nginx -> Docker apps)
```

### Configuração do Origin Pool

Crie um origin pool do F5 XC apontando para o IP público desta VM:

| Configuração | Valor |
|---|---|
| **Origin Server Type** | Public IP of Origin Server |
| **IP Address** | `<terraform output public_ip>` |
| **Port** | `80` |
| **Health Check** | HTTP, path `/health` |

### Configuração do HTTP Load Balancer

Crie um HTTP load balancer com o origin pool:

| Configuração | Valor |
|---|---|
| **Domains** | Seu domínio de demonstração (ex.: `demo.example.com`) |
| **Origin Pool** | O pool criado acima |
| **WAF** | Anexe sua política de WAF |
| **Bot Defense** | Habilite conforme necessário |
| **API Discovery** | Habilite para os endpoints do VAmPI |

### Roteamento Baseado em Caminho

Cada aplicação é acessível por meio de seu prefixo de caminho através do load balancer:

| URL do F5 XC LB | Caminho na Origem | Aplicação |
|---|---|---|
| `https://demo.example.com/juice-shop/` | `/juice-shop/` | Juice Shop (4 instâncias, sticky por cookie) |

### Verificar Injeção de Headers do F5 XC

Use o endpoint whoami para verificar quais headers o F5 XC injeta nas requisições que chegam à origem:

```bash
LB_DOMAIN="demo.example.com"

curl -sk "https://${LB_DOMAIN}/whoami/"
```

Procure por estes headers injetados pelo F5 XC na resposta:

| Header | Significado |
|---|---|
| `X-Forwarded-For` | Cadeia de IPs do cliente através do F5 XC |
| `True-Client-IP` | IP original do cliente |
| `X-Forwarded-Proto` | `https` se o TLS é terminado no F5 XC |
| `X-Volterra-Bot-Type` | Classificação de bot (quando Bot Defense está habilitado) |
| `X-Request-ID` | ID de rastreamento de requisição do F5 XC |

### Teste de WAF Através do F5 XC

Com uma política de WAF anexada ao HTTP load balancer, teste payloads de ataque:

```bash
LB_DOMAIN="demo.example.com"

# SQL Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--&Submit=Submit"

# XSS (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/juice-shop/rest/products/search?q=<script>alert(1)</script>"

# Command Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/exec/" \
  -d "ip=127.0.0.1;cat+/etc/passwd&Submit=Submit"

# Normal request (should pass through)
curl -sk "https://${LB_DOMAIN}/httpbin/get" | jq .
```

### Teste de API Security

Habilite o API Discovery no HTTP load balancer para mapear os endpoints da API do VAmPI:

```bash
# These requests build the API inventory in F5 XC
curl -sk "https://${LB_DOMAIN}/vampi/users/v1"
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123","email":"api@test.com"}'
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123"}'
```

### Integração do crAPI (Porta 8888)

O crAPI executa em uma porta dedicada (8888) porque é uma aplicação de página única que possui caminhos de API fixos no código e não pode ser servida atrás de um prefixo de caminho. Para integrar o crAPI com o F5 XC:

| Configuração | Valor |
|---|---|
| **Origin Server Type** | Public IP of Origin Server |
| **IP Address** | `<terraform output public_ip>` |
| **Port** | `8888` |
| **Health Check** | HTTP, path `/` |

Crie um **origin pool separado** para o crAPI na porta 8888, ou adicione um segundo membro de origin pool ao seu pool existente com a porta 8888 e use regras de rota para direcionar o tráfego:

```bash
# Test crAPI through F5 XC (if configured)
curl -sk "https://${LB_DOMAIN}:8888/"

# Or if using route rules on the same LB domain:
# Configure an F5 XC route rule matching Host header or path prefix
# to forward to the crAPI origin pool (port 8888)
```

:::note
O NSG na VM de origem permite tráfego de entrada na porta 8888. Nenhuma alteração adicional na rede do Azure é necessária.
:::

### Consciência sobre Sessões Sticky

O servidor de origem utiliza sessões sticky do nginx internamente para rotear aplicações com estado para contêineres backend consistentes. Ao configurar o HTTP load balancer do F5 XC, esteja ciente:

| Aplicação | Método Sticky | Motivo |
|---|---|---|
| Juice Shop | `hash $cookie_token` | Estado de sessão Node.js |
| DVWA | `hash $cookie_PHPSESSID` | Estado de sessão PHP |
| VAmPI | `ip_hash` | Banco de dados SQLite por instância |
| CSD Demo | `ip_hash` | Log de exfiltração em memória por instância |
| DVGA | `ip_hash` | Banco de dados SQLite por instância |
| RESTaurant | Round-robin | Backend PostgreSQL compartilhado |
| crAPI | -- (porta única 8888) | 7 microsserviços, PostgreSQL + MongoDB |
| httpbin | Round-robin | Stateless |
| whoami | Round-robin | Stateless |

O F5 XC não precisa replicar estas sessões sticky -- o nginx na VM de origem gerencia o roteamento do backend. O F5 XC deve tratar a origem como um único endpoint (o IP público da VM na porta 80).

### Arquitetura Multi-Componente

Quando combinado com o CDN Simulator, a arquitetura completa do laboratório é:

```
End User -> CDN Simulator (nginx cache) -> F5 XC HTTP LB -> Origin Server (this component)
```

O CDN Simulator utiliza o VIP do HTTP load balancer do F5 XC como sua origem, e o load balancer do F5 XC utiliza este servidor de origem como seu backend. Isso cria uma arquitetura multi-camada realista para testes abrangentes.
