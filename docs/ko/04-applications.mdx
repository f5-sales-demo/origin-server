## 애플리케이션 개요

각 애플리케이션은 nginx 업스트림 풀 뒤에서 4개의 로드 밸런싱된 Docker 컨테이너로 실행되며, VM의 nginx 리버스 프록시를 통해 접근할 수 있습니다. 스티키 세션은 상태를 유지하는 애플리케이션(Juice Shop, DVWA, VAmPI, CSD Demo)이 일관되게 라우팅되도록 보장합니다. 모든 애플리케이션은 의도적으로 취약하며 보안 테스트용으로 설계되었습니다.

아래 모든 예제에서 `<ORIGIN>`을 `http://<PUBLIC_IP>`로 대체하세요. Terraform을 통해 배포한 후 `terraform output -raw public_ip`로 IP를 확인할 수 있습니다.

## OWASP Juice Shop

| | |
|---|---|
| **경로** | `/juice-shop/` |
| **이미지** | `bkimminich/juice-shop:latest` |
| **인스턴스** | 4개 (포트 3001-3004), `hash $cookie_token`으로 스티키, 프록시 캐시 (60초 TTL) |
| **리소스** | 인스턴스당 2 CPU / 1 GiB RAM |
| **프레임워크** | Node.js / Angular |
| **프로젝트** | [owasp.org/www-project-juice-shop](https://owasp.org/www-project-juice-shop/) |

OWASP Juice Shop은 가장 현대적이고 활발하게 유지보수되는 취약한 웹 애플리케이션입니다. 현실적인 전자상거래 애플리케이션에서 100개 이상의 챌린지로 OWASP Top 10 전체를 다룹니다.

### 데모 시나리오

| 시나리오 | 카테고리 | 공격 벡터 |
|---|---|---|
| SQL 인젝션 로그인 우회 | WAF | 로그인 이메일 필드에 `' OR 1=1--` |
| 반사형 XSS | WAF, CSD | 검색 파라미터를 통한 스크립트 인젝션 |
| DOM 기반 XSS | CSD | URL 프래그먼트 내 페이로드 |
| 인증 취약점 | WAF | 무차별 대입 로그인, JWT 조작 |
| API 남용 | API 보안 | `/api/` 엔드포인트에 대한 비인가 접근 |
| 민감 데이터 노출 | API 보안 | 인가 없이 사용자 데이터 접근 |
| CSRF | WAF | 프로필 변경에 대한 크로스 사이트 요청 위조 |

### 접근

```bash
curl -s "http://<PUBLIC_IP>/juice-shop/" -o /dev/null -w "%{http_code}"
```

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **경로** | `/dvwa/` |
| **이미지** | 커스텀 `dvwa-fpm:latest` (php-fpm + nginx, `ghcr.io/digininja/dvwa:latest`에서 빌드) |
| **인스턴스** | 4개 (포트 8101-8104), `hash $cookie_PHPSESSID`로 스티키 |
| **리소스** | 인스턴스당 0.5 CPU / 256 MiB RAM |
| **데이터베이스** | 공유 MariaDB 10.11 (`dvwa-db` 컨테이너, 1 CPU / 768 MiB) |
| **프레임워크** | PHP 8 / php-fpm / MariaDB |
| **자격 증명** | `admin` / `password` |

DVWA는 WAF 테스트를 위한 업계 표준입니다. 조정 가능한 보안 레벨(Low, Medium, High, Impossible)이 있어 점진적으로 입력 검증과 출력 인코딩이 추가됩니다.

### 데모 시나리오

| 시나리오 | 카테고리 | 보안 레벨 |
|---|---|---|
| SQL 인젝션 | WAF | Low: 간단한 `' OR 1=1#`, High: 블라인드 SQLi |
| 커맨드 인젝션 | WAF | Low: `; ls`, High: 필터링된 문자 |
| 파일 인클루전 | WAF | Low: 직접 경로 탐색, High: 살균 처리 |
| XSS (반사형) | WAF, CSD | Low: 기본 `<script>`, High: 인코딩 우회 |
| XSS (저장형) | WAF, CSD | Low: 방명록에 영구 스크립트 |
| 파일 업로드 | WAF | Low: PHP 쉘 업로드, High: 확장자 필터링 |
| 무차별 대입 | 봇 방어 | 자동화된 로그인 시도 |

### 보안 레벨

로그인 후 `/dvwa/security.php`에서 보안 레벨을 설정하세요:

- **Low** -- 입력 검증 없음. 모든 공격이 동작합니다. WAF 시그니처 시연에 적합합니다.
- **Medium** -- 기본 필터링. 일부 공격은 인코딩 또는 우회 기법이 필요합니다.
- **High** -- 강력한 필터링. 고급 기법만 성공합니다. WAF의 한계를 보여주기에 적합합니다.
- **Impossible** -- 완전히 안전한 구현. 올바른 방어적 코딩을 시연합니다.

### 초기 설정

DVWA는 최초 배포 후 일회성 데이터베이스 설정이 필요합니다:

1. `http://<PUBLIC_IP>/dvwa/setup.php`로 이동합니다
2. **Create / Reset Database**를 클릭합니다
3. `admin` / `password`로 로그인합니다

## VAmPI (Vulnerable API)

| | |
|---|---|
| **경로** | `/vampi/` |
| **이미지** | gunicorn 엔트리포인트가 있는 `erev0s/vampi:latest` (4 워커) |
| **인스턴스** | 4개 (포트 5101-5104), `ip_hash`로 스티키 (인스턴스당 SQLite) |
| **리소스** | 인스턴스당 0.5 CPU / 512 MiB RAM |
| **프레임워크** | Python / Flask / gunicorn |
| **프로젝트** | [github.com/erev0s/VAmPI](https://github.com/erev0s/VAmPI) |

VAmPI는 OWASP API 보안 Top 10 테스트를 위해 특별히 설계되었습니다. 의도적인 취약점이 있는 현실적인 REST API를 제공합니다. 각 인스턴스는 4개의 워커가 있는 gunicorn과 자체 SQLite 데이터베이스를 실행합니다. `ip_hash` 스티키 세션은 동일한 클라이언트 IP에서의 등록과 로그인이 항상 동일한 인스턴스에 도달하도록 보장합니다.

### 데모 시나리오

| 시나리오 | OWASP API Top 10 | 방법 |
|---|---|---|
| 객체 수준 인가 취약점 (BOLA) | API1 | ID를 조작하여 다른 사용자의 데이터에 접근 |
| 인증 취약점 | API2 | 취약한 토큰 처리, 속도 제한 없음 |
| 과도한 데이터 노출 | API3 | 클라이언트에 필요한 것 이상의 데이터를 API가 반환 |
| 대량 할당 | API6 | 예상치 못한 파라미터를 통해 관리자 필드 수정 |
| SQL 인젝션 | API8 | API 파라미터를 통한 인젝션 |
| 부적절한 자산 관리 | API9 | 문서화되지 않은 API 엔드포인트 |

### API 엔드포인트

```bash
# 새 사용자 등록
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123","email":"test@test.com"}'

# 로그인
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123"}'

# 사용자 목록 (과도한 데이터 노출)
curl "http://<PUBLIC_IP>/vampi/users/v1"
```

## httpbin

| | |
|---|---|
| **경로** | `/httpbin/` |
| **이미지** | gunicorn CMD 오버라이드가 있는 `kennethreitz/httpbin:latest` (`-w 4 -k gevent --timeout 30`) |
| **인스턴스** | 4개 (포트 8201-8204), 라운드 로빈 (스테이트리스) |
| **리소스** | 인스턴스당 0.5 CPU / 256 MiB RAM |
| **프레임워크** | Python / Flask / gunicorn + gevent |
| **프로젝트** | [httpbin.org](https://httpbin.org) |

httpbin은 기본 API 데모, 요청 헤더 테스트, 프록시 동작 확인에 유용한 간단한 HTTP 요청/응답 서비스입니다.

### 유용한 엔드포인트

| 엔드포인트 | 용도 |
|---|---|
| `/httpbin/get` | GET 요청 데이터 반환 (헤더, 인수, 오리진) |
| `/httpbin/post` | POST 요청 데이터 반환 (본문, 폼, JSON) |
| `/httpbin/headers` | 요청 헤더 반환 |
| `/httpbin/ip` | 오리진 IP 반환 |
| `/httpbin/user-agent` | User-Agent 헤더 반환 |
| `/httpbin/status/:code` | 지정된 HTTP 상태 코드 반환 |
| `/httpbin/delay/:seconds` | N초만큼 응답 지연 |
| `/httpbin/anything` | 요청에 전달된 모든 것을 반환 |

### 예제

```bash
# 오리진이 어떤 헤더를 보는지 확인 (F5 XC 헤더 인젝션 확인에 유용)
curl -s "http://<PUBLIC_IP>/httpbin/headers" | jq .

# 특정 HTTP 상태 코드 테스트
curl -s -o /dev/null -w "%{http_code}" "http://<PUBLIC_IP>/httpbin/status/403"
```

## whoami (요청 진단)

| | |
|---|---|
| **경로** | `/whoami/` |
| **이미지** | `traefik/whoami:latest` |
| **인스턴스** | 4개 (포트 8082-8085), 라운드 로빈 (스테이트리스) |
| **리소스** | 인스턴스당 0.25 CPU / 64 MiB RAM |
| **프레임워크** | Go |
| **프로젝트** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami는 Traefik의 경량 요청 에코 서버입니다. 오리진이 보는 수신 HTTP 요청의 모든 세부 정보(호스트명, IP 주소, 모든 헤더, 메서드, URL)를 표시합니다. F5 XC가 올바른 헤더를 인젝션하는지 확인할 때 가장 중요한 단일 진단 도구입니다.

### 주요 사용 사례

| 사용 사례 | 확인할 사항 |
|---|---|
| F5 XC 헤더 인젝션 확인 | `X-Forwarded-For`, `True-Client-IP`, `X-Volterra-*` 헤더 |
| 클라이언트 IP 가시성 확인 | `X-Real-IP` vs `RemoteAddr` |
| WAF 오탐 디버그 | F5 XC 전후의 요청 헤더 비교 |
| 봇 방어 태깅 검증 | `X-Volterra-Bot-Type`, `X-Volterra-Bot-Verified` 헤더 |
| TLS 종료 확인 | F5 XC에서 TLS가 종료될 때 `X-Forwarded-Proto`가 `https`를 표시 |

### 예제

```bash
# 기본 요청 -- 오리진이 수신하는 내용 확인
curl "http://<PUBLIC_IP>/whoami/"

# F5 XC를 통한 요청 시뮬레이션 (인젝션된 헤더 포함)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

출력 예시:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA (Damn Vulnerable GraphQL Application)

| | |
|---|---|
| **경로** | `/dvga/` |
| **이미지** | `dolevf/dvga:latest` |
| **인스턴스** | 4개 (포트 5201-5204), `ip_hash`로 스티키 (인스턴스당 SQLite) |
| **리소스** | 인스턴스당 0.5 CPU / 256 MiB RAM |
| **프레임워크** | Python / Flask / GraphQL |
| **프로젝트** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA는 GraphQL 특화 취약점 테스트를 위해 특별히 설계되었습니다. `/dvga/`에서 대화형 쿼리 탐색을 위한 GraphiQL IDE를 제공하고 `/dvga/graphql`에서 GraphQL API 엔드포인트를 제공합니다. 각 인스턴스는 자체 SQLite 데이터베이스를 사용하므로 `ip_hash` 스티키 세션이 일관된 상태를 보장합니다.

### 데모 시나리오

| 시나리오 | 카테고리 | 공격 벡터 |
|---|---|---|
| GraphQL 인젝션 | API 보안 | 문자열 보간을 통한 악성 쿼리 |
| 서비스 거부 | API 보안 | 깊이 중첩된 쿼리, 배치 쿼리, 리소스 고갈 |
| 인가 우회 | API 보안 | GraphQL을 통한 비인가 데이터 접근 |
| 정보 노출 | API 보안 | 스키마 세부 정보를 드러내는 인트로스펙션 쿼리 |
| 배치 공격 | API 보안 | 단일 요청에 여러 작업 포함 |

### 주요 엔드포인트

```bash
# GraphiQL UI (대화형 IDE)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# 인트로스펙션 쿼리 -- 전체 스키마 열거
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# 페이스트 목록 (예제 쿼리)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# 페이스트 생성 (뮤테이션)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

| **경로** | `/restaurant/` |
| **이미지** | `theowni/Damn-Vulnerable-RESTaurant-API-Game`에서 커스텀 빌드 |
| **인스턴스** | 4개 (포트 8301-8304), 라운드 로빈 (공유 PostgreSQL) |
| **리소스** | 인스턴스당 0.5 CPU / 256 MiB RAM |
| **데이터베이스** | 공유 PostgreSQL 15.4 (`restaurant-db` 컨테이너, 0.5 CPU / 512 MiB) |
| **프레임워크** | Python / FastAPI / PostgreSQL |
| **자격 증명** | `admin` / `password` (PostgreSQL) |
| **프로젝트** | [github.com/theowni/Damn-Vulnerable-RESTaurant-API-Game](https://github.com/theowni/Damn-Vulnerable-RESTaurant-API-Game) |

RESTaurant는 OWASP API 보안 Top 10 2023을 다루는 게임화된 취약한 REST API입니다. `/restaurant/docs`에서 자동 Swagger UI 문서가 제공되는 FastAPI를 사용합니다. 4개의 인스턴스 모두 단일 PostgreSQL 데이터베이스를 공유하므로 스티키 세션 없이 라운드 로빈 로드 밸런싱이 동작합니다.

### 데모 시나리오

| 시나리오 | OWASP API Top 10 2023 | 방법 |
|---|---|---|
| 객체 수준 인가 취약점 (BOLA) | API1 | ID를 조작하여 다른 사용자의 주문에 접근 |
| 인증 취약점 | API2 | 취약한 토큰 처리, 크리덴셜 스터핑 |
| 객체 속성 수준 인가 취약점 | API3 | 사용자 프로필 필드에 대한 대량 할당 |
| 무제한 리소스 소비 | API4 | 엔드포인트에 속도 제한 없음 |
| 기능 수준 인가 취약점 (BFLA) | API5 | 일반 사용자로 관리자 엔드포인트 접근 |
| 서버 측 요청 위조 (SSRF) | API7 | 서버 측 URL 요청 조작 |
| 보안 설정 오류 | API8 | 상세한 오류 메시지, 기본 자격 증명 |

### 주요 엔드포인트

```bash
# Swagger UI
curl -sf "http://<PUBLIC_IP>/restaurant/docs" -o /dev/null -w "%{http_code}"

# OpenAPI 스펙
curl -s "http://<PUBLIC_IP>/restaurant/openapi.json" | jq .info

# BOLA -- 다른 사용자의 주문에 접근 (인증 후)
curl -s "http://<PUBLIC_IP>/restaurant/orders/1" \
  -H "Authorization: Bearer <token>"

# BFLA -- 일반 사용자로 관리자 작업 시도
curl -s -X POST "http://<PUBLIC_IP>/restaurant/admin/users" \
  -H "Authorization: Bearer <user_token>" \
  -H "Content-Type: application/json"
```

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **포트** | `8888` (전용 -- 경로 프리픽스 미사용) |
| **이미지** | `crapi/crapi-web`, `crapi/crapi-identity`, `crapi/crapi-community`, `crapi/crapi-workshop`, PostgreSQL, MongoDB, MailHog |
| **인스턴스** | 7개 마이크로서비스 (각각 단일 인스턴스) |
| **리소스** | 총 ~3.0 CPU / ~2.0 GiB RAM |
| **프레임워크** | React SPA + Java/Go/Python 마이크로서비스 |
| **프로젝트** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI는 API 보안 테스트를 위한 OWASP 플래그십 프로젝트입니다. React SPA가 API 경로를 하드코딩하고 있어 경로 프리픽스 뒤에서 서비스할 수 없기 때문에 전용 포트(8888)에서 7개 마이크로서비스로 실행됩니다. NSG는 포트 8888의 인바운드 트래픽을 허용합니다.

MailHog는 crAPI가 보내는 모든 이메일(계정 인증, 비밀번호 재설정)을 캡처합니다. 포트 18025에서 SSH 터널을 통해 MailHog에 접근하세요.

### 챌린지 카테고리

| 카테고리 | 취약점 |
|---|---|
| BOLA (객체 수준 인가 취약점) | 다른 사용자의 차량, 주문, 보고서에 접근 |
| BFLA (기능 수준 인가 취약점) | 관리자로 권한 상승, 제한된 엔드포인트 접근 |
| 대량 할당 | API를 통해 보호된 필드(역할, 잔액) 수정 |
| SSRF (서버 측 요청 위조) | 서버 측 URL 가져오기 조작 |
| JWT 조작 | 권한 상승을 위한 JWT 토큰 위조 또는 수정 |
| NoSQL 인젝션 | MongoDB 기반 엔드포인트에 쿼리 인젝션 |
| 과도한 데이터 노출 | API가 민감한 사용자 데이터를 반환 |

### 시작하기

```bash
# crAPI 실행 확인
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# 새 사용자 회원가입
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# 로그인
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# MailHog 접근 (이메일 인증을 위한 SSH 터널 사용)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# 그런 다음 브라우저에서 http://localhost:18025를 열어주세요
```

### crAPI 아키텍처

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, 사용자 인증, JWT)
             -> crapi-community (Go, 포럼, 게시물)
             -> crapi-workshop (Python, 차량 서비스)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (이메일 캡처, 포트 18025)
```
