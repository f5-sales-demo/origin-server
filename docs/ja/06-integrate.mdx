## F5 XC オリジンプール統合

このオリジンサーバーは、F5 Distributed Cloud HTTPロードバランサーの背後で**オリジンプールメンバー**として使用するように設計されています。F5 XCプラットフォームは、このオリジンの前段でWAF、Bot Defense、API Security、Client-Side Defenseを提供します。

### アーキテクチャ

```
エンドユーザー -> F5 XC HTTP LB (WAF/Bot/API/CSD) -> オリジンサーバーVM (nginx -> Dockerアプリ)
```

### オリジンプールの構成

このVMのパブリックIPを指すF5 XCオリジンプールを作成します：

| 設定 | 値 |
|---|---|
| **オリジンサーバータイプ** | Public IP of Origin Server |
| **IPアドレス** | `<terraform output public_ip>` |
| **ポート** | `80` |
| **ヘルスチェック** | HTTP、パス `/health` |

### HTTPロードバランサーの構成

オリジンプールを使用してHTTPロードバランサーを作成します：

| 設定 | 値 |
|---|---|
| **ドメイン** | デモ用ドメイン（例：`demo.example.com`） |
| **オリジンプール** | 上記で作成したプール |
| **WAF** | WAFポリシーをアタッチ |
| **Bot Defense** | 必要に応じて有効化 |
| **API Discovery** | VAmPIエンドポイント用に有効化 |

### パスベースルーティング

各アプリケーションは、ロードバランサーを通じてパスプレフィックスでアクセスできます：

| F5 XC LB URL | オリジンパス | アプリケーション |
|---|---|---|
| `https://demo.example.com/juice-shop/` | `/juice-shop/` | Juice Shop（4インスタンス、Cookieスティッキー） |
| `https://demo.example.com/dvwa/` | `/dvwa/` | DVWA（4インスタンス + MariaDB、Cookieスティッキー） |
| `https://demo.example.com/vampi/` | `/vampi/` | VAmPI（4インスタンス、ip_hashスティッキー） |
| `https://demo.example.com/httpbin/` | `/httpbin/` | httpbin（4インスタンス、ラウンドロビン） |
| `https://demo.example.com/whoami/` | `/whoami/` | リクエスト診断（4インスタンス） |
| `https://demo.example.com/csd-demo/` | `/csd-demo/` | CSD Demo（4インスタンス、ip_hashスティッキー） |
| `https://demo.example.com/dvga/` | `/dvga/` | DVGA（4インスタンス、ip_hashスティッキー） |
| `https://demo.example.com/restaurant/` | `/restaurant/` | RESTaurant（4インスタンス、ラウンドロビン） |
| `https://demo.example.com/health` | `/health` | ヘルスチェック（nginx直接） |

### F5 XC ヘッダーインジェクションの検証

whoamiエンドポイントを使用して、F5 XCがオリジンに到達するリクエストにインジェクトするヘッダーを確認します：

```bash
LB_DOMAIN="demo.example.com"

curl -sk "https://${LB_DOMAIN}/whoami/"
```

レスポンスで以下のF5 XCインジェクトヘッダーを確認してください：

| ヘッダー | 意味 |
|---|---|
| `X-Forwarded-For` | F5 XCを経由したクライアントIPチェーン |
| `True-Client-IP` | 元のクライアントIP |
| `X-Forwarded-Proto` | F5 XCでTLSが終端される場合 `https` |
| `X-Volterra-Bot-Type` | ボット分類（Bot Defense有効時） |
| `X-Request-ID` | F5 XCリクエストトラッキングID |

### F5 XC経由のWAFテスト

HTTPロードバランサーにWAFポリシーをアタッチした状態で、攻撃ペイロードをテストします：

```bash
LB_DOMAIN="demo.example.com"

# SQLインジェクション（WAFによりブロックされるはず）
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--&Submit=Submit"

# XSS（WAFによりブロックされるはず）
curl -sk "https://${LB_DOMAIN}/juice-shop/rest/products/search?q=<script>alert(1)</script>"

# コマンドインジェクション（WAFによりブロックされるはず）
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/exec/" \
  -d "ip=127.0.0.1;cat+/etc/passwd&Submit=Submit"

# 通常のリクエスト（通過するはず）
curl -sk "https://${LB_DOMAIN}/httpbin/get" | jq .
```

### APIセキュリティテスト

HTTPロードバランサーでAPI Discoveryを有効にして、VAmPIのAPIエンドポイントをマッピングします：

```bash
# これらのリクエストによりF5 XCでAPIインベントリが構築されます
curl -sk "https://${LB_DOMAIN}/vampi/users/v1"
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123","email":"api@test.com"}'
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123"}'
```

### crAPI統合（ポート8888）

crAPIは専用ポート（8888）で動作します。これは、crAPIがシングルページアプリケーションであり、APIパスがハードコードされているため、パスプレフィックスの背後で提供できないためです。crAPIをF5 XCと統合するには：

| 設定 | 値 |
|---|---|
| **オリジンサーバータイプ** | Public IP of Origin Server |
| **IPアドレス** | `<terraform output public_ip>` |
| **ポート** | `8888` |
| **ヘルスチェック** | HTTP、パス `/` |

crAPI用にポート8888で**別のオリジンプール**を作成するか、既存のプールにポート8888の2番目のオリジンプールメンバーを追加し、ルートルールを使用してトラフィックを振り分けます：

```bash
# F5 XC経由でcrAPIをテスト（構成済みの場合）
curl -sk "https://${LB_DOMAIN}:8888/"

# または同じLBドメインでルートルールを使用する場合：
# HostヘッダーまたはパスプレフィックスにマッチするF5 XCルートルールを構成し、
# crAPIオリジンプール（ポート8888）に転送します
```

:::note
オリジンVMのNSGはポート8888のインバウンドトラフィックを許可しています。追加のAzureネットワーク変更は不要です。
:::

### スティッキーセッションの考慮事項

オリジンサーバーは内部的にnginxスティッキーセッションを使用して、ステートフルなアプリケーションを一貫したバックエンドコンテナにルーティングしています。F5 XC HTTPロードバランサーを構成する際は、以下に注意してください：

| アプリケーション | スティッキー方式 | 理由 |
|---|---|---|
| Juice Shop | `hash $cookie_token` | Node.jsセッションステート |
| DVWA | `hash $cookie_PHPSESSID` | PHPセッションステート |
| VAmPI | `ip_hash` | インスタンスごとのSQLiteデータベース |
| CSD Demo | `ip_hash` | インスタンスごとのインメモリ漏洩ログ |
| DVGA | `ip_hash` | インスタンスごとのSQLiteデータベース |
| RESTaurant | ラウンドロビン | 共有PostgreSQLバックエンド |
| crAPI | --（単一ポート8888） | 7マイクロサービス、PostgreSQL + MongoDB |
| httpbin | ラウンドロビン | ステートレス |
| whoami | ラウンドロビン | ステートレス |

F5 XCはこれらのスティッキーセッションを複製する必要はありません -- オリジンVM上のnginxがバックエンドルーティングを処理します。F5 XCはオリジンを単一のエンドポイント（VMのパブリックIPのポート80）として扱うべきです。

### マルチコンポーネントアーキテクチャ

CDNシミュレーターと組み合わせると、完全なラボアーキテクチャは以下のようになります：

```
エンドユーザー -> CDNシミュレーター（nginxキャッシュ） -> F5 XC HTTP LB -> オリジンサーバー（このコンポーネント）
```

CDNシミュレーターはF5 XC HTTPロードバランサーのVIPをオリジンとして使用し、F5 XCロードバランサーはこのオリジンサーバーをバックエンドとして使用します。これにより、包括的なテストのための現実的な多層アーキテクチャが構築されます。
