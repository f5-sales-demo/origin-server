## アプリケーション概要

各アプリケーションは nginx のアップストリームプールの背後で、4 つのロードバランスされた Docker コンテナとして実行され、VM 上の nginx リバースプロキシを通じてアクセスできます。スティッキーセッションにより、ステートフルなアプリケーション（Juice Shop、DVWA、VAmPI、CSD Demo）は一貫してルーティングされます。すべてのアプリケーションは意図的に脆弱であり、セキュリティテスト用に設計されています。

以下のすべての例で `<ORIGIN>` を `http://<PUBLIC_IP>` に置き換えてください。Terraform でデプロイした後、`terraform output -raw public_ip` で IP を取得できます。

## OWASP Juice Shop

| | |
|---|---|
| **パス** | `/juice-shop/` |
| **イメージ** | `bkimminich/juice-shop:latest` |
| **インスタンス** | 4（ポート 3001-3004）、`hash $cookie_token` によるスティッキー、プロキシキャッシュ（60 秒 TTL） |
| **リソース** | インスタンスあたり 2 CPU / 1 GiB RAM |
| **フレームワーク** | Node.js / Angular |
| **プロジェクト** | [owasp.org/www-project-juice-shop](https://owasp.org/www-project-juice-shop/) |

OWASP Juice Shop は、最も現代的で活発にメンテナンスされている脆弱なウェブアプリケーションです。リアルな E コマースアプリケーションで、OWASP Top 10 全体をカバーする 100 以上のチャレンジを備えています。

### デモシナリオ

| シナリオ | カテゴリ | 攻撃ベクター |
|---|---|---|
| SQL インジェクションによるログインバイパス | WAF | ログインメールフィールドに `' OR 1=1--` |
| 反射型 XSS | WAF、CSD | 検索パラメータ経由のスクリプトインジェクション |
| DOM ベース XSS | CSD | URL フラグメント内のペイロード |
| 認証の不備 | WAF | ブルートフォースログイン、JWT 操作 |
| API 悪用 | API セキュリティ | `/api/` エンドポイントへの不正アクセス |
| 機密データの露出 | API セキュリティ | 認証なしでのユーザーデータへのアクセス |
| CSRF | WAF | プロフィール変更に対するクロスサイトリクエストフォージェリ |

### アクセス

```bash
curl -s "http://<PUBLIC_IP>/juice-shop/" -o /dev/null -w "%{http_code}"
```

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **パス** | `/dvwa/` |
| **イメージ** | カスタム `dvwa-fpm:latest`（php-fpm + nginx、`ghcr.io/digininja/dvwa:latest` からビルド） |
| **インスタンス** | 4（ポート 8101-8104）、`hash $cookie_PHPSESSID` によるスティッキー |
| **リソース** | インスタンスあたり 0.5 CPU / 256 MiB RAM |
| **データベース** | 共有 MariaDB 10.11（`dvwa-db` コンテナ、1 CPU / 768 MiB） |
| **フレームワーク** | PHP 8 / php-fpm / MariaDB |
| **認証情報** | `admin` / `password` |

DVWA は WAF テストの業界標準です。調整可能なセキュリティレベル（Low、Medium、High、Impossible）があり、入力検証と出力エンコーディングが段階的に追加されます。

### デモシナリオ

| シナリオ | カテゴリ | セキュリティレベル |
|---|---|---|
| SQL インジェクション | WAF | Low: 簡単な `' OR 1=1#`、High: ブラインド SQLi |
| コマンドインジェクション | WAF | Low: `; ls`、High: フィルタされた文字 |
| ファイルインクルージョン | WAF | Low: 直接パストラバーサル、High: サニタイズ済み |
| XSS（反射型） | WAF、CSD | Low: 基本的な `<script>`、High: エンコードバイパス |
| XSS（格納型） | WAF、CSD | Low: ゲストブックへの永続スクリプト |
| ファイルアップロード | WAF | Low: PHP シェルアップロード、High: 拡張子フィルタリング |
| ブルートフォース | Bot 防御 | 自動化されたログイン試行 |

### セキュリティレベル

ログイン後、`/dvwa/security.php` でセキュリティレベルを設定します：

- **Low** -- 入力検証なし。すべての攻撃が成功します。WAF シグネチャのデモンストレーションに適しています。
- **Medium** -- 基本的なフィルタリング。一部の攻撃にはエンコーディングやバイパス技術が必要です。
- **High** -- 強力なフィルタリング。高度な技術のみが成功します。WAF の限界を示すのに適しています。
- **Impossible** -- 完全にセキュアな実装。適切な防御コーディングを実演します。

### 初期セットアップ

DVWA は初回デプロイ後に一度だけデータベースセットアップが必要です：

1. `http://<PUBLIC_IP>/dvwa/setup.php` にアクセス
2. **Create / Reset Database** をクリック
3. `admin` / `password` でログイン

## VAmPI（脆弱な API）

| | |
|---|---|
| **パス** | `/vampi/` |
| **イメージ** | `erev0s/vampi:latest`（gunicorn エントリポイント、4 ワーカー） |
| **インスタンス** | 4（ポート 5101-5104）、`ip_hash` によるスティッキー（インスタンスごとの SQLite） |
| **リソース** | インスタンスあたり 0.5 CPU / 512 MiB RAM |
| **フレームワーク** | Python / Flask / gunicorn |
| **プロジェクト** | [github.com/erev0s/VAmPI](https://github.com/erev0s/VAmPI) |

VAmPI は OWASP API Security Top 10 のテスト専用に構築されています。意図的な脆弱性を持つリアルな REST API を提供します。各インスタンスは 4 ワーカーの gunicorn と独自の SQLite データベースを実行します。`ip_hash` スティッキーセッションにより、同じクライアント IP からの登録とログインは常に同じインスタンスに到達します。

### デモシナリオ

| シナリオ | OWASP API Top 10 | 手法 |
|---|---|---|
| オブジェクトレベル認可の不備（BOLA） | API1 | ID を操作して他のユーザーのデータにアクセス |
| 認証の不備 | API2 | 弱いトークン処理、レート制限なし |
| 過剰なデータ露出 | API3 | API がクライアントに必要以上のデータを返す |
| マスアサインメント | API6 | 予期しないパラメータで管理者フィールドを変更 |
| SQL インジェクション | API8 | API パラメータ経由のインジェクション |
| 不適切な資産管理 | API9 | 文書化されていない API エンドポイント |

### API エンドポイント

```bash
# 新しいユーザーを登録
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123","email":"test@test.com"}'

# ログイン
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123"}'

# ユーザー一覧（過剰なデータ露出）
curl "http://<PUBLIC_IP>/vampi/users/v1"
```

## httpbin

| | |
|---|---|
| **パス** | `/httpbin/` |
| **イメージ** | `kennethreitz/httpbin:latest`（gunicorn CMD オーバーライド `(-w 4 -k gevent --timeout 30)`） |
| **インスタンス** | 4（ポート 8201-8204）、ラウンドロビン（ステートレス） |
| **リソース** | インスタンスあたり 0.5 CPU / 256 MiB RAM |
| **フレームワーク** | Python / Flask / gunicorn + gevent |
| **プロジェクト** | [httpbin.org](https://httpbin.org) |

httpbin は、基本的な API デモ、リクエストヘッダーのテスト、プロキシ動作の検証に役立つシンプルな HTTP リクエスト/レスポンスサービスです。

### 便利なエンドポイント

| エンドポイント | 用途 |
|---|---|
| `/httpbin/get` | GET リクエストデータを返す（ヘッダー、引数、オリジン） |
| `/httpbin/post` | POST リクエストデータを返す（ボディ、フォーム、JSON） |
| `/httpbin/headers` | リクエストヘッダーを返す |
| `/httpbin/ip` | オリジン IP を返す |
| `/httpbin/user-agent` | User-Agent ヘッダーを返す |
| `/httpbin/status/:code` | 指定された HTTP ステータスコードを返す |
| `/httpbin/delay/:seconds` | N 秒レスポンスを遅延させる |
| `/httpbin/anything` | リクエストで渡されたすべてを返す |

### 使用例

```bash
# オリジンが受け取るヘッダーを確認（F5 XC ヘッダーインジェクションの検証に有用）
curl -s "http://<PUBLIC_IP>/httpbin/headers" | jq .

# 特定の HTTP ステータスコードをテスト
curl -s -o /dev/null -w "%{http_code}" "http://<PUBLIC_IP>/httpbin/status/403"
```

## whoami（リクエスト診断）

| | |
|---|---|
| **パス** | `/whoami/` |
| **イメージ** | `traefik/whoami:latest` |
| **インスタンス** | 4（ポート 8082-8085）、ラウンドロビン（ステートレス） |
| **リソース** | インスタンスあたり 0.25 CPU / 64 MiB RAM |
| **フレームワーク** | Go |
| **プロジェクト** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami は Traefik の軽量リクエストエコーサーバーです。オリジンが受け取る受信 HTTP リクエストのすべての詳細（ホスト名、IP アドレス、全ヘッダー、メソッド、URL）を表示します。F5 XC が正しいヘッダーを挿入しているかを検証する際の最も重要な診断ツールです。

### 主なユースケース

| ユースケース | 確認すべき内容 |
|---|---|
| F5 XC ヘッダーインジェクションの検証 | `X-Forwarded-For`、`True-Client-IP`、`X-Volterra-*` ヘッダー |
| クライアント IP の可視性確認 | `X-Real-IP` vs `RemoteAddr` |
| WAF 誤検知のデバッグ | F5 XC 前後のリクエストヘッダーを比較 |
| Bot 防御タグ付けの検証 | `X-Volterra-Bot-Type`、`X-Volterra-Bot-Verified` ヘッダー |
| TLS 終端の確認 | F5 XC で TLS が終端される場合 `X-Forwarded-Proto` に `https` が表示 |

### 使用例

```bash
# 基本リクエスト -- オリジンが受け取る内容を確認
curl "http://<PUBLIC_IP>/whoami/"

# F5 XC 経由のリクエストをシミュレート（挿入されたヘッダー付き）
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

出力例：

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA (Damn Vulnerable GraphQL Application)

| | |
|---|---|
| **パス** | `/dvga/` |
| **イメージ** | `dolevf/dvga:latest` |
| **インスタンス** | 4（ポート 5201-5204）、`ip_hash` によるスティッキー（インスタンスごとの SQLite） |
| **リソース** | インスタンスあたり 0.5 CPU / 256 MiB RAM |
| **フレームワーク** | Python / Flask / GraphQL |
| **プロジェクト** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA は GraphQL 固有の脆弱性テスト専用に構築されています。インタラクティブなクエリ探索用の GraphiQL IDE を `/dvga/` で提供し、GraphQL API エンドポイントを `/dvga/graphql` で提供します。各インスタンスは独自の SQLite データベースを使用するため、`ip_hash` スティッキーセッションにより一貫した状態が確保されます。

### デモシナリオ

| シナリオ | カテゴリ | 攻撃ベクター |
|---|---|---|
| GraphQL インジェクション | API セキュリティ | 文字列補間による悪意のあるクエリ |
| サービス拒否 | API セキュリティ | 深くネストされたクエリ、バッチクエリ、リソース枯渇 |
| 認可バイパス | API セキュリティ | GraphQL 経由での不正データへのアクセス |
| 情報漏洩 | API セキュリティ | スキーマ詳細を明らかにするイントロスペクションクエリ |
| バッチ攻撃 | API セキュリティ | 単一リクエストでの複数オペレーション |

### 主要エンドポイント

```bash
# GraphiQL UI（インタラクティブ IDE）
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# イントロスペクションクエリ -- 完全なスキーマを列挙
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# ペースト一覧（クエリ例）
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# ペースト作成（ミューテーション）
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

## RESTaurant (Damn Vulnerable RESTaurant API Game)

| | |
|---|---|
| **パス** | `/restaurant/` |
| **イメージ** | `theowni/Damn-Vulnerable-RESTaurant-API-Game` からのカスタムビルド |
| **インスタンス** | 4（ポート 8301-8304）、ラウンドロビン（共有 PostgreSQL） |
| **リソース** | インスタンスあたり 0.5 CPU / 256 MiB RAM |
| **データベース** | 共有 PostgreSQL 15.4（`restaurant-db` コンテナ、0.5 CPU / 512 MiB） |
| **フレームワーク** | Python / FastAPI / PostgreSQL |
| **認証情報** | `admin` / `password`（PostgreSQL） |
| **プロジェクト** | [github.com/theowni/Damn-Vulnerable-RESTaurant-API-Game](https://github.com/theowni/Damn-Vulnerable-RESTaurant-API-Game) |

RESTaurant は OWASP API Security Top 10 2023 をカバーするゲーミフィケーションされた脆弱な REST API です。FastAPI を使用し、`/restaurant/docs` で自動 Swagger UI ドキュメントを提供します。4 つのインスタンスすべてが単一の PostgreSQL データベースを共有するため、スティッキーセッションなしでラウンドロビンロードバランシングが機能します。

### デモシナリオ

| シナリオ | OWASP API Top 10 2023 | 手法 |
|---|---|---|
| オブジェクトレベル認可の不備（BOLA） | API1 | ID を操作して他のユーザーの注文にアクセス |
| 認証の不備 | API2 | 弱いトークン処理、クレデンシャルスタッフィング |
| オブジェクトプロパティレベル認可の不備 | API3 | ユーザープロフィールフィールドへのマスアサインメント |
| 無制限のリソース消費 | API4 | エンドポイントにレート制限なし |
| 機能レベル認可の不備（BFLA） | API5 | 一般ユーザーとして管理者エンドポイントにアクセス |
| サーバーサイドリクエストフォージェリ（SSRF） | API7 | サーバーサイド URL リクエストの操作 |
| セキュリティの設定ミス | API8 | 詳細なエラーメッセージ、デフォルト認証情報 |

### 主要エンドポイント

```bash
# Swagger UI
curl -sf "http://<PUBLIC_IP>/restaurant/docs" -o /dev/null -w "%{http_code}"

# OpenAPI 仕様
curl -s "http://<PUBLIC_IP>/restaurant/openapi.json" | jq .info

# BOLA -- 他のユーザーの注文にアクセス（認証後）
curl -s "http://<PUBLIC_IP>/restaurant/orders/1" \
  -H "Authorization: Bearer <token>"

# BFLA -- 一般ユーザーとして管理者アクションを試行
curl -s -X POST "http://<PUBLIC_IP>/restaurant/admin/users" \
  -H "Authorization: Bearer <user_token>" \
  -H "Content-Type: application/json"
```

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **ポート** | `8888`（専用 -- パスプレフィックスなし） |
| **イメージ** | `crapi/crapi-web`、`crapi/crapi-identity`、`crapi/crapi-community`、`crapi/crapi-workshop`、PostgreSQL、MongoDB、MailHog |
| **インスタンス** | 7 マイクロサービス（各 1 インスタンス） |
| **リソース** | 合計約 3.0 CPU / 約 2.0 GiB RAM |
| **フレームワーク** | React SPA + Java/Go/Python マイクロサービス |
| **プロジェクト** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI は API セキュリティテストのための OWASP フラグシッププロジェクトです。React SPA が API パスをハードコードしており、パスプレフィックスの背後では提供できないため、専用ポート（8888）で 7 つのマイクロサービスとして実行されます。NSG はポート 8888 でのインバウンドトラフィックを許可します。

MailHog は crAPI が送信するすべてのメール（アカウント検証、パスワードリセット）をキャプチャします。MailHog にはポート 18025 の SSH トンネル経由でアクセスします。

### チャレンジカテゴリ

| カテゴリ | 脆弱性 |
|---|---|
| BOLA（オブジェクトレベル認可の不備） | 他のユーザーの車両、注文、レポートにアクセス |
| BFLA（機能レベル認可の不備） | 管理者への権限昇格、制限されたエンドポイントへのアクセス |
| マスアサインメント | API 経由で保護されたフィールド（ロール、残高）を変更 |
| SSRF（サーバーサイドリクエストフォージェリ） | サーバーサイド URL フェッチの操作 |
| JWT 操作 | 権限昇格のための JWT トークンの偽造または変更 |
| NoSQL インジェクション | MongoDB バックエンドのエンドポイントへのクエリインジェクション |
| 過剰なデータ露出 | API が機密ユーザーデータを返す |

### はじめに

```bash
# crAPI が実行中であることを確認
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# 新しいユーザーをサインアップ
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# ログイン
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# MailHog にアクセス（メール検証用の SSH トンネル経由）
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# その後ブラウザで http://localhost:18025 を開く
```

### crAPI アーキテクチャ

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, ユーザー認証, JWT)
             -> crapi-community (Go, フォーラム, 投稿)
             -> crapi-workshop (Python, 車両サービス)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (メールキャプチャ, ポート 18025)
```
