すべてのTerraformファイルは [`terraform/`](https://github.com/f5-sales-demo/origin-server/tree/main/terraform) ディレクトリにあります。リポジトリをクローンして直接デプロイします:

```bash
git clone https://github.com/f5-sales-demo/origin-server.git
cd origin-server/terraform
cp terraform.tfvars.example terraform.tfvars
# terraform.tfvarsをAzureサブスクリプションIDとSSHキーパスで編集
```

## Terraform 設定

### Terraform ファイル構造

terraformディレクトリには、デモリソース標準に従った9つのファイルが含まれています:

- `versions.tf` — TerraformおよびプロバイダーバージョンConstraints (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Azure RMおよびAzure ADプロバイダー設定
- `data.tf` — デプロイヤー自動解決用のAzure ADデータソース
- `locals.tf` — デプロイヤー解決、Azure Cloud Adoption Frameworkリソース命名、標準タグ
- `main.tf` — リソースグループ (名称: `rg-origin-server-{environment}-{deployer}`)
- `variables.tf` — すべての入力変数 (必須1つ、オプション8つ)
- `network.tf` — VNet (10.200.0.0/16)、サブネット、パブリックIP、NSG (ポート 22/80/443/8888)、NIC
- `vm.tf` — templatefile()によるcloud-initを使用したUbuntu 24.04 VM
- `outputs.tf` — 25の出力 (標準15 + コンポーネント固有のアプリケーションURL 10)

`main.tf` にはリソースグループのみが含まれています:

```hcl file=../_data/main.tf
```

`variables.tf` はすべての設定可能なパラメーターを定義します。`deployer` 識別子はAzure ADアカウントから自動解決されます — `subscription_id` のみ設定する必要があります。`vm_size` のデフォルトは `Standard_D16s_v3` (16 vCPU、64 GiB RAM) で、41のDockerコンテナ向けにサイジングされています:

```hcl file=../_data/variables.tf
```

### ネットワークインフラストラクチャ

`network.tf` はVNet、サブネット、パブリックIP、NSG (ポート 22/80/443/8888)、およびNICを作成します。ポート8888は専用ポートで動作するcrAPIに必要です:

```hcl file=../_data/network.tf
```

### Cloud-Initを使用した仮想マシン

`vm.tf` はUbuntu 24.04 VMを作成し、cloud-initプロビジョニングスクリプトを渡します。60 GiBのPremium SSDは、DockerイメージとコンテナボリュームのためのHTMLに十分なスペースを提供します:

```hcl file=../_data/vm.tf
```

### Cloud-Init プロビジョニング

`cloud-init.yaml` はハード化された設定でVMをプロビジョニングします。共有ヘルパーライブラリ (`/usr/local/lib/cloud-init-helpers.sh`) は、すべてのネットワーク操作にリトライロジックを提供し、`/var/log/cloud-init-progress.log` への進捗ロギングを行います。固定時間のスリープの代わりに、アクティブなヘルスチェックポーリングでコンテナの準備完了を確認します。プロビジョニング内容:

- **カーネルチューニング**: TCP keepalive、TIME_WAIT再利用、2M tw_buckets、64K somaxconn
- **nginx**: スティッキーセッション (ip_hash、cookie hash)、keepaliveプール (64--128)、Juice Shop用プロキシキャッシュ、error_log critレベルを備えた8つのアップストリームブロック
- **Docker Compose**: 9つのアプリケーションにまたがる41のコンテナ (コンテナごとのCPU/メモリ制限あり)
- **カスタムビルド**: DVWA-FPM (pm.max_requests=200のphp-fpm)、CSD Demo (Flask + gunicorn)、RESTaurant (GitHubからクローン)
- **ワーカーリサイクル**: 継続的な負荷下でのメモリリークを防ぐために、uvicorn --limit-max-requests 200 (RESTaurant)、pm.max_requests 200 (DVWA-FPM)

```yaml file=../_data/cloud-init.yaml
```

### 出力

`outputs.tf` はデモリソース標準に従った25の出力を公開します — すべてのデモリソースで共有される15の標準出力 (`deployer`、`public_ip`、`private_ip`、`ssh_command`、`resource_group_name`、`vm_name`、`nsg_name`、`vnet_name`、`subnet_id`、`component`、`environment`、`resource_group_id`、`vm_id`、`nsg_id`、`location`) と、10のコンポーネント固有のアプリケーションURL (`origin_url`、`health_check_url`、`juice_shop_url`、`dvwa_url`、`vampi_url`、`httpbin_url`、`whoami_url`、`dvga_url`、`restaurant_url`、`crapi_url`):

```hcl file=../_data/outputs.tf
```

### 変数ファイルの例

`terraform.tfvars.example` を `terraform.tfvars` にコピーして値を入力します。`.gitignore` は認証情報のコミットを防ぐために `terraform.tfvars` を除外します:

```hcl file=../_data/terraform.tfvars.example
```

## デプロイ

```bash
cp terraform.tfvars.example terraform.tfvars
# terraform.tfvarsをAzureサブスクリプションIDとSSHキーパスで編集

terraform init

terraform plan

terraform apply
```

Terraformはデプロイ成功後にパブリックIP、SSHコマンド、およびアプリケーションURLを出力します。

## デプロイ後

`terraform apply` が完了したら、cloud-initがDockerのインストール、コンテナイメージのプル、nginxの設定を完了するまで5〜10分待ってください。DockerイメージのプルがボトルネックになりますJuice Shopだけで約400 MiBあります。

ヘルスエンドポイントを確認します:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

期待されるレスポンス:

```json
{
  "status": "healthy",
  "component": "origin-server",
  "applications": ["juice-shop", "dvwa", "vampi", "httpbin", "whoami", "csd-demo", "dvga", "restaurant", "crapi"]
}
```

### DVWA データベースセットアップ

DVWAは1回限りのデータベース初期化が必要です。ブラウザで `http://<PUBLIC_IP>/dvwa/setup.php` を開き、**Create / Reset Database** をクリックします。デフォルトの認証情報は `admin` / `password` です。

:::note
VAmPIのSQLiteデータベースはgunicornエントリーポイントスクリプトによって自動的に初期化されます。各コンテナは起動時に `db.create_all()` を実行するため、手動での操作は不要です。
:::

## ダウンストリームコンポーネントへの接続

デプロイ後、他のコンポーネントはオリジンサーバーの出力を入力として使用します:

```bash
# CDN シミュレーター — origin-serverのパブリックIPをアップストリームとして渡す
cd ../cdn-simulator/terraform
origin_ip=$(cd ../../origin-server/terraform && terraform output -raw public_ip)
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF

# トラフィックジェネレーター — F5 XCロードバランサーのFQDNを渡す (オリジンIPを直接渡さない)
cd ../traffic-generator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
target_fqdn     = "your-xc-load-balancer.example.com"
EOF
```

| 出力 | ダウンストリームコンポーネント | 入力変数 |
|---|---|---|
| `public_ip` | CDN シミュレーター | `origin_server` (`http://` プレフィックス付き) |
| `public_ip` | CDN シミュレーター | `origin_host` (`:80` サフィックス付き、スキームなし) |
| `public_ip` | F5 XC オリジンプール | オリジンサーバーアドレス |

[アプリケーション](../04-applications/) の使用ガイドまたは [検証](../05-verify/) のスモークテストに進んでください。
