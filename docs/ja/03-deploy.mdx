すべての Terraform ファイルは [`terraform/`](https://github.com/f5xc-salesdemos/origin-server/tree/main/terraform) ディレクトリにあります。リポジトリをクローンして直接デプロイしてください：

```bash
git clone https://github.com/f5xc-salesdemos/origin-server.git
cd origin-server/terraform
cp terraform.tfvars.example terraform.tfvars
# terraform.tfvars を Azure サブスクリプション ID と SSH キーパスで編集してください
```

## Terraform 構成

### Terraform ファイル構造

terraform ディレクトリには、デモリソース標準に従った 9 つのファイルが含まれています：

- `versions.tf` — Terraform およびプロバイダーのバージョン制約 (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Azure RM および Azure AD プロバイダー構成
- `data.tf` — デプロイヤー自動解決のための Azure AD データソース
- `locals.tf` — デプロイヤー解決、Azure クラウド導入フレームワークのリソース命名、標準タグ
- `main.tf` — リソースグループ（`rg-origin-server-{environment}-{deployer}` という命名）
- `variables.tf` — すべての入力変数（1 つの必須、8 つのオプション）
- `network.tf` — VNet (10.200.0.0/16)、サブネット、パブリック IP、NSG（ポート 22/80/443/8888）、NIC
- `vm.tf` — templatefile() による cloud-init を備えた Ubuntu 24.04 VM
- `outputs.tf` — 25 の出力（15 の標準出力 + 10 のコンポーネント固有アプリケーション URL）

`main.tf` にはリソースグループのみが含まれています：

```hcl file=./_data/main.tf
```

`variables.tf` はすべての設定可能なパラメータを定義します。`deployer` 識別子は Azure AD アカウントから自動解決されるため、`subscription_id` の設定のみが必要です。`vm_size` のデフォルトは `Standard_D16s_v3`（16 vCPU、64 GiB RAM）で、41 の Docker コンテナ向けにサイジングされています：

```hcl file=./_data/variables.tf
```

### ネットワークインフラストラクチャ

`network.tf` は VNet、サブネット、パブリック IP、NSG（ポート 22/80/443/8888）、および NIC を作成します。ポート 8888 は専用ポートで動作する crAPI に必要です：

```hcl file=./_data/network.tf
```

### Cloud-Init を備えた仮想マシン

`vm.tf` は Ubuntu 24.04 VM を作成し、cloud-init プロビジョニングスクリプトを渡します。60 GiB の Premium SSD は Docker イメージとコンテナボリュームに十分なスペースを提供します：

```hcl file=./_data/vm.tf
```

### Cloud-Init プロビジョニング

`cloud-init.yaml` は堅牢な構成で VM をプロビジョニングします。共有ヘルパーライブラリ（`/usr/local/lib/cloud-init-helpers.sh`）がすべてのネットワーク操作にリトライロジックを提供し、`/var/log/cloud-init-progress.log` への進捗ログを記録します。コンテナの準備状態確認には、固定時間の sleep の代わりにアクティブなヘルスチェックポーリングが使用されます。プロビジョニング内容：

- **カーネルチューニング**: TCP キープアライブ、TIME_WAIT 再利用、2M tw_buckets、64K somaxconn
- **nginx**: スティッキーセッション（ip_hash、cookie hash）を備えた 8 つのアップストリームブロック、キープアライブプール（64--128）、Juice Shop 用のプロキシキャッシュ、error_log crit レベル
- **Docker Compose**: コンテナごとの CPU/メモリ制限を持つ 9 つのアプリケーションにわたる 41 のコンテナ
- **カスタムビルド**: DVWA-FPM（pm.max_requests=200 の php-fpm）、CSD Demo（Flask + gunicorn）、RESTaurant（GitHub からクローン）
- **ワーカーリサイクル**: 持続的な負荷下でのメモリリークを防止するための uvicorn --limit-max-requests 200（RESTaurant）、pm.max_requests 200（DVWA-FPM）

```yaml file=./_data/cloud-init.yaml
```

### 出力

`outputs.tf` はデモリソース標準に従った 25 の出力を公開します — すべてのデモリソースで共有される 15 の標準出力（`deployer`、`public_ip`、`private_ip`、`ssh_command`、`resource_group_name`、`vm_name`、`nsg_name`、`vnet_name`、`subnet_id`、`component`、`environment`、`resource_group_id`、`vm_id`、`nsg_id`、`location`）に加え、10 のコンポーネント固有アプリケーション URL（`origin_url`、`health_check_url`、`juice_shop_url`、`dvwa_url`、`vampi_url`、`httpbin_url`、`whoami_url`、`dvga_url`、`restaurant_url`、`crapi_url`）：

```hcl file=./_data/outputs.tf
```

### 変数ファイルの例

`terraform.tfvars.example` を `terraform.tfvars` にコピーして、値を入力してください。`.gitignore` は認証情報のコミットを防ぐために `terraform.tfvars` を除外しています：

```hcl file=./_data/terraform.tfvars.example
```

## デプロイ

```bash
cp terraform.tfvars.example terraform.tfvars
# terraform.tfvars を Azure サブスクリプション ID と SSH キーパスで編集してください

terraform init

terraform plan

terraform apply
```

デプロイが成功すると、Terraform はパブリック IP、SSH コマンド、およびアプリケーション URL を出力します。

## デプロイ後

`terraform apply` が完了した後、cloud-init が Docker のインストール、コンテナイメージのプル、nginx の設定を完了するまで 5〜10 分お待ちください。Docker イメージのプルがボトルネックとなります — Juice Shop だけで約 400 MiB あります。

ヘルスエンドポイントを確認してください：

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

期待されるレスポンス：

```json
{
  "status": "healthy",
  "component": "origin-server",
  "applications": ["juice-shop", "dvwa", "vampi", "httpbin", "whoami", "csd-demo", "dvga", "restaurant", "crapi"]
}
```

### DVWA データベースセットアップ

DVWA は一回限りのデータベース初期化が必要です。ブラウザで `http://<PUBLIC_IP>/dvwa/setup.php` を開き、**Create / Reset Database** をクリックしてください。デフォルトの認証情報は `admin` / `password` です。

:::note
VAmPI の SQLite データベースは gunicorn のエントリポイントスクリプトによって自動的に初期化されます。各コンテナは起動時に `db.create_all()` を実行するため、手動での操作は不要です。
:::

## 下流コンポーネントへの接続

デプロイ後、他のコンポーネントはオリジンサーバーの出力を入力として使用します：

```bash
# CDN Simulator — origin-server のパブリック IP をアップストリームとして渡す
cd ../cdn-simulator/terraform
origin_ip=$(cd ../../origin-server/terraform && terraform output -raw public_ip)
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF

# Traffic Generator — F5 XC ロードバランサーの FQDN を渡す（オリジン IP を直接渡すのではない）
cd ../traffic-generator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
target_fqdn     = "your-xc-load-balancer.example.com"
EOF
```

| 出力 | 下流コンポーネント | 入力変数 |
|---|---|---|
| `public_ip` | CDN Simulator | `origin_server`（`http://` プレフィックス付き） |
| `public_ip` | CDN Simulator | `origin_host`（`:80` サフィックス付き、スキームなし） |
| `public_ip` | F5 XC Origin Pool | オリジンサーバーアドレス |

使用ガイドは[アプリケーション](../04-applications/)へ、スモークテストは[検証](../05-verify/)へ進んでください。
