## Panoramica delle Applicazioni

Ogni applicazione viene eseguita come 4 container Docker bilanciati dietro upstream pool nginx, accessibili attraverso il reverse proxy nginx sulla VM. Le sessioni sticky garantiscono che le applicazioni stateful (Juice Shop, DVWA, VAmPI, CSD Demo) vengano instradate in modo coerente. Tutte le applicazioni sono intenzionalmente vulnerabili e progettate per il testing di sicurezza.

Sostituire `<ORIGIN>` con `http://<PUBLIC_IP>` in tutti gli esempi seguenti. Dopo il deploy tramite Terraform, ottenere l'IP con `terraform output -raw public_ip`.

| **Progetto** | [owasp.org/www-project-juice-shop](https://owasp.org/www-project-juice-shop/) |

OWASP Juice Shop è l'applicazione web vulnerabile più moderna e attivamente mantenuta. Copre l'intera OWASP Top 10 con oltre 100 sfide in un'applicazione e-commerce realistica.

### Scenari di Dimostrazione

| Scenario | Categoria | Vettore di Attacco |
|---|---|---|
| Bypass login tramite SQL Injection | WAF | `' OR 1=1--` nel campo email di login |
| XSS Riflesso | WAF, CSD | Iniezione di script tramite parametro di ricerca |
| XSS basato su DOM | CSD | Payload nel frammento URL |
| Autenticazione compromessa | WAF | Brute force login, manipolazione JWT |
| Abuso API | API Security | Accesso non autorizzato agli endpoint `/api/` |
| Esposizione di dati sensibili | API Security | Accesso ai dati utente senza autorizzazione |
| CSRF | WAF | Cross-site request forgery sulle modifiche del profilo |

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **Percorso** | `/dvwa/` |
| **Immagine** | Personalizzata `dvwa-fpm:latest` (php-fpm + nginx, costruita da `ghcr.io/digininja/dvwa:latest`) |
| **Istanze** | 4 (porte 8101-8104), sticky tramite `hash $cookie_PHPSESSID` |
| **Risorse** | 0.5 CPU / 256 MiB RAM per istanza |
| **Database** | MariaDB 10.11 condiviso (container `dvwa-db`, 1 CPU / 768 MiB) |
| **Framework** | PHP 8 / php-fpm / MariaDB |
| **Credenziali** | `admin` / `password` |

DVWA è lo standard di settore per il testing WAF. Dispone di livelli di sicurezza regolabili (Low, Medium, High, Impossible) che aggiungono progressivamente validazione dell'input e codifica dell'output.

### Scenari di Dimostrazione

| Scenario | Categoria | Livello di Sicurezza |
|---|---|---|
| SQL Injection | WAF | Low: banale `' OR 1=1#`, High: blind SQLi |
| Command Injection | WAF | Low: `; ls`, High: caratteri filtrati |
| File Inclusion | WAF | Low: path traversal diretto, High: sanitizzato |
| XSS (Riflesso) | WAF, CSD | Low: `<script>` basilare, High: bypass con codifica |
| XSS (Memorizzato) | WAF, CSD | Low: script persistente nel guestbook |
| File Upload | WAF | Low: upload di shell PHP, High: filtraggio estensioni |
| Brute Force | Bot Defense | Tentativi di login automatizzati |

### Livelli di Sicurezza

Impostare il livello di sicurezza in `/dvwa/security.php` dopo il login:

- **Low** -- Nessuna validazione dell'input. Ogni attacco funziona. Ideale per dimostrazioni delle firme WAF.
- **Medium** -- Filtraggio basilare. Alcuni attacchi richiedono codifica o tecniche di bypass.
- **High** -- Filtraggio robusto. Solo le tecniche avanzate hanno successo. Ideale per mostrare i limiti del WAF.
- **Impossible** -- Implementazione completamente sicura. Dimostra la codifica difensiva corretta.

### Configurazione Iniziale

DVWA richiede una configurazione del database una tantum dopo il primo deploy:

1. Navigare a `http://<PUBLIC_IP>/dvwa/setup.php`
2. Fare clic su **Create / Reset Database**
3. Effettuare il login con `admin` / `password`

VAmPI è appositamente progettato per testare la OWASP API Security Top 10. Fornisce un'API REST realistica con vulnerabilità deliberate. Ogni istanza esegue gunicorn con 4 worker e il proprio database SQLite. La sessione sticky `ip_hash` garantisce che registrazione e login dallo stesso IP client raggiungano sempre la stessa istanza.

### Scenari di Dimostrazione

| SQL Injection | API8 | Iniezione attraverso i parametri API |
| Improper Assets Management | API9 | Endpoint API non documentati |

### Endpoint API

```bash
# Registrare un nuovo utente

  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123","email":"test@test.com"}'

curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/login" \
  -H "Content-Type: application/json" \

```

## whoami (Diagnostica delle Richieste)

| | |
|---|---|
| **Percorso** | `/whoami/` |
| **Immagine** | `traefik/whoami:latest` |
| **Istanze** | 4 (porte 8082-8085), round-robin (stateless) |
| **Risorse** | 0.25 CPU / 64 MiB RAM per istanza |
| **Framework** | Go |
| **Progetto** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami è il server echo delle richieste leggero di Traefik. Mostra ogni dettaglio della richiesta HTTP in arrivo così come la vede l'origin -- hostname, indirizzi IP, tutti gli header, metodo e URL. Questo è lo strumento diagnostico più importante in assoluto per verificare che F5 XC stia iniettando gli header corretti.

### Casi d'Uso Principali

| Caso d'Uso | Cosa Cercare |
|---|---|
| Verificare l'iniezione di header di F5 XC | Header `X-Forwarded-For`, `True-Client-IP`, `X-Volterra-*` |
| Confermare la visibilità dell'IP client | `X-Real-IP` vs `RemoteAddr` |
| Debug dei falsi positivi WAF | Confrontare gli header della richiesta prima/dopo F5 XC |
| Validare il tagging della bot defense | Header `X-Volterra-Bot-Type`, `X-Volterra-Bot-Verified` |
| Verificare la terminazione TLS | `X-Forwarded-Proto` mostra `https` quando il TLS termina su F5 XC |

### Esempio

```bash
# Richiesta base -- vedere cosa riceve l'origin
curl "http://<PUBLIC_IP>/whoami/"

# Simulare una richiesta attraverso F5 XC (con header iniettati)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

Output di esempio:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA (Damn Vulnerable GraphQL Application)

| | |
|---|---|
| **Percorso** | `/dvga/` |
| **Immagine** | `dolevf/dvga:latest` |
| **Istanze** | 4 (porte 5201-5204), sticky tramite `ip_hash` (SQLite per istanza) |
| **Risorse** | 0.5 CPU / 256 MiB RAM per istanza |
| **Framework** | Python / Flask / GraphQL |
| **Progetto** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA è appositamente progettato per testare vulnerabilità specifiche di GraphQL. Fornisce un IDE GraphiQL in `/dvga/` per l'esplorazione interattiva delle query e un endpoint API GraphQL in `/dvga/graphql`. Ogni istanza utilizza il proprio database SQLite, quindi le sessioni sticky `ip_hash` garantiscono uno stato coerente.

### Scenari di Dimostrazione

| Scenario | Categoria | Vettore di Attacco |
|---|---|---|
| GraphQL Injection | API Security | Query malevole tramite interpolazione di stringhe |
| Denial of Service | API Security | Query profondamente annidate, query in batch, esaurimento risorse |
| Bypass dell'Autorizzazione | API Security | Accesso a dati non autorizzati tramite GraphQL |
| Divulgazione di Informazioni | API Security | Query di introspezione che rivelano dettagli dello schema |
| Attacco Batching | API Security | Operazioni multiple in una singola richiesta |

### Endpoint Principali

```bash
# GraphiQL UI (IDE interattivo)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# Query di introspezione -- enumerare lo schema completo
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# Elencare i paste (query di esempio)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# Creare un paste (mutation)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

RESTaurant è un'API REST vulnerabile gamificata che copre la OWASP API Security Top 10 2023. Utilizza FastAPI con documentazione Swagger UI automatica in `/restaurant/docs`. Tutte e 4 le istanze condividono un singolo database PostgreSQL, quindi il bilanciamento del carico round-robin funziona senza sessioni sticky.

### Scenari di Dimostrazione

| Scenario | OWASP API Top 10 2023 | Metodo |
|---|---|---|
| Broken Object Level Authorization (BOLA) | API1 | Accesso agli ordini di altri utenti manipolando gli ID |
| Broken Authentication | API2 | Gestione debole dei token, credential stuffing |
| Broken Object Property Level Authorization | API3 | Mass assignment sui campi del profilo utente |
| Unrestricted Resource Consumption | API4 | Nessun rate limiting sugli endpoint |
| Broken Function Level Authorization (BFLA) | API5 | Accesso agli endpoint admin come utente normale |
| Server Side Request Forgery (SSRF) | API7 | Manipolazione delle richieste URL lato server |

### Endpoint Principali

```bash
# Swagger UI
curl -sf "http://<PUBLIC_IP>/restaurant/docs" -o /dev/null -w "%{http_code}"

# Specifica OpenAPI
curl -s "http://<PUBLIC_IP>/restaurant/openapi.json" | jq .info

# BOLA -- accedere all'ordine di un altro utente (dopo l'autenticazione)
curl -s "http://<PUBLIC_IP>/restaurant/orders/1" \
  -H "Authorization: Bearer <token>"

# BFLA -- tentare un'azione admin come utente normale
curl -s -X POST "http://<PUBLIC_IP>/restaurant/admin/users" \
  -H "Authorization: Bearer <user_token>" \
  -H "Content-Type: application/json"
```

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **Porta** | `8888` (dedicata -- non con prefisso di percorso) |
| **Immagini** | `crapi/crapi-web`, `crapi/crapi-identity`, `crapi/crapi-community`, `crapi/crapi-workshop`, PostgreSQL, MongoDB, MailHog |
| **Istanze** | 7 microservizi (singola istanza ciascuno) |
| **Risorse** | ~3.0 CPU / ~2.0 GiB RAM totali |
| **Framework** | React SPA + microservizi Java/Go/Python |
| **Progetto** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI è il progetto di punta OWASP per il testing della sicurezza API. Viene eseguito come 7 microservizi su una porta dedicata (8888) perché la React SPA ha i percorsi API hardcoded e non può essere servita dietro un prefisso di percorso. Il NSG consente il traffico in entrata sulla porta 8888.

MailHog cattura tutte le email inviate da crAPI (verifica account, reset password). Accedere a MailHog tramite tunnel SSH sulla porta 18025.

### Categorie di Sfide

| Categoria | Vulnerabilità |
|---|---|
| BOLA (Broken Object Level Authorization) | Accesso a veicoli, ordini e report di altri utenti |
| BFLA (Broken Function Level Authorization) | Escalation ad admin, accesso a endpoint ristretti |
| Mass Assignment | Modifica di campi protetti (ruolo, saldo) tramite API |
| SSRF (Server Side Request Forgery) | Manipolazione del fetching URL lato server |
| Manipolazione JWT | Forgiare o modificare token JWT per escalation dei privilegi |
| NoSQL Injection | Iniezione di query negli endpoint backed da MongoDB |
| Esposizione Eccessiva di Dati | L'API restituisce dati utente sensibili |

### Per Iniziare

```bash
# Verificare che crAPI sia in esecuzione
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# Registrare un nuovo utente
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# Effettuare il login
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# Accedere a MailHog (tramite tunnel SSH per la verifica email)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# Poi aprire http://localhost:18025 nel browser
```

### Architettura di crAPI

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, autenticazione utente, JWT)
             -> crapi-community (Go, forum, post)
             -> crapi-workshop (Python, servizio veicoli)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (cattura email, porta 18025)
```
