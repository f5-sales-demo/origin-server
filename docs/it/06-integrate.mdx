## Integrazione con F5 XC Origin Pool

Questo server di origine è progettato per essere utilizzato come **membro di un origin pool** dietro un HTTP load balancer di F5 Distributed Cloud. La piattaforma F5 XC fornisce WAF, Bot Defense, API Security e Client-Side Defense davanti a questo server di origine.

### Architettura

```
End User -> F5 XC HTTP LB (WAF/Bot/API/CSD) -> Origin Server VM (nginx -> Docker apps)
```

### Configurazione dell'Origin Pool

Creare un origin pool F5 XC che punta all'IP pubblico di questa VM:

| Impostazione | Valore |
|---|---|
| **Origin Server Type** | Public IP of Origin Server |
| **IP Address** | `<terraform output public_ip>` |
| **Port** | `80` |
| **Health Check** | HTTP, path `/health` |

### Configurazione dell'HTTP Load Balancer

Creare un HTTP load balancer con l'origin pool:

| Impostazione | Valore |
|---|---|
| **Domains** | Il dominio demo (es. `demo.example.com`) |
| **Origin Pool** | Il pool creato sopra |
| **WAF** | Associare la propria policy WAF |
| **Bot Defense** | Abilitare secondo necessità |
| **API Discovery** | Abilitare per gli endpoint VAmPI |

### Routing basato sul percorso

Ogni applicazione è accessibile tramite il proprio prefisso di percorso attraverso il load balancer:

| URL F5 XC LB | Percorso Origin | Applicazione |
|---|---|---|
| `https://demo.example.com/juice-shop/` | `/juice-shop/` | Juice Shop (4 istanze, cookie sticky) |
| `https://demo.example.com/dvwa/` | `/dvwa/` | DVWA (4 istanze + MariaDB, cookie sticky) |
| `https://demo.example.com/vampi/` | `/vampi/` | VAmPI (4 istanze, ip_hash sticky) |
| `https://demo.example.com/httpbin/` | `/httpbin/` | httpbin (4 istanze, round-robin) |
| `https://demo.example.com/whoami/` | `/whoami/` | Diagnostica delle richieste (4 istanze) |
| `https://demo.example.com/csd-demo/` | `/csd-demo/` | CSD Demo (4 istanze, ip_hash sticky) |
| `https://demo.example.com/dvga/` | `/dvga/` | DVGA (4 istanze, ip_hash sticky) |
| `https://demo.example.com/restaurant/` | `/restaurant/` | RESTaurant (4 istanze, round-robin) |
| `https://demo.example.com/health` | `/health` | Health check (nginx diretto) |

### Verifica dell'iniezione di header F5 XC

Utilizzare l'endpoint whoami per verificare quali header F5 XC inietta nelle richieste che raggiungono il server di origine:

```bash
LB_DOMAIN="demo.example.com"

curl -sk "https://${LB_DOMAIN}/whoami/"
```

Cercare questi header iniettati da F5 XC nella risposta:

| Header | Significato |
|---|---|
| `X-Forwarded-For` | Catena di IP del client attraverso F5 XC |
| `True-Client-IP` | IP originale del client |
| `X-Forwarded-Proto` | `https` se il TLS termina su F5 XC |
| `X-Volterra-Bot-Type` | Classificazione bot (quando Bot Defense è abilitato) |
| `X-Request-ID` | ID di tracciamento richiesta F5 XC |

### Test WAF tramite F5 XC

Con una policy WAF associata all'HTTP load balancer, testare payload di attacco:

```bash
LB_DOMAIN="demo.example.com"

# SQL Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--&Submit=Submit"

# XSS (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/juice-shop/rest/products/search?q=<script>alert(1)</script>"

# Command Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/exec/" \
  -d "ip=127.0.0.1;cat+/etc/passwd&Submit=Submit"

# Normal request (should pass through)
curl -sk "https://${LB_DOMAIN}/httpbin/get" | jq .
```

### Test di API Security

Abilitare API Discovery sull'HTTP load balancer per mappare gli endpoint API di VAmPI:

```bash
# These requests build the API inventory in F5 XC
curl -sk "https://${LB_DOMAIN}/vampi/users/v1"
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123","email":"api@test.com"}'
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123"}'
```

### Integrazione crAPI (Porta 8888)

crAPI viene eseguito su una porta dedicata (8888) perché è un'applicazione single-page che codifica in modo fisso i propri percorsi API e non può essere servita dietro un prefisso di percorso. Per integrare crAPI con F5 XC:

| Impostazione | Valore |
|---|---|
| **Origin Server Type** | Public IP of Origin Server |
| **IP Address** | `<terraform output public_ip>` |
| **Port** | `8888` |
| **Health Check** | HTTP, path `/` |

Creare un **origin pool separato** per crAPI sulla porta 8888, oppure aggiungere un secondo membro dell'origin pool al pool esistente con la porta 8888 e utilizzare regole di routing per indirizzare il traffico:

```bash
# Test crAPI through F5 XC (if configured)
curl -sk "https://${LB_DOMAIN}:8888/"

# Or if using route rules on the same LB domain:
# Configure an F5 XC route rule matching Host header or path prefix
# to forward to the crAPI origin pool (port 8888)
```

:::note
Il NSG sulla VM di origine consente il traffico in entrata sulla porta 8888. Non sono necessarie modifiche aggiuntive alla rete Azure.
:::

### Consapevolezza delle sessioni sticky

Il server di origine utilizza internamente le sessioni sticky di nginx per instradare le applicazioni con stato verso container backend coerenti. Quando si configura l'HTTP load balancer F5 XC, tenere presente:

| Applicazione | Metodo Sticky | Motivazione |
|---|---|---|
| Juice Shop | `hash $cookie_token` | Stato della sessione Node.js |
| DVWA | `hash $cookie_PHPSESSID` | Stato della sessione PHP |
| VAmPI | `ip_hash` | Database SQLite per istanza |
| CSD Demo | `ip_hash` | Log di esfiltrazione in memoria per istanza |
| DVGA | `ip_hash` | Database SQLite per istanza |
| RESTaurant | Round-robin | Backend PostgreSQL condiviso |
| crAPI | -- (porta singola 8888) | 7 microservizi, PostgreSQL + MongoDB |
| httpbin | Round-robin | Stateless |
| whoami | Round-robin | Stateless |

F5 XC non ha bisogno di replicare queste sessioni sticky -- nginx sulla VM di origine gestisce il routing verso i backend. F5 XC dovrebbe trattare l'origine come un singolo endpoint (l'IP pubblico della VM sulla porta 80).

### Architettura multi-componente

Quando combinata con il CDN Simulator, l'architettura completa del laboratorio è:

```
End User -> CDN Simulator (nginx cache) -> F5 XC HTTP LB -> Origin Server (this component)
```

Il CDN Simulator utilizza il VIP dell'HTTP load balancer F5 XC come propria origine, e il load balancer F5 XC utilizza questo server di origine come proprio backend. Questo crea un'architettura multi-livello realistica per test completi.
