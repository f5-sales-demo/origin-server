Tutti i file Terraform si trovano nella directory [`terraform/`](https://github.com/f5-sales-demo/origin-server/tree/main/terraform). Clona il repository e distribuisci direttamente:

```bash
git clone https://github.com/f5-sales-demo/origin-server.git
cd origin-server/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and SSH key path
```

## Configurazione Terraform

### Struttura dei file Terraform

La directory terraform contiene 9 file che seguono il Demo Resource Standard:

- `versions.tf` — Vincoli di versione per Terraform e provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Configurazione dei provider Azure RM e Azure AD
- `data.tf` — Sorgenti dati Azure AD per la risoluzione automatica del deployer
- `locals.tf` — Risoluzione del deployer, denominazione delle risorse Azure Cloud Adoption Framework, tag standard
- `main.tf` — Gruppo di risorse (denominato `rg-origin-server-{environment}-{deployer}`)
- `variables.tf` — Tutte le variabili di input (1 obbligatoria, 8 opzionali)
- `network.tf` — VNet (10.200.0.0/16), subnet, IP pubblico, NSG (porte 22/80/443/8888), NIC
- `vm.tf` — VM Ubuntu 24.04 con cloud-init tramite templatefile()
- `outputs.tf` — 25 output (15 standard + 10 URL di applicazioni specifiche per componente)

`main.tf` contiene solo il gruppo di risorse:

```hcl file=../_data/main.tf
```

`variables.tf` definisce tutti i parametri configurabili. L'identificatore `deployer` viene risolto automaticamente dall'account Azure AD — è sufficiente impostare `subscription_id`. Il valore predefinito di `vm_size` è `Standard_D16s_v3` (16 vCPU, 64 GiB RAM), dimensionato per 41 container Docker:

```hcl file=../_data/variables.tf
```

### Infrastruttura di rete

`network.tf` crea la VNet, la subnet, l'IP pubblico, l'NSG (porte 22/80/443/8888) e la NIC. La porta 8888 è richiesta da crAPI, che viene eseguita su una porta dedicata:

```hcl file=../_data/network.tf
```

### Macchina virtuale con Cloud-Init

`vm.tf` crea la VM Ubuntu 24.04 e passa lo script di provisioning cloud-init. Il SSD Premium da 60 GiB fornisce spazio sufficiente per le immagini Docker e i volumi dei container:

```hcl file=../_data/vm.tf
```

### Provisioning Cloud-Init

`cloud-init.yaml` esegue il provisioning della VM con una configurazione robusta. Una libreria helper condivisa (`/usr/local/lib/cloud-init-helpers.sh`) fornisce logica di retry su tutte le operazioni di rete e registrazione dei progressi in `/var/log/cloud-init-progress.log`. Il polling attivo di health-check sostituisce il sleep a durata fissa per la disponibilità dei container. Esegue il provisioning di:

- **Ottimizzazione del kernel**: TCP keepalive, riutilizzo TIME_WAIT, 2M tw_buckets, 64K somaxconn
- **nginx**: 8 blocchi upstream con sessioni sticky (ip_hash, cookie hash), pool keepalive (64--128), cache proxy per Juice Shop, livello error_log crit
- **Docker Compose**: 41 container su 9 applicazioni con limiti CPU/memoria per container
- **Build personalizzate**: DVWA-FPM (php-fpm con pm.max_requests=200), CSD Demo (Flask + gunicorn), RESTaurant (clonato da GitHub)
- **Riciclo dei worker**: uvicorn --limit-max-requests 200 (RESTaurant), pm.max_requests 200 (DVWA-FPM) per prevenire perdite di memoria sotto carico sostenuto

```yaml file=../_data/cloud-init.yaml
```

### Output

`outputs.tf` espone 25 output seguendo il Demo Resource Standard — 15 output standard condivisi da tutte le risorse demo (`deployer`, `public_ip`, `private_ip`, `ssh_command`, `resource_group_name`, `vm_name`, `nsg_name`, `vnet_name`, `subnet_id`, `component`, `environment`, `resource_group_id`, `vm_id`, `nsg_id`, `location`) più 10 URL di applicazioni specifiche per componente (`origin_url`, `health_check_url`, `juice_shop_url`, `dvwa_url`, `vampi_url`, `httpbin_url`, `whoami_url`, `dvga_url`, `restaurant_url`, `crapi_url`):

```hcl file=../_data/outputs.tf
```

## Deploy

```bash
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and SSH key path

terraform init

terraform plan

terraform apply
```

Terraform restituisce l'IP pubblico, il comando SSH e gli URL delle applicazioni dopo una distribuzione riuscita.

## Post-Deploy

Dopo il completamento di `terraform apply`, attendere 5--10 minuti affinché cloud-init termini l'installazione di Docker, il pull delle immagini container e la configurazione di nginx. Il pull delle immagini Docker rappresenta il collo di bottiglia — solo Juice Shop è circa 400 MiB.

Verificare l'endpoint di health:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

Risposta attesa:

```json
{
  "status": "healthy",
  "component": "origin-server",
  "applications": ["juice-shop", "dvwa", "vampi", "httpbin", "whoami", "csd-demo", "dvga", "restaurant", "crapi"]
}
```

### Configurazione del database DVWA

DVWA richiede un'inizializzazione del database una tantum. Aprire `http://<PUBLIC_IP>/dvwa/setup.php` in un browser e fare clic su **Create / Reset Database**. Le credenziali predefinite sono `admin` / `password`.

:::note
Il database SQLite di VAmPI viene inizializzato automaticamente dallo script entrypoint di gunicorn. Ogni container esegue `db.create_all()` all'avvio, quindi non è necessario alcun intervento manuale.
:::

## Collegamento ai componenti downstream

Dopo la distribuzione, gli altri componenti utilizzano gli output del server di origine come input:

```bash
# CDN Simulator — pass origin-server's public IP as its upstream
cd ../cdn-simulator/terraform
origin_ip=$(cd ../../origin-server/terraform && terraform output -raw public_ip)
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF

# Traffic Generator — pass the F5 XC load balancer FQDN (not the origin IP directly)
cd ../traffic-generator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
target_fqdn     = "your-xc-load-balancer.example.com"
EOF
```

| Output | Componente downstream | Variabile di input |
|---|---|---|
| `public_ip` | Simulatore CDN | `origin_server` (con prefisso `http://`) |
| `public_ip` | Simulatore CDN | `origin_host` (con suffisso `:80`, senza schema) |
| `public_ip` | F5 XC Origin Pool | Indirizzo del server di origine |

Procedere ad [Applicazioni](../04-applications/) per le guide all'utilizzo o a [Verifica](../05-verify/) per i test di smoke.
