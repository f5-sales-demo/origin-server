## Intégration du pool d'origines F5 XC

Ce serveur d'origine est conçu pour être utilisé comme **membre d'un pool d'origines** derrière un répartiteur de charge HTTP F5 Distributed Cloud. La plateforme F5 XC fournit le WAF, la défense contre les bots, la sécurité API et la défense côté client en amont de cette origine.

### Architecture

```
End User -> F5 XC HTTP LB (WAF/Bot/API/CSD) -> Origin Server VM (nginx -> Docker apps)
```

### Configuration du pool d'origines

Créez un pool d'origines F5 XC pointant vers l'adresse IP publique de cette VM :

| Paramètre | Valeur |
|---|---|
| **Type de serveur d'origine** | IP publique du serveur d'origine |
| **Adresse IP** | `<terraform output public_ip>` |
| **Port** | `80` |
| **Vérification de santé** | HTTP, chemin `/health` |

### Configuration du répartiteur de charge HTTP

Créez un répartiteur de charge HTTP avec le pool d'origines :

| Paramètre | Valeur |
|---|---|
| **Domaines** | Votre domaine de démonstration (ex. : `demo.example.com`) |
| **Pool d'origines** | Le pool créé ci-dessus |
| **WAF** | Attachez votre politique WAF |
| **Défense contre les bots** | Activez selon les besoins |
| **Découverte d'API** | Activez pour les points de terminaison VAmPI |

### Routage basé sur le chemin

Chaque application est accessible via son préfixe de chemin à travers le répartiteur de charge :

| URL du LB F5 XC | Chemin d'origine | Application |
|---|---|---|
| `https://demo.example.com/juice-shop/` | `/juice-shop/` | Juice Shop (4 instances, affinité par cookie) |
| `https://demo.example.com/dvwa/` | `/dvwa/` | DVWA (4 instances + MariaDB, affinité par cookie) |
| `https://demo.example.com/vampi/` | `/vampi/` | VAmPI (4 instances, affinité ip_hash) |
| `https://demo.example.com/httpbin/` | `/httpbin/` | httpbin (4 instances, round-robin) |
| `https://demo.example.com/whoami/` | `/whoami/` | Diagnostics de requêtes (4 instances) |
| `https://demo.example.com/csd-demo/` | `/csd-demo/` | CSD Demo (4 instances, affinité ip_hash) |
| `https://demo.example.com/dvga/` | `/dvga/` | DVGA (4 instances, affinité ip_hash) |
| `https://demo.example.com/restaurant/` | `/restaurant/` | RESTaurant (4 instances, round-robin) |
| `https://demo.example.com/health` | `/health` | Vérification de santé (nginx direct) |

### Vérification de l'injection d'en-têtes F5 XC

Utilisez le point de terminaison whoami pour vérifier quels en-têtes F5 XC injecte dans les requêtes atteignant l'origine :

```bash
LB_DOMAIN="demo.example.com"

curl -sk "https://${LB_DOMAIN}/whoami/"
```

Recherchez ces en-têtes injectés par F5 XC dans la réponse :

| En-tête | Signification |
|---|---|
| `X-Forwarded-For` | Chaîne d'adresses IP du client à travers F5 XC |
| `True-Client-IP` | Adresse IP originale du client |
| `X-Forwarded-Proto` | `https` si le TLS est terminé au niveau de F5 XC |
| `X-Volterra-Bot-Type` | Classification du bot (lorsque la défense contre les bots est activée) |
| `X-Request-ID` | Identifiant de suivi de requête F5 XC |

### Tests WAF à travers F5 XC

Avec une politique WAF attachée au répartiteur de charge HTTP, testez des charges utiles d'attaque :

```bash
LB_DOMAIN="demo.example.com"

# SQL Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--&Submit=Submit"

# XSS (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/juice-shop/rest/products/search?q=<script>alert(1)</script>"

# Command Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/exec/" \
  -d "ip=127.0.0.1;cat+/etc/passwd&Submit=Submit"

# Normal request (should pass through)
curl -sk "https://${LB_DOMAIN}/httpbin/get" | jq .
```

### Tests de sécurité API

Activez la découverte d'API sur le répartiteur de charge HTTP pour cartographier les points de terminaison de l'API VAmPI :

```bash
# These requests build the API inventory in F5 XC
curl -sk "https://${LB_DOMAIN}/vampi/users/v1"
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123","email":"api@test.com"}'
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123"}'
```

### Intégration de crAPI (Port 8888)

crAPI s'exécute sur un port dédié (8888) car il s'agit d'une application monopage qui code en dur ses chemins d'API et ne peut pas être servie derrière un préfixe de chemin. Pour intégrer crAPI avec F5 XC :

| Paramètre | Valeur |
|---|---|
| **Type de serveur d'origine** | IP publique du serveur d'origine |
| **Adresse IP** | `<terraform output public_ip>` |
| **Port** | `8888` |
| **Vérification de santé** | HTTP, chemin `/` |

Créez un **pool d'origines séparé** pour crAPI sur le port 8888, ou ajoutez un second membre de pool d'origines à votre pool existant avec le port 8888 et utilisez des règles de routage pour diriger le trafic :

```bash
# Test crAPI through F5 XC (if configured)
curl -sk "https://${LB_DOMAIN}:8888/"

# Or if using route rules on the same LB domain:
# Configure an F5 XC route rule matching Host header or path prefix
# to forward to the crAPI origin pool (port 8888)
```

:::note
Le NSG sur la VM d'origine autorise le trafic entrant sur le port 8888. Aucune modification réseau Azure supplémentaire n'est nécessaire.
:::

### Prise en compte des sessions persistantes

Le serveur d'origine utilise les sessions persistantes nginx en interne pour acheminer les applications avec état vers des conteneurs backend cohérents. Lors de la configuration du répartiteur de charge HTTP F5 XC, tenez compte de ce qui suit :

| Application | Méthode de persistance | Raison |
|---|---|---|
| Juice Shop | `hash $cookie_token` | État de session Node.js |
| DVWA | `hash $cookie_PHPSESSID` | État de session PHP |
| VAmPI | `ip_hash` | Base de données SQLite par instance |
| CSD Demo | `ip_hash` | Journal d'exfiltration en mémoire par instance |
| DVGA | `ip_hash` | Base de données SQLite par instance |
| RESTaurant | Round-robin | Backend PostgreSQL partagé |
| crAPI | -- (port unique 8888) | 7 microservices, PostgreSQL + MongoDB |
| httpbin | Round-robin | Sans état |
| whoami | Round-robin | Sans état |

F5 XC n'a pas besoin de répliquer ces sessions persistantes -- nginx sur la VM d'origine gère le routage backend. F5 XC doit traiter l'origine comme un point de terminaison unique (l'adresse IP publique de la VM sur le port 80).

### Architecture multi-composants

Combinée avec le simulateur CDN, l'architecture complète du laboratoire est la suivante :

```
End User -> CDN Simulator (nginx cache) -> F5 XC HTTP LB -> Origin Server (this component)
```

Le simulateur CDN utilise le VIP du répartiteur de charge HTTP F5 XC comme son origine, et le répartiteur de charge F5 XC utilise ce serveur d'origine comme son backend. Cela crée une architecture multi-couches réaliste pour des tests complets.
