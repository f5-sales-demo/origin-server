## एप्लिकेशन अवलोकन

प्रत्येक एप्लिकेशन nginx अपस्ट्रीम पूल के पीछे 4 लोड-बैलेंस्ड Docker कंटेनरों के रूप में चलता है, जो VM पर nginx रिवर्स प्रॉक्सी के माध्यम से सुलभ है। स्टिकी सेशन सुनिश्चित करते हैं कि स्टेटफुल एप्लिकेशन (Juice Shop, DVWA, VAmPI, CSD Demo) लगातार रूट होते रहें। सभी एप्लिकेशन जानबूझकर कमजोर हैं और सुरक्षा परीक्षण के लिए डिज़ाइन किए गए हैं।

नीचे दिए गए सभी उदाहरणों में `<ORIGIN>` को `http://<PUBLIC_IP>` से बदलें। Terraform के माध्यम से डिप्लॉय करने के बाद, `terraform output -raw public_ip` से IP प्राप्त करें।

## OWASP Juice Shop

| | |
|---|---|
| **पथ** | `/juice-shop/` |
| **इमेज** | `bkimminich/juice-shop:latest` |
| **इंस्टेंस** | 4 (पोर्ट 3001-3004), `hash $cookie_token` के माध्यम से स्टिकी, प्रॉक्सी कैश (60 s TTL) |
| **संसाधन** | 2 CPU / 1 GiB RAM प्रति इंस्टेंस |
| **फ्रेमवर्क** | Node.js / Angular |
| **प्रोजेक्ट** | [owasp.org/www-project-juice-shop](https://owasp.org/www-project-juice-shop/) |

OWASP Juice Shop सबसे आधुनिक और सक्रिय रूप से मेंटेन किया जाने वाला कमजोर वेब एप्लिकेशन है। यह एक यथार्थवादी ई-कॉमर्स एप्लिकेशन में 100+ चुनौतियों के साथ संपूर्ण OWASP Top 10 को कवर करता है।

### डेमो परिदृश्य

| परिदृश्य | श्रेणी | अटैक वेक्टर |
|---|---|---|
| SQL इंजेक्शन लॉगिन बाईपास | WAF | लॉगिन ईमेल फ़ील्ड में `' OR 1=1--` |
| रिफ्लेक्टेड XSS | WAF, CSD | सर्च पैरामीटर के माध्यम से स्क्रिप्ट इंजेक्शन |
| DOM-आधारित XSS | CSD | URL फ्रैगमेंट में पेलोड |
| टूटा हुआ प्रमाणीकरण | WAF | ब्रूट फोर्स लॉगिन, JWT मैनिपुलेशन |
| API दुरुपयोग | API सुरक्षा | `/api/` एंडपॉइंट्स तक अनधिकृत पहुँच |
| संवेदनशील डेटा एक्सपोज़र | API सुरक्षा | बिना प्राधिकरण के उपयोगकर्ता डेटा तक पहुँच |
| CSRF | WAF | प्रोफ़ाइल परिवर्तनों पर क्रॉस-साइट रिक्वेस्ट फ़ॉर्जरी |

### पहुँच

```bash
curl -s "http://<PUBLIC_IP>/juice-shop/" -o /dev/null -w "%{http_code}"
```

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **पथ** | `/dvwa/` |
| **इमेज** | कस्टम `dvwa-fpm:latest` (php-fpm + nginx, `ghcr.io/digininja/dvwa:latest` से बनाया गया) |
| **इंस्टेंस** | 4 (पोर्ट 8101-8104), `hash $cookie_PHPSESSID` के माध्यम से स्टिकी |
| **संसाधन** | 0.5 CPU / 256 MiB RAM प्रति इंस्टेंस |
| **डेटाबेस** | साझा MariaDB 10.11 (`dvwa-db` कंटेनर, 1 CPU / 768 MiB) |
| **फ्रेमवर्क** | PHP 8 / php-fpm / MariaDB |
| **क्रेडेंशियल** | `admin` / `password` |

DVWA WAF परीक्षण के लिए उद्योग मानक है। इसमें समायोज्य सुरक्षा स्तर (Low, Medium, High, Impossible) हैं जो प्रगतिशील रूप से इनपुट सत्यापन और आउटपुट एन्कोडिंग जोड़ते हैं।

### डेमो परिदृश्य

| परिदृश्य | श्रेणी | सुरक्षा स्तर |
|---|---|---|
| SQL इंजेक्शन | WAF | Low: सरल `' OR 1=1#`, High: ब्लाइंड SQLi |
| कमांड इंजेक्शन | WAF | Low: `; ls`, High: फ़िल्टर किए गए कैरेक्टर |
| फ़ाइल इन्क्लूजन | WAF | Low: सीधा पथ ट्रैवर्सल, High: सैनिटाइज़्ड |
| XSS (रिफ्लेक्टेड) | WAF, CSD | Low: बेसिक `<script>`, High: एन्कोडेड बाईपास |
| XSS (स्टोर्ड) | WAF, CSD | Low: गेस्टबुक में पर्सिस्टेंट स्क्रिप्ट |
| फ़ाइल अपलोड | WAF | Low: PHP शेल अपलोड, High: एक्सटेंशन फ़िल्टरिंग |
| ब्रूट फोर्स | बॉट डिफेंस | स्वचालित लॉगिन प्रयास |

### सुरक्षा स्तर

लॉगिन करने के बाद `/dvwa/security.php` पर सुरक्षा स्तर सेट करें:

- **Low** -- कोई इनपुट सत्यापन नहीं। हर हमला काम करता है। WAF सिग्नेचर प्रदर्शनों के लिए अच्छा है।
- **Medium** -- बेसिक फ़िल्टरिंग। कुछ हमलों के लिए एन्कोडिंग या बाईपास तकनीकों की आवश्यकता होती है।
- **High** -- मजबूत फ़िल्टरिंग। केवल उन्नत तकनीकें सफल होती हैं। WAF की सीमाएँ दिखाने के लिए अच्छा है।
- **Impossible** -- पूरी तरह से सुरक्षित कार्यान्वयन। उचित रक्षात्मक कोडिंग का प्रदर्शन करता है।

### प्रारंभिक सेटअप

DVWA को पहले डिप्लॉयमेंट के बाद एक बार डेटाबेस सेटअप की आवश्यकता होती है:

1. `http://<PUBLIC_IP>/dvwa/setup.php` पर नेविगेट करें
2. **Create / Reset Database** पर क्लिक करें
3. `admin` / `password` से लॉगिन करें

## VAmPI (Vulnerable API)

| | |
|---|---|
| **पथ** | `/vampi/` |
| **इमेज** | `erev0s/vampi:latest` gunicorn एंट्रीपॉइंट के साथ (4 वर्कर) |
| **इंस्टेंस** | 4 (पोर्ट 5101-5104), `ip_hash` के माध्यम से स्टिकी (प्रति इंस्टेंस SQLite) |
| **संसाधन** | 0.5 CPU / 512 MiB RAM प्रति इंस्टेंस |
| **फ्रेमवर्क** | Python / Flask / gunicorn |
| **प्रोजेक्ट** | [github.com/erev0s/VAmPI](https://github.com/erev0s/VAmPI) |

VAmPI विशेष रूप से OWASP API Security Top 10 के परीक्षण के लिए बनाया गया है। यह जानबूझकर कमजोरियों वाला एक यथार्थवादी REST API प्रदान करता है। प्रत्येक इंस्टेंस 4 वर्कर के साथ gunicorn और अपना स्वयं का SQLite डेटाबेस चलाता है। `ip_hash` स्टिकी सेशन सुनिश्चित करता है कि एक ही क्लाइंट IP से पंजीकरण और लॉगिन हमेशा उसी इंस्टेंस पर जाए।

### डेमो परिदृश्य

| परिदृश्य | OWASP API Top 10 | विधि |
|---|---|---|
| टूटा हुआ ऑब्जेक्ट लेवल ऑथराइज़ेशन (BOLA) | API1 | IDs में हेरफेर करके अन्य उपयोगकर्ताओं का डेटा एक्सेस करना |
| टूटा हुआ प्रमाणीकरण | API2 | कमजोर टोकन हैंडलिंग, कोई दर सीमा नहीं |
| अत्यधिक डेटा एक्सपोज़र | API3 | API क्लाइंट की आवश्यकता से अधिक डेटा लौटाता है |
| मास असाइनमेंट | API6 | अप्रत्याशित पैरामीटर के माध्यम से एडमिन फ़ील्ड संशोधित करना |
| SQL इंजेक्शन | API8 | API पैरामीटर के माध्यम से इंजेक्शन |
| अनुचित एसेट प्रबंधन | API9 | अनदस्तावेज़ित API एंडपॉइंट्स |

### API एंडपॉइंट्स

```bash
# नए उपयोगकर्ता का पंजीकरण
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123","email":"test@test.com"}'

# लॉगिन
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123"}'

# उपयोगकर्ताओं की सूची (अत्यधिक डेटा एक्सपोज़र)
curl "http://<PUBLIC_IP>/vampi/users/v1"
```

## httpbin

| | |
|---|---|
| **पथ** | `/httpbin/` |
| **इमेज** | `kennethreitz/httpbin:latest` gunicorn CMD ओवरराइड के साथ (`-w 4 -k gevent --timeout 30`) |
| **इंस्टेंस** | 4 (पोर्ट 8201-8204), राउंड-रॉबिन (स्टेटलेस) |
| **संसाधन** | 0.5 CPU / 256 MiB RAM प्रति इंस्टेंस |
| **फ्रेमवर्क** | Python / Flask / gunicorn + gevent |
| **प्रोजेक्ट** | [httpbin.org](https://httpbin.org) |

httpbin एक सरल HTTP अनुरोध/प्रतिक्रिया सेवा है जो बेसिक API डेमो, अनुरोध हेडर परीक्षण, और प्रॉक्सी व्यवहार सत्यापन के लिए उपयोगी है।

### उपयोगी एंडपॉइंट्स

| एंडपॉइंट | उद्देश्य |
|---|---|
| `/httpbin/get` | GET अनुरोध डेटा लौटाता है (हेडर, args, origin) |
| `/httpbin/post` | POST अनुरोध डेटा लौटाता है (body, form, JSON) |
| `/httpbin/headers` | अनुरोध हेडर लौटाता है |
| `/httpbin/ip` | ओरिजिन IP लौटाता है |
| `/httpbin/user-agent` | User-Agent हेडर लौटाता है |
| `/httpbin/status/:code` | निर्दिष्ट HTTP स्टेटस कोड लौटाता है |
| `/httpbin/delay/:seconds` | N सेकंड तक प्रतिक्रिया में देरी करता है |
| `/httpbin/anything` | अनुरोध में भेजी गई कोई भी चीज़ लौटाता है |

### उदाहरण

```bash
# जाँचें कि ओरिजिन कौन से हेडर देखता है (F5 XC हेडर इंजेक्शन सत्यापित करने के लिए उपयोगी)
curl -s "http://<PUBLIC_IP>/httpbin/headers" | jq .

# एक विशिष्ट HTTP स्टेटस कोड का परीक्षण करें
curl -s -o /dev/null -w "%{http_code}" "http://<PUBLIC_IP>/httpbin/status/403"
```

## whoami (अनुरोध डायग्नोस्टिक्स)

| | |
|---|---|
| **पथ** | `/whoami/` |
| **इमेज** | `traefik/whoami:latest` |
| **इंस्टेंस** | 4 (पोर्ट 8082-8085), राउंड-रॉबिन (स्टेटलेस) |
| **संसाधन** | 0.25 CPU / 64 MiB RAM प्रति इंस्टेंस |
| **फ्रेमवर्क** | Go |
| **प्रोजेक्ट** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami Traefik का हल्का अनुरोध इको सर्वर है। यह आने वाले HTTP अनुरोध का हर विवरण दिखाता है जैसा कि ओरिजिन उसे देखता है -- होस्टनेम, IP पते, सभी हेडर, मेथड, और URL। यह सबसे महत्वपूर्ण डायग्नोस्टिक टूल है जब यह सत्यापित करना हो कि F5 XC सही हेडर इंजेक्ट कर रहा है।

### प्रमुख उपयोग मामले

| उपयोग मामला | क्या देखना है |
|---|---|
| F5 XC हेडर इंजेक्शन सत्यापित करें | `X-Forwarded-For`, `True-Client-IP`, `X-Volterra-*` हेडर |
| क्लाइंट IP दृश्यता की पुष्टि करें | `X-Real-IP` बनाम `RemoteAddr` |
| WAF फॉल्स पॉज़िटिव डीबग करें | F5 XC से पहले/बाद अनुरोध हेडर की तुलना करें |
| बॉट डिफेंस टैगिंग सत्यापित करें | `X-Volterra-Bot-Type`, `X-Volterra-Bot-Verified` हेडर |
| TLS टर्मिनेशन जाँचें | `X-Forwarded-Proto` `https` दिखाता है जब TLS F5 XC पर टर्मिनेट होता है |

### उदाहरण

```bash
# बेसिक अनुरोध -- देखें कि ओरिजिन क्या प्राप्त करता है
curl "http://<PUBLIC_IP>/whoami/"

# F5 XC के माध्यम से अनुरोध सिमुलेट करें (इंजेक्ट किए गए हेडर के साथ)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

उदाहरण आउटपुट:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

| | |
|---|---|

DVGA विशेष रूप से GraphQL-विशिष्ट कमजोरियों के परीक्षण के लिए बनाया गया है। यह इंटरैक्टिव क्वेरी अन्वेषण के लिए `/dvga/` पर एक GraphiQL IDE और `/dvga/graphql` पर एक GraphQL API एंडपॉइंट प्रदान करता है। प्रत्येक इंस्टेंस अपना स्वयं का SQLite डेटाबेस उपयोग करता है, इसलिए `ip_hash` स्टिकी सेशन सुसंगत स्थिति सुनिश्चित करते हैं।

## RESTaurant (Damn Vulnerable RESTaurant API Game)

| | |
|---|---|
| **पथ** | `/restaurant/` |
| **इमेज** | `theowni/Damn-Vulnerable-RESTaurant-API-Game` से कस्टम बिल्ड |
| **इंस्टेंस** | 4 (पोर्ट 8301-8304), राउंड-रॉबिन (साझा PostgreSQL) |
| **संसाधन** | 0.5 CPU / 256 MiB RAM प्रति इंस्टेंस |
| **डेटाबेस** | साझा PostgreSQL 15.4 (`restaurant-db` कंटेनर, 0.5 CPU / 512 MiB) |
| **फ्रेमवर्क** | Python / FastAPI / PostgreSQL |
| **क्रेडेंशियल** | `admin` / `password` (PostgreSQL) |
| **प्रोजेक्ट** | [github.com/theowni/Damn-Vulnerable-RESTaurant-API-Game](https://github.com/theowni/Damn-Vulnerable-RESTaurant-API-Game) |

RESTaurant OWASP API Security Top 10 2023 को कवर करने वाला एक गेमिफाइड कमजोर REST API है। यह `/restaurant/docs` पर स्वचालित Swagger UI दस्तावेज़ीकरण के साथ FastAPI का उपयोग करता है। सभी 4 इंस्टेंस एक ही PostgreSQL डेटाबेस साझा करते हैं, इसलिए राउंड-रॉबिन लोड बैलेंसिंग स्टिकी सेशन के बिना काम करती है।

### डेमो परिदृश्य

| परिदृश्य | OWASP API Top 10 2023 | विधि |
|---|---|---|
| टूटा हुआ ऑब्जेक्ट लेवल ऑथराइज़ेशन (BOLA) | API1 | IDs में हेरफेर करके अन्य उपयोगकर्ताओं के ऑर्डर एक्सेस करना |
| टूटा हुआ प्रमाणीकरण | API2 | कमजोर टोकन हैंडलिंग, क्रेडेंशियल स्टफिंग |
| टूटा हुआ ऑब्जेक्ट प्रॉपर्टी लेवल ऑथराइज़ेशन | API3 | उपयोगकर्ता प्रोफ़ाइल फ़ील्ड पर मास असाइनमेंट |
| अप्रतिबंधित संसाधन उपभोग | API4 | एंडपॉइंट्स पर कोई दर सीमा नहीं |
| टूटा हुआ फंक्शन लेवल ऑथराइज़ेशन (BFLA) | API5 | सामान्य उपयोगकर्ता के रूप में एडमिन एंडपॉइंट्स एक्सेस करना |
| सर्वर साइड रिक्वेस्ट फ़ॉर्जरी (SSRF) | API7 | सर्वर-साइड URL अनुरोधों में हेरफेर करना |
| सुरक्षा गलत कॉन्फ़िगरेशन | API8 | वर्बोज़ त्रुटि संदेश, डिफ़ॉल्ट क्रेडेंशियल |

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **पोर्ट** | `8888` (समर्पित -- पथ-प्रीफिक्स्ड नहीं) |
| **इमेज** | `crapi/crapi-web`, `crapi/crapi-identity`, `crapi/crapi-community`, `crapi/crapi-workshop`, PostgreSQL, MongoDB, MailHog |
| **इंस्टेंस** | 7 माइक्रोसर्विसेज (प्रत्येक का एक इंस्टेंस) |
| **संसाधन** | ~3.0 CPU / ~2.0 GiB RAM कुल |
| **फ्रेमवर्क** | React SPA + Java/Go/Python माइक्रोसर्विसेज |
| **प्रोजेक्ट** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI API सुरक्षा परीक्षण के लिए OWASP का प्रमुख प्रोजेक्ट है। यह एक समर्पित पोर्ट (8888) पर 7 माइक्रोसर्विसेज के रूप में चलता है क्योंकि React SPA अपने API पथ हार्डकोड करता है और पथ प्रीफिक्स के पीछे सर्व नहीं किया जा सकता। NSG पोर्ट 8888 पर इनबाउंड ट्रैफ़िक की अनुमति देता है।

MailHog crAPI द्वारा भेजे गए सभी ईमेल कैप्चर करता है (खाता सत्यापन, पासवर्ड रीसेट)। पोर्ट 18025 पर SSH टनल के माध्यम से MailHog एक्सेस करें।

### चुनौती श्रेणियाँ

| श्रेणी | कमजोरियाँ |
|---|---|
| BOLA (टूटा हुआ ऑब्जेक्ट लेवल ऑथराइज़ेशन) | अन्य उपयोगकर्ताओं के वाहन, ऑर्डर और रिपोर्ट एक्सेस करना |
| BFLA (टूटा हुआ फंक्शन लेवल ऑथराइज़ेशन) | एडमिन तक एस्केलेट करना, प्रतिबंधित एंडपॉइंट्स एक्सेस करना |
| मास असाइनमेंट | API के माध्यम से संरक्षित फ़ील्ड (role, balance) संशोधित करना |
| SSRF (सर्वर साइड रिक्वेस्ट फ़ॉर्जरी) | सर्वर-साइड URL फ़ेचिंग में हेरफेर करना |
| JWT मैनिपुलेशन | विशेषाधिकार वृद्धि के लिए JWT टोकन जालसाजी या संशोधन |
| NoSQL इंजेक्शन | MongoDB-बैक्ड एंडपॉइंट्स में क्वेरी इंजेक्ट करना |
| अत्यधिक डेटा एक्सपोज़र | API संवेदनशील उपयोगकर्ता डेटा लौटाता है |

### शुरुआत करना

```bash
# सत्यापित करें कि crAPI चल रहा है
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# नए उपयोगकर्ता का साइनअप
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# लॉग इन करें
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# MailHog एक्सेस करें (ईमेल सत्यापन के लिए SSH टनल के माध्यम से)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# फिर अपने ब्राउज़र में http://localhost:18025 खोलें
```
