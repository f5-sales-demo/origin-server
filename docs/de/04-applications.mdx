## Anwendungsübersicht

Jede Anwendung wird als 4 lastverteilte Docker-Container hinter nginx-Upstream-Pools betrieben und ist über den nginx-Reverse-Proxy auf der VM erreichbar. Sticky Sessions stellen sicher, dass zustandsbehaftete Anwendungen (Juice Shop, DVWA, VAmPI, CSD Demo) konsistent geroutet werden. Alle Anwendungen sind absichtlich verwundbar und für Sicherheitstests konzipiert.

Ersetzen Sie `<ORIGIN>` in allen nachfolgenden Beispielen durch `http://<PUBLIC_IP>`. Nach dem Deployment über Terraform erhalten Sie die IP mit `terraform output -raw public_ip`.

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **Pfad** | `/dvwa/` |
| **Image** | Benutzerdefiniertes `dvwa-fpm:latest` (php-fpm + nginx, erstellt aus `ghcr.io/digininja/dvwa:latest`) |
| **Instanzen** | 4 (Ports 8101-8104), Sticky via `hash $cookie_PHPSESSID` |
| **Ressourcen** | 0,5 CPU / 256 MiB RAM pro Instanz |
| **Datenbank** | Gemeinsame MariaDB 10.11 (`dvwa-db` Container, 1 CPU / 768 MiB) |
| **Framework** | PHP 8 / php-fpm / MariaDB |
| **Zugangsdaten** | `admin` / `password` |

DVWA ist der Industriestandard für WAF-Tests. Es verfügt über einstellbare Sicherheitsstufen (Low, Medium, High, Impossible), die schrittweise Eingabevalidierung und Ausgabekodierung hinzufügen.

### Demo-Szenarien

| Szenario | Kategorie | Sicherheitsstufe |
|---|---|---|
| SQL-Injection | WAF | Low: trivial `' OR 1=1#`, High: Blind SQLi |
| Command Injection | WAF | Low: `; ls`, High: gefilterte Zeichen |
| File Inclusion | WAF | Low: direktes Path Traversal, High: bereinigt |
| XSS (Reflected) | WAF, CSD | Low: einfaches `<script>`, High: kodierter Bypass |
| XSS (Stored) | WAF, CSD | Low: persistentes Script im Gästebuch |
| File Upload | WAF | Low: PHP-Shell-Upload, High: Erweiterungsfilterung |
| Brute Force | Bot Defense | Automatisierte Login-Versuche |

### Sicherheitsstufen

Stellen Sie die Sicherheitsstufe unter `/dvwa/security.php` nach dem Login ein:

- **Low** -- Keine Eingabevalidierung. Jeder Angriff funktioniert. Gut für WAF-Signatur-Demonstrationen.
- **Medium** -- Grundlegende Filterung. Einige Angriffe erfordern Kodierung oder Bypass-Techniken.
- **High** -- Starke Filterung. Nur fortgeschrittene Techniken sind erfolgreich. Gut um WAF-Limitierungen aufzuzeigen.
- **Impossible** -- Vollständig sichere Implementierung. Demonstriert korrekte defensive Programmierung.

### Ersteinrichtung

DVWA erfordert eine einmalige Datenbankeinrichtung nach dem ersten Deployment:

1. Navigieren Sie zu `http://<PUBLIC_IP>/dvwa/setup.php`
2. Klicken Sie auf **Create / Reset Database**
3. Melden Sie sich mit `admin` / `password` an

VAmPI wurde speziell zum Testen der OWASP API Security Top 10 entwickelt. Es bietet eine realistische REST-API mit absichtlichen Schwachstellen. ## httpbin

| | |
|---|---|
| **Pfad** | `/httpbin/` |
| **Image** | `kennethreitz/httpbin:latest` mit gunicorn CMD-Override (`-w 4 -k gevent --timeout 30`) |
| **Instanzen** | 4 (Ports 8201-8204), Round-Robin (zustandslos) |
| **Ressourcen** | 0,5 CPU / 256 MiB RAM pro Instanz |
| **Framework** | Python / Flask / gunicorn + gevent |
| **Projekt** | [httpbin.org](https://httpbin.org) |

httpbin ist ein einfacher HTTP-Request/Response-Dienst, nützlich für grundlegende API-Demos, das Testen von Request-Headern und die Überprüfung des Proxy-Verhaltens.

### Nützliche Endpunkte

| Endpunkt | Zweck |
|---|---|
| `/httpbin/get` | Gibt GET-Request-Daten zurück (Header, Args, Origin) |
| `/httpbin/post` | Gibt POST-Request-Daten zurück (Body, Form, JSON) |
| `/httpbin/headers` | Gibt Request-Header zurück |
| `/httpbin/ip` | Gibt Quell-IP zurück |
| `/httpbin/user-agent` | Gibt User-Agent-Header zurück |
| `/httpbin/status/:code` | Gibt den angegebenen HTTP-Statuscode zurück |
| `/httpbin/delay/:seconds` | Verzögert die Antwort um N Sekunden |
| `/httpbin/anything` | Gibt alles zurück, was im Request übergeben wurde |

### Beispiel

```bash
# Prüfen, welche Header der Origin sieht (nützlich zur Überprüfung der F5 XC Header-Injection)
curl -s "http://<PUBLIC_IP>/httpbin/headers" | jq .

# Einen bestimmten HTTP-Statuscode testen
curl -s -o /dev/null -w "%{http_code}" "http://<PUBLIC_IP>/httpbin/status/403"
```

## whoami (Request-Diagnose)

| | |
|---|---|
| **Pfad** | `/whoami/` |
| **Image** | `traefik/whoami:latest` |
| **Instanzen** | 4 (Ports 8082-8085), Round-Robin (zustandslos) |
| **Ressourcen** | 0,25 CPU / 64 MiB RAM pro Instanz |
| **Framework** | Go |
| **Projekt** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami ist Traefiks leichtgewichtiger Request-Echo-Server. Er zeigt jedes Detail der eingehenden HTTP-Anfrage an, wie der Origin sie sieht -- Hostname, IP-Adressen, alle Header, Methode und URL. Dies ist das wichtigste Diagnosewerkzeug zur Überprüfung, ob F5 XC die korrekten Header injiziert.

### Wichtige Anwendungsfälle

| Anwendungsfall | Worauf zu achten ist |
|---|---|
| F5 XC Header-Injection überprüfen | `X-Forwarded-For`, `True-Client-IP`, `X-Volterra-*` Header |
| Client-IP-Sichtbarkeit bestätigen | `X-Real-IP` vs. `RemoteAddr` |
| WAF False Positives debuggen | Request-Header vor/nach F5 XC vergleichen |
| Bot-Defense-Tagging validieren | `X-Volterra-Bot-Type`, `X-Volterra-Bot-Verified` Header |
| TLS-Terminierung prüfen | `X-Forwarded-Proto` zeigt `https`, wenn TLS bei F5 XC terminiert wird |

### Beispiel

```bash
# Einfache Anfrage -- sehen, was der Origin empfängt
curl "http://<PUBLIC_IP>/whoami/"

# Anfrage durch F5 XC simulieren (mit injizierten Headern)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

Beispielausgabe:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA (Damn Vulnerable GraphQL Application)

| | |
|---|---|
| **Pfad** | `/dvga/` |
| **Image** | `dolevf/dvga:latest` |
| **Instanzen** | 4 (Ports 5201-5204), Sticky via `ip_hash` (SQLite pro Instanz) |
| **Ressourcen** | 0,5 CPU / 256 MiB RAM pro Instanz |
| **Framework** | Python / Flask / GraphQL |
| **Projekt** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA wurde speziell zum Testen von GraphQL-spezifischen Schwachstellen entwickelt. Es bietet eine GraphiQL-IDE unter `/dvga/` für interaktive Query-Erkundung und einen GraphQL-API-Endpunkt unter `/dvga/graphql`. Jede Instanz verwendet ihre eigene SQLite-Datenbank, sodass `ip_hash` Sticky Sessions einen konsistenten Zustand gewährleisten.

### Demo-Szenarien

| Szenario | Kategorie | Angriffsvektor |
|---|---|---|
| GraphQL-Injection | API Security | Bösartige Queries über String-Interpolation |
| Denial of Service | API Security | Tief verschachtelte Queries, Batch-Queries, Ressourcenerschöpfung |
| Autorisierungs-Bypass | API Security | Zugriff auf unautorisierte Daten über GraphQL |
| Informationsoffenlegung | API Security | Introspection-Queries, die Schema-Details preisgeben |
| Batching-Angriff | API Security | Mehrere Operationen in einer einzelnen Anfrage |

### Wichtige Endpunkte

```bash
# GraphiQL UI (interaktive IDE)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# Introspection-Query -- das vollständige Schema enumerieren
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# Pastes auflisten (Beispiel-Query)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# Einen Paste erstellen (Mutation)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **Port** | `8888` (dediziert -- nicht pfad-präfixiert) |
| **Images** | `crapi/crapi-web`, `crapi/crapi-identity`, `crapi/crapi-community`, `crapi/crapi-workshop`, PostgreSQL, MongoDB, MailHog |
| **Instanzen** | 7 Microservices (jeweils eine Instanz) |
| **Ressourcen** | ~3,0 CPU / ~2,0 GiB RAM insgesamt |
| **Framework** | React SPA + Java/Go/Python Microservices |
| **Projekt** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI ist das OWASP-Flaggschiffprojekt für API-Sicherheitstests. Es läuft als 7 Microservices auf einem dedizierten Port (8888), da die React-SPA ihre API-Pfade fest kodiert hat und nicht hinter einem Pfad-Präfix bereitgestellt werden kann. Die NSG erlaubt eingehenden Datenverkehr auf Port 8888.

MailHog erfasst alle von crAPI gesendeten E-Mails (Kontoverifizierung, Passwort-Zurücksetzung). Greifen Sie auf MailHog über einen SSH-Tunnel auf Port 18025 zu.

### Challenge-Kategorien

| Kategorie | Schwachstellen |
|---|---|
| BOLA (Broken Object Level Authorization) | Zugriff auf Fahrzeuge, Bestellungen und Berichte anderer Benutzer |
| BFLA (Broken Function Level Authorization) | Eskalation zum Admin, Zugriff auf eingeschränkte Endpunkte |
| Mass Assignment | Änderung geschützter Felder (Rolle, Guthaben) über die API |
| SSRF (Server Side Request Forgery) | Manipulation serverseitigen URL-Abrufs |
| JWT-Manipulation | Fälschen oder Ändern von JWT-Tokens zur Privilegienerhöhung |
| NoSQL-Injection | Injection von Queries in MongoDB-gestützte Endpunkte |
| Excessive Data Exposure | API gibt sensible Benutzerdaten zurück |

### Erste Schritte

```bash
# Überprüfen, ob crAPI läuft
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# Neuen Benutzer registrieren
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# Anmelden
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# Auf MailHog zugreifen (über SSH-Tunnel für E-Mail-Verifizierung)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# Dann http://localhost:18025 im Browser öffnen
```

### crAPI-Architektur

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, Benutzer-Auth, JWT)
             -> crapi-community (Go, Foren, Beiträge)
             -> crapi-workshop (Python, Fahrzeugservice)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (E-Mail-Erfassung, Port 18025)
```
