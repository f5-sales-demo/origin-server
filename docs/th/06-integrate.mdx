## การผสานรวม F5 XC Origin Pool

เซิร์ฟเวอร์ต้นทางนี้ถูกออกแบบมาเพื่อใช้เป็น **สมาชิก origin pool** ที่อยู่เบื้องหลัง F5 Distributed Cloud HTTP load balancer แพลตฟอร์ม F5 XC มีความสามารถด้าน WAF, Bot Defense, API Security และ Client-Side Defense อยู่ด้านหน้าของ origin นี้

### สถาปัตยกรรม

```
End User -> F5 XC HTTP LB (WAF/Bot/API/CSD) -> Origin Server VM (nginx -> Docker apps)
```

### การกำหนดค่า Origin Pool

สร้าง F5 XC origin pool โดยชี้ไปยัง public IP ของ VM นี้:

| การตั้งค่า | ค่า |
|---|---|
| **Origin Server Type** | Public IP of Origin Server |
| **IP Address** | `<terraform output public_ip>` |
| **Port** | `80` |
| **Health Check** | HTTP, path `/health` |

### การกำหนดค่า HTTP Load Balancer

สร้าง HTTP load balancer พร้อม origin pool:

| การตั้งค่า | ค่า |
|---|---|
| **Domains** | โดเมนสาธิตของคุณ (เช่น `demo.example.com`) |
| **Origin Pool** | pool ที่สร้างไว้ข้างต้น |
| **WAF** | แนบนโยบาย WAF ของคุณ |
| **Bot Defense** | เปิดใช้งานตามต้องการ |
| **API Discovery** | เปิดใช้งานสำหรับ VAmPI endpoints |

### การกำหนดเส้นทางตาม Path

แต่ละแอปพลิเคชันสามารถเข้าถึงได้ผ่าน path prefix ผ่าน load balancer:

| F5 XC LB URL | Origin Path | แอปพลิเคชัน |
|---|---|---|
| `https://demo.example.com/juice-shop/` | `/juice-shop/` | Juice Shop (4 instances, cookie sticky) |
| `https://demo.example.com/dvwa/` | `/dvwa/` | DVWA (4 instances + MariaDB, cookie sticky) |
| `https://demo.example.com/vampi/` | `/vampi/` | VAmPI (4 instances, ip_hash sticky) |
| `https://demo.example.com/httpbin/` | `/httpbin/` | httpbin (4 instances, round-robin) |
| `https://demo.example.com/whoami/` | `/whoami/` | Request diagnostics (4 instances) |
| `https://demo.example.com/csd-demo/` | `/csd-demo/` | CSD Demo (4 instances, ip_hash sticky) |
| `https://demo.example.com/dvga/` | `/dvga/` | DVGA (4 instances, ip_hash sticky) |
| `https://demo.example.com/restaurant/` | `/restaurant/` | RESTaurant (4 instances, round-robin) |
| `https://demo.example.com/health` | `/health` | Health check (nginx direct) |

### ตรวจสอบการแทรก Header ของ F5 XC

ใช้ whoami endpoint เพื่อตรวจสอบว่า F5 XC แทรก header ใดบ้างในคำร้องขอที่ส่งไปยัง origin:

```bash
LB_DOMAIN="demo.example.com"

curl -sk "https://${LB_DOMAIN}/whoami/"
```

มองหา header ที่ F5 XC แทรกเข้ามาในการตอบกลับเหล่านี้:

| Header | ความหมาย |
|---|---|
| `X-Forwarded-For` | ห่วงโซ่ Client IP ผ่าน F5 XC |
| `True-Client-IP` | Client IP ดั้งเดิม |
| `X-Forwarded-Proto` | `https` หาก TLS สิ้นสุดที่ F5 XC |
| `X-Volterra-Bot-Type` | การจำแนกประเภท Bot (เมื่อเปิดใช้งาน Bot Defense) |
| `X-Request-ID` | ID การติดตามคำร้องขอของ F5 XC |

### การทดสอบ WAF ผ่าน F5 XC

เมื่อแนบนโยบาย WAF กับ HTTP load balancer แล้ว ทดสอบด้วย payload การโจมตี:

```bash
LB_DOMAIN="demo.example.com"

# SQL Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--&Submit=Submit"

# XSS (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/juice-shop/rest/products/search?q=<script>alert(1)</script>"

# Command Injection (should be blocked by WAF)
curl -sk "https://${LB_DOMAIN}/dvwa/vulnerabilities/exec/" \
  -d "ip=127.0.0.1;cat+/etc/passwd&Submit=Submit"

# Normal request (should pass through)
curl -sk "https://${LB_DOMAIN}/httpbin/get" | jq .
```

### การทดสอบ API Security

เปิดใช้งาน API Discovery บน HTTP load balancer เพื่อทำแผนที่ API endpoints ของ VAmPI:

```bash
# These requests build the API inventory in F5 XC
curl -sk "https://${LB_DOMAIN}/vampi/users/v1"
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123","email":"api@test.com"}'
curl -sk -X POST "https://${LB_DOMAIN}/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"apitest","password":"test123"}'
```

### การผสานรวม crAPI (พอร์ต 8888)

crAPI ทำงานบนพอร์ตเฉพาะ (8888) เนื่องจากเป็น single-page application ที่กำหนด API paths แบบ hardcode ไว้และไม่สามารถให้บริการภายใต้ path prefix ได้ ในการผสานรวม crAPI กับ F5 XC:

| การตั้งค่า | ค่า |
|---|---|
| **Origin Server Type** | Public IP of Origin Server |
| **IP Address** | `<terraform output public_ip>` |
| **Port** | `8888` |
| **Health Check** | HTTP, path `/` |

สร้าง **origin pool แยกต่างหาก** สำหรับ crAPI บนพอร์ต 8888 หรือเพิ่มสมาชิก origin pool ที่สองในพูลที่มีอยู่พร้อมพอร์ต 8888 และใช้ route rules เพื่อกำหนดทิศทางของทราฟฟิก:

```bash
# Test crAPI through F5 XC (if configured)
curl -sk "https://${LB_DOMAIN}:8888/"

# Or if using route rules on the same LB domain:
# Configure an F5 XC route rule matching Host header or path prefix
# to forward to the crAPI origin pool (port 8888)
```

:::note
NSG บน origin VM อนุญาตทราฟฟิกขาเข้าบนพอร์ต 8888 ไม่จำเป็นต้องเปลี่ยนแปลงเครือข่าย Azure เพิ่มเติม
:::

### การรับรู้ Sticky Session

เซิร์ฟเวอร์ต้นทางใช้ nginx sticky sessions ภายในเพื่อกำหนดเส้นทางแอปพลิเคชันที่มีสถานะไปยัง backend container ที่สอดคล้องกัน เมื่อกำหนดค่า F5 XC HTTP load balancer โปรดทราบ:

| แอปพลิเคชัน | วิธี Sticky | เหตุผล |
|---|---|---|
| Juice Shop | `hash $cookie_token` | สถานะ session ของ Node.js |
| DVWA | `hash $cookie_PHPSESSID` | สถานะ session ของ PHP |
| VAmPI | `ip_hash` | ฐานข้อมูล SQLite ต่อ instance |
| CSD Demo | `ip_hash` | บันทึก exfil ใน memory ต่อ instance |
| DVGA | `ip_hash` | ฐานข้อมูล SQLite ต่อ instance |
| RESTaurant | Round-robin | ใช้ PostgreSQL backend ร่วมกัน |
| crAPI | -- (พอร์ตเดียว 8888) | 7 microservices, PostgreSQL + MongoDB |
| httpbin | Round-robin | ไม่มีสถานะ |
| whoami | Round-robin | ไม่มีสถานะ |

F5 XC ไม่จำเป็นต้องจำลอง sticky sessions เหล่านี้ -- nginx บน origin VM จัดการการกำหนดเส้นทาง backend เอง F5 XC ควรถือว่า origin เป็น endpoint เดียว (public IP ของ VM บนพอร์ต 80)

### สถาปัตยกรรมหลายองค์ประกอบ

เมื่อรวมกับ CDN Simulator สถาปัตยกรรมแล็บเต็มรูปแบบจะเป็น:

```
End User -> CDN Simulator (nginx cache) -> F5 XC HTTP LB -> Origin Server (this component)
```

CDN Simulator ใช้ F5 XC HTTP load balancer VIP เป็น origin ของมัน และ F5 XC load balancer ใช้เซิร์ฟเวอร์ต้นทางนี้เป็น backend สิ่งนี้สร้างสถาปัตยกรรมหลายชั้นที่สมจริงสำหรับการทดสอบอย่างครอบคลุม
