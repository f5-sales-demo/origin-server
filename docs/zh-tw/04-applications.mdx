## 應用程式概覽

每個應用程式以 4 個負載平衡的 Docker 容器運行，位於 nginx 上游池後方，可透過 VM 上的 nginx 反向代理存取。黏性工作階段確保有狀態的應用程式（Juice Shop、DVWA、VAmPI、CSD Demo）路由一致。所有應用程式均為刻意設計的易受攻擊應用程式，專為安全性測試而設計。

在以下所有範例中，請將 `<ORIGIN>` 替換為 `http://<PUBLIC_IP>`。透過 Terraform 部署後，使用 `terraform output -raw public_ip` 取得 IP。

## OWASP Juice Shop

| | |
|---|---|
| **路徑** | `/juice-shop/` |
| **映像檔** | `bkimminich/juice-shop:latest` |
| **實例** | 4 個（連接埠 3001-3004），透過 `hash $cookie_token` 進行黏性路由，代理快取（60 秒 TTL） |
| **資源** | 每個實例 2 CPU / 1 GiB RAM |
| **框架** | Node.js / Angular |
| **專案** | [owasp.org/www-project-juice-shop](https://owasp.org/www-project-juice-shop/) |

OWASP Juice Shop 是最現代且積極維護的易受攻擊 Web 應用程式。它以逼真的電子商務應用程式涵蓋整個 OWASP Top 10，包含 100 多個挑戰。

### 示範場景

| 場景 | 類別 | 攻擊向量 |
|---|---|---|
| SQL 注入登入繞過 | WAF | 在登入電子郵件欄位中輸入 `' OR 1=1--` |
| 反射型 XSS | WAF、CSD | 透過搜尋參數進行腳本注入 |
| DOM 型 XSS | CSD | URL 片段中的有效負載 |
| 驗證機制缺陷 | WAF | 暴力破解登入、JWT 操縱 |
| API 濫用 | API 安全性 | 未經授權存取 `/api/` 端點 |
| 敏感資料曝露 | API 安全性 | 未經授權存取使用者資料 |
| CSRF | WAF | 針對個人資料變更的跨站請求偽造 |

### 存取方式

```bash
curl -s "http://<PUBLIC_IP>/juice-shop/" -o /dev/null -w "%{http_code}"
```

## DVWA（Damn Vulnerable Web Application）

| | |
|---|---|
| **路徑** | `/dvwa/` |
| **映像檔** | 自訂 `dvwa-fpm:latest`（php-fpm + nginx，從 `ghcr.io/digininja/dvwa:latest` 建置） |
| **實例** | 4 個（連接埠 8101-8104），透過 `hash $cookie_PHPSESSID` 進行黏性路由 |
| **資源** | 每個實例 0.5 CPU / 256 MiB RAM |
| **資料庫** | 共享 MariaDB 10.11（`dvwa-db` 容器，1 CPU / 768 MiB） |
| **框架** | PHP 8 / php-fpm / MariaDB |
| **憑證** | `admin` / `password` |

DVWA 是 WAF 測試的業界標準。它具有可調整的安全等級（Low、Medium、High、Impossible），逐步增加輸入驗證和輸出編碼。

### 示範場景

| 場景 | 類別 | 安全等級 |
|---|---|---|
| SQL 注入 | WAF | Low：簡單的 `' OR 1=1#`，High：盲注 SQLi |
| 命令注入 | WAF | Low：`; ls`，High：字元過濾 |
| 檔案包含 | WAF | Low：直接路徑遍歷，High：已清理 |
| XSS（反射型） | WAF、CSD | Low：基本 `<script>`，High：編碼繞過 |
| XSS（儲存型） | WAF、CSD | Low：留言板中的持久性腳本 |
| 檔案上傳 | WAF | Low：PHP shell 上傳，High：副檔名過濾 |
| 暴力破解 | Bot 防禦 | 自動化登入嘗試 |

### 安全等級

登入後在 `/dvwa/security.php` 設定安全等級：

- **Low** -- 無輸入驗證。每種攻擊都有效。適合 WAF 簽章示範。
- **Medium** -- 基本過濾。某些攻擊需要編碼或繞過技術。
- **High** -- 強力過濾。只有進階技術才能成功。適合展示 WAF 的限制。
- **Impossible** -- 完全安全的實作。展示正確的防禦性編碼。

### 初始設定

DVWA 在首次部署後需要一次性的資料庫設定：

1. 導覽至 `http://<PUBLIC_IP>/dvwa/setup.php`
2. 點擊 **Create / Reset Database**
3. 使用 `admin` / `password` 登入

## VAmPI（Vulnerable API）

| | |
|---|---|
| **路徑** | `/vampi/` |
| **映像檔** | `erev0s/vampi:latest`，使用 gunicorn 進入點（4 個 worker） |
| **實例** | 4 個（連接埠 5101-5104），透過 `ip_hash` 進行黏性路由（每個實例獨立 SQLite） |
| **資源** | 每個實例 0.5 CPU / 512 MiB RAM |
| **框架** | Python / Flask / gunicorn |
| **專案** | [github.com/erev0s/VAmPI](https://github.com/erev0s/VAmPI) |

VAmPI 專為測試 OWASP API 安全性 Top 10 而設計。它提供具有刻意漏洞的逼真 REST API。每個實例執行 gunicorn（4 個 worker）並擁有自己的 SQLite 資料庫。`ip_hash` 黏性工作階段確保來自相同用戶端 IP 的註冊和登入始終命中相同的實例。

### 示範場景

| 場景 | OWASP API Top 10 | 方法 |
|---|---|---|
| 物件層級授權缺陷（BOLA） | API1 | 透過操縱 ID 存取其他使用者的資料 |
| 驗證機制缺陷 | API2 | 弱權杖處理、無速率限制 |
| 過度資料曝露 | API3 | API 回傳超過用戶端所需的資料 |
| 大量賦值 | API6 | 透過非預期參數修改管理員欄位 |
| SQL 注入 | API8 | 透過 API 參數進行注入 |
| 不當資產管理 | API9 | 未記錄的 API 端點 |

### API 端點

```bash
# Register a new user
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123","email":"test@test.com"}'

# Login
curl -X POST "http://<PUBLIC_IP>/vampi/users/v1/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test123"}'

# List users (excessive data exposure)
curl "http://<PUBLIC_IP>/vampi/users/v1"
```

## httpbin

| | |
|---|---|
| **路徑** | `/httpbin/` |
| **映像檔** | `kennethreitz/httpbin:latest`，使用 gunicorn CMD 覆寫（`-w 4 -k gevent --timeout 30`） |
| **實例** | 4 個（連接埠 8201-8204），輪詢（無狀態） |
| **資源** | 每個實例 0.5 CPU / 256 MiB RAM |
| **框架** | Python / Flask / gunicorn + gevent |
| **專案** | [httpbin.org](https://httpbin.org) |

httpbin 是一個簡單的 HTTP 請求/回應服務，適用於基本 API 示範、測試請求標頭和驗證代理行為。

### 實用端點

| 端點 | 用途 |
|---|---|
| `/httpbin/get` | 回傳 GET 請求資料（標頭、參數、來源） |
| `/httpbin/post` | 回傳 POST 請求資料（主體、表單、JSON） |
| `/httpbin/headers` | 回傳請求標頭 |
| `/httpbin/ip` | 回傳來源 IP |
| `/httpbin/user-agent` | 回傳 User-Agent 標頭 |
| `/httpbin/status/:code` | 回傳指定的 HTTP 狀態碼 |
| `/httpbin/delay/:seconds` | 延遲 N 秒後回應 |
| `/httpbin/anything` | 回傳請求中傳入的所有內容 |

### 範例

```bash
# Check what headers the origin sees (useful for verifying F5 XC header injection)
curl -s "http://<PUBLIC_IP>/httpbin/headers" | jq .

# Test a specific HTTP status code
curl -s -o /dev/null -w "%{http_code}" "http://<PUBLIC_IP>/httpbin/status/403"
```

## whoami（請求診斷）

| | |
|---|---|
| **路徑** | `/whoami/` |
| **映像檔** | `traefik/whoami:latest` |
| **實例** | 4 個（連接埠 8082-8085），輪詢（無狀態） |
| **資源** | 每個實例 0.25 CPU / 64 MiB RAM |
| **框架** | Go |
| **專案** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami 是 Traefik 的輕量級請求回聲伺服器。它顯示來源伺服器所看到的每個傳入 HTTP 請求的詳細資訊——主機名稱、IP 位址、所有標頭、方法和 URL。這是驗證 F5 XC 是否注入正確標頭時最重要的診斷工具。

### 主要使用案例

| 使用案例 | 需觀察的內容 |
|---|---|
| 驗證 F5 XC 標頭注入 | `X-Forwarded-For`、`True-Client-IP`、`X-Volterra-*` 標頭 |
| 確認用戶端 IP 可見性 | `X-Real-IP` 與 `RemoteAddr` 的比較 |
| 除錯 WAF 誤判 | 比較 F5 XC 前後的請求標頭 |
| 驗證 Bot 防禦標記 | `X-Volterra-Bot-Type`、`X-Volterra-Bot-Verified` 標頭 |
| 檢查 TLS 終止 | `X-Forwarded-Proto` 在 TLS 於 F5 XC 終止時顯示 `https` |

### 範例

```bash
# Basic request -- see what the origin receives
curl "http://<PUBLIC_IP>/whoami/"

# Simulate request through F5 XC (with injected headers)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

範例輸出：

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA（Damn Vulnerable GraphQL Application）

| | |
|---|---|
| **路徑** | `/dvga/` |
| **映像檔** | `dolevf/dvga:latest` |
| **實例** | 4 個（連接埠 5201-5204），透過 `ip_hash` 進行黏性路由（每個實例獨立 SQLite） |
| **資源** | 每個實例 0.5 CPU / 256 MiB RAM |
| **框架** | Python / Flask / GraphQL |
| **專案** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA 專為測試 GraphQL 特定漏洞而設計。它在 `/dvga/` 提供 GraphiQL IDE 供互動式查詢探索，並在 `/dvga/graphql` 提供 GraphQL API 端點。每個實例使用自己的 SQLite 資料庫，因此 `ip_hash` 黏性工作階段確保狀態一致。

### 示範場景

| 場景 | 類別 | 攻擊向量 |
|---|---|---|
| GraphQL 注入 | API 安全性 | 透過字串插值的惡意查詢 |
| 阻斷服務 | API 安全性 | 深度巢狀查詢、批次查詢、資源耗盡 |
| 授權繞過 | API 安全性 | 透過 GraphQL 存取未經授權的資料 |
| 資訊揭露 | API 安全性 | 內省查詢揭露結構描述詳細資訊 |
| 批次攻擊 | API 安全性 | 單一請求中包含多個操作 |

### 主要端點

```bash
# GraphiQL UI (interactive IDE)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# Introspection query -- enumerate the full schema
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# List pastes (example query)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# Create a paste (mutation)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

| **路徑** | `/restaurant/` |

RESTaurant 是一個遊戲化的易受攻擊 REST API，涵蓋 OWASP API 安全性 Top 10 2023。它使用 FastAPI 並在 `/restaurant/docs` 提供自動化 Swagger UI 文件。所有 4 個實例共享單一 PostgreSQL 資料庫，因此輪詢負載平衡無需黏性工作階段即可運作。

### 示範場景

| 場景 | OWASP API Top 10 2023 | 方法 |
|---|---|---|

| 不受限制的資源消耗 | API4 | 端點無速率限制 |
| 功能層級授權缺陷（BFLA） | API5 | 以一般使用者身份存取管理員端點 |

## crAPI（OWASP Completely Ridiculous API）

| | |
|---|---|
| **連接埠** | `8888`（專用——非路徑前綴） |
| **映像檔** | `crapi/crapi-web`、`crapi/crapi-identity`、`crapi/crapi-community`、`crapi/crapi-workshop`、PostgreSQL、MongoDB、MailHog |
| **實例** | 7 個微服務（每個單一實例） |
| **資源** | 總計約 3.0 CPU / 約 2.0 GiB RAM |
| **框架** | React SPA + Java/Go/Python 微服務 |
| **專案** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI 是 OWASP 旗艦的 API 安全性測試專案。它以 7 個微服務運行在專用連接埠（8888）上，因為 React SPA 硬編碼了其 API 路徑，無法在路徑前綴後方提供服務。NSG 允許連接埠 8888 的入站流量。

MailHog 擷取 crAPI 發送的所有電子郵件（帳戶驗證、密碼重設）。透過連接埠 18025 的 SSH 通道存取 MailHog。

### 挑戰類別

| 類別 | 漏洞 |
|---|---|
| BOLA（物件層級授權缺陷） | 存取其他使用者的車輛、訂單和報告 |
| BFLA（功能層級授權缺陷） | 提權至管理員、存取受限端點 |
| 大量賦值 | 透過 API 修改受保護的欄位（角色、餘額） |
| SSRF（伺服器端請求偽造） | 操縱伺服器端 URL 擷取 |
| JWT 操縱 | 偽造或修改 JWT 權杖以進行權限提升 |
| NoSQL 注入 | 對 MongoDB 支援的端點注入查詢 |
| 過度資料曝露 | API 回傳敏感的使用者資料 |

### 開始使用

```bash
# Verify crAPI is running
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# Sign up a new user
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# Log in
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# Access MailHog (via SSH tunnel for email verification)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# Then open http://localhost:18025 in your browser
```

### crAPI 架構

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, user auth, JWT)
             -> crapi-community (Go, forums, posts)
             -> crapi-workshop (Python, vehicle service)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (email capture, port 18025)
```
