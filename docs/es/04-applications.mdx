## Descripción general de las aplicaciones

Cada aplicación se ejecuta como 4 contenedores Docker con balanceo de carga detrás de pools upstream de nginx, accesibles a través del proxy inverso nginx en la VM. Las sesiones persistentes aseguran que las aplicaciones con estado (Juice Shop, DVWA, VAmPI, CSD Demo) se enruten de manera consistente. Todas las aplicaciones son intencionalmente vulnerables y están diseñadas para pruebas de seguridad.

Reemplace `<ORIGIN>` con `http://<PUBLIC_IP>` en todos los ejemplos a continuación. Después de desplegar mediante Terraform, obtenga la IP con `terraform output -raw public_ip`.

OWASP Juice Shop es la aplicación web vulnerable más moderna y activamente mantenida. Cubre todo el OWASP Top 10 con más de 100 desafíos en una aplicación de comercio electrónico realista.

### Escenarios de demostración

| Escenario | Categoría | Vector de ataque |
|---|---|---|
| Bypass de login por inyección SQL | WAF | `' OR 1=1--` en el campo de email de inicio de sesión |
| XSS reflejado | WAF, CSD | Inyección de script vía parámetro de búsqueda |
| XSS basado en DOM | CSD | Payload en fragmento de URL |
| Autenticación rota | WAF | Fuerza bruta de login, manipulación de JWT |
| Abuso de API | Seguridad de API | Acceso no autorizado a endpoints `/api/` |
| Exposición de datos sensibles | Seguridad de API | Acceso a datos de usuario sin autorización |
| CSRF | WAF | Falsificación de solicitud entre sitios en cambios de perfil |

## DVWA (Damn Vulnerable Web Application)

| | |
|---|---|
| **Ruta** | `/dvwa/` |
| **Imagen** | Personalizada `dvwa-fpm:latest` (php-fpm + nginx, construida desde `ghcr.io/digininja/dvwa:latest`) |
| **Instancias** | 4 (puertos 8101-8104), persistencia vía `hash $cookie_PHPSESSID` |
| **Recursos** | 0.5 CPU / 256 MiB RAM por instancia |
| **Base de datos** | MariaDB 10.11 compartida (contenedor `dvwa-db`, 1 CPU / 768 MiB) |
| **Framework** | PHP 8 / php-fpm / MariaDB |
| **Credenciales** | `admin` / `password` |

DVWA es el estándar de la industria para pruebas de WAF. Tiene niveles de seguridad ajustables (Low, Medium, High, Impossible) que agregan progresivamente validación de entrada y codificación de salida.

### Escenarios de demostración

| Escenario | Categoría | Nivel de seguridad |
|---|---|---|
| Inyección SQL | WAF | Low: trivial `' OR 1=1#`, High: SQLi ciega |
| Inyección de comandos | WAF | Low: `; ls`, High: caracteres filtrados |
| Inclusión de archivos | WAF | Low: recorrido directo de rutas, High: saneado |
| XSS (Reflejado) | WAF, CSD | Low: básico `<script>`, High: bypass con codificación |
| XSS (Almacenado) | WAF, CSD | Low: script persistente en libro de visitas |
| Carga de archivos | WAF | Low: carga de shell PHP, High: filtrado de extensiones |
| Fuerza bruta | Defensa contra bots | Intentos de inicio de sesión automatizados |

### Niveles de seguridad

Configure el nivel de seguridad en `/dvwa/security.php` después de iniciar sesión:

- **Low** -- Sin validación de entrada. Todos los ataques funcionan. Ideal para demostraciones de firmas WAF.
- **Medium** -- Filtrado básico. Algunos ataques requieren codificación o técnicas de bypass.
- **High** -- Filtrado fuerte. Solo técnicas avanzadas tienen éxito. Ideal para mostrar limitaciones del WAF.
- **Impossible** -- Implementación completamente segura. Demuestra codificación defensiva adecuada.

### Configuración inicial

DVWA requiere una configuración de base de datos única después del primer despliegue:

1. Navegue a `http://<PUBLIC_IP>/dvwa/setup.php`
2. Haga clic en **Create / Reset Database**
3. Inicie sesión con `admin` / `password`

## httpbin

| | |
|---|---|

| **Imagen** | `kennethreitz/httpbin:latest` con CMD override de gunicorn (`-w 4 -k gevent --timeout 30`) |

httpbin es un servicio simple de solicitud/respuesta HTTP útil para demostraciones básicas de API, pruebas de encabezados de solicitud y verificación del comportamiento del proxy.

### Endpoints útiles

| Endpoint | Propósito |

## whoami (Diagnósticos de solicitud)

| | |
|---|---|
| **Ruta** | `/whoami/` |
| **Imagen** | `traefik/whoami:latest` |
| **Instancias** | 4 (puertos 8082-8085), round-robin (sin estado) |
| **Recursos** | 0.25 CPU / 64 MiB RAM por instancia |
| **Framework** | Go |
| **Proyecto** | [github.com/traefik/whoami](https://github.com/traefik/whoami) |

whoami es el servidor ligero de eco de solicitudes de Traefik. Muestra cada detalle de la solicitud HTTP entrante tal como la ve el origen: nombre de host, direcciones IP, todos los encabezados, método y URL. Esta es la herramienta de diagnóstico más importante al verificar que F5 XC está inyectando los encabezados correctos.

### Casos de uso clave

| Caso de uso | Qué buscar |
|---|---|
| Verificar inyección de encabezados de F5 XC | Encabezados `X-Forwarded-For`, `True-Client-IP`, `X-Volterra-*` |
| Confirmar visibilidad de IP del cliente | `X-Real-IP` vs `RemoteAddr` |
| Depurar falsos positivos del WAF | Comparar encabezados de solicitud antes/después de F5 XC |
| Validar etiquetado de defensa contra bots | Encabezados `X-Volterra-Bot-Type`, `X-Volterra-Bot-Verified` |
| Verificar terminación TLS | `X-Forwarded-Proto` muestra `https` cuando TLS termina en F5 XC |

### Ejemplo

```bash
# Solicitud básica -- ver qué recibe el origen
curl "http://<PUBLIC_IP>/whoami/"

# Simular solicitud a través de F5 XC (con encabezados inyectados)
curl "http://<PUBLIC_IP>/whoami/" \
  -H "X-Forwarded-For: 203.0.113.50" \
  -H "True-Client-IP: 203.0.113.50" \
  -H "X-Forwarded-Proto: https"
```

Salida de ejemplo:

```
Hostname: 534c5084e169
IP: 127.0.0.1
RemoteAddr: 172.17.0.1:55118
GET / HTTP/1.1
Host: 20.12.78.159
User-Agent: curl/8.5.0
Accept: */*
True-Client-Ip: 203.0.113.50
X-Forwarded-For: 203.0.113.50, 10.0.0.1
X-Forwarded-Proto: https
X-Real-Ip: 104.219.105.84
```

## DVGA (Damn Vulnerable GraphQL Application)

| | |
|---|---|
| **Ruta** | `/dvga/` |
| **Imagen** | `dolevf/dvga:latest` |
| **Instancias** | 4 (puertos 5201-5204), persistencia vía `ip_hash` (SQLite por instancia) |
| **Recursos** | 0.5 CPU / 256 MiB RAM por instancia |
| **Framework** | Python / Flask / GraphQL |
| **Proyecto** | [github.com/dolevf/Damn-Vulnerable-GraphQL-Application](https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application) |

DVGA está diseñada específicamente para probar vulnerabilidades específicas de GraphQL. Proporciona un IDE GraphiQL en `/dvga/` para exploración interactiva de consultas y un endpoint de API GraphQL en `/dvga/graphql`. Cada instancia usa su propia base de datos SQLite, por lo que las sesiones persistentes `ip_hash` aseguran un estado consistente.

### Escenarios de demostración

| Escenario | Categoría | Vector de ataque |
|---|---|---|
| Inyección GraphQL | Seguridad de API | Consultas maliciosas vía interpolación de cadenas |
| Denegación de servicio | Seguridad de API | Consultas profundamente anidadas, consultas por lotes, agotamiento de recursos |
| Bypass de autorización | Seguridad de API | Acceso a datos no autorizados a través de GraphQL |
| Divulgación de información | Seguridad de API | Consultas de introspección revelando detalles del esquema |
| Ataque por lotes | Seguridad de API | Múltiples operaciones en una sola solicitud |

### Endpoints clave

```bash
# UI GraphiQL (IDE interactivo)
curl -sf "http://<PUBLIC_IP>/dvga/" -o /dev/null -w "%{http_code}"

# Consulta de introspección -- enumerar el esquema completo
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# Listar pastes (consulta de ejemplo)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ pastes { title content } }"}'

# Crear un paste (mutación)
curl -s "http://<PUBLIC_IP>/dvga/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { createPaste(title:\"test\", content:\"hello\", public:true) { paste { title } } }"}'
```

## crAPI (OWASP Completely Ridiculous API)

| | |
|---|---|
| **Puerto** | `8888` (dedicado -- sin prefijo de ruta) |
| **Imágenes** | `crapi/crapi-web`, `crapi/crapi-identity`, `crapi/crapi-community`, `crapi/crapi-workshop`, PostgreSQL, MongoDB, MailHog |
| **Instancias** | 7 microservicios (una instancia cada uno) |
| **Recursos** | ~3.0 CPU / ~2.0 GiB RAM en total |
| **Framework** | React SPA + microservicios Java/Go/Python |
| **Proyecto** | [github.com/OWASP/crAPI](https://github.com/OWASP/crAPI) |

crAPI es el proyecto insignia de OWASP para pruebas de seguridad de API. Se ejecuta como 7 microservicios en un puerto dedicado (8888) porque la SPA de React tiene codificadas sus rutas de API y no puede servirse detrás de un prefijo de ruta. El NSG permite tráfico entrante en el puerto 8888.

MailHog captura todos los correos electrónicos enviados por crAPI (verificación de cuenta, restablecimiento de contraseña). Acceda a MailHog mediante túnel SSH en el puerto 18025.

### Categorías de desafíos

| Categoría | Vulnerabilidades |
|---|---|
| BOLA (Autorización a nivel de objeto rota) | Acceder a vehículos, pedidos e informes de otros usuarios |
| BFLA (Autorización a nivel de función rota) | Escalar a administrador, acceder a endpoints restringidos |
| Asignación masiva | Modificar campos protegidos (rol, saldo) vía API |
| SSRF (Falsificación de solicitud del lado del servidor) | Manipular la obtención de URL del lado del servidor |
| Manipulación de JWT | Falsificar o modificar tokens JWT para escalamiento de privilegios |
| Inyección NoSQL | Inyectar consultas en endpoints respaldados por MongoDB |
| Exposición excesiva de datos | La API devuelve datos sensibles de usuarios |

### Primeros pasos

```bash
# Verificar que crAPI está ejecutándose
curl -sf "http://<PUBLIC_IP>:8888" -o /dev/null -w "%{http_code}"

# Registrar un nuevo usuario
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/signup" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test User","email":"test@example.com","number":"1234567890","password":"Test1234!"}'

# Iniciar sesión
curl -s -X POST "http://<PUBLIC_IP>:8888/identity/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# Acceder a MailHog (vía túnel SSH para verificación de correo electrónico)
# ssh -L 18025:localhost:18025 azureuser@<PUBLIC_IP>
# Luego abra http://localhost:18025 en su navegador
```

### Arquitectura de crAPI

```
Port 8888 -> crapi-web (React SPA + nginx)
             -> crapi-identity (Java, user auth, JWT)
             -> crapi-community (Go, forums, posts)
             -> crapi-workshop (Python, vehicle service)
             -> crapi-postgres (PostgreSQL)
             -> crapi-mongo (MongoDB)
             -> crapi-mailhog (email capture, port 18025)
```
